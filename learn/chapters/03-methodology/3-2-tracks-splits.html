3 tracks × 3 splits 与训练/测试隔离

上一节的 5 个能力轴是愿景。这一节我们落到地面真相：docs 与代码里真实存在的评测结构其实简洁得多——3 个 track × 3 个 split。它怎么切、为什么这么切，决定了你提交一次评测时到底在测什么。

三个 track：按本体使用方式分

docs 把 26 个任务组织进 3 个 track，区分依据是「这台 lift2 本体怎么被用」：

  • mobile_manip——移动操作。机器人要先驱动底盘走过去，再操作。强调 Mobility 与「移动 + 操作」的协调。
  • table_top_manip——桌面操作。底盘基本不动，聚焦双臂在桌面上的灵巧与精度。
  • generalist——通才。一个模型同时面对前两类全部任务，考的是「不分工种」的综合泛化能力。

注意 generalist 并不是第三批新任务，而是不让你按 track 分头训练专用模型——它把整个任务集合丢给同一个策略。这呼应了愿景里 Specialist vs Generalist 的对照：专才可以为某个 track 偏科调优，通才必须一碗水端平。

各 track 对应的训练子集

EBench 的训练数据（2.4 详拆）分三个子集，每个子集喂给哪些 track 是对齐好的——你为某个 track 训练时，用的就是该 track 声明覆盖的子集：

训练子集来源任务×条数 = episodes覆盖 track
long_horizonrule-based / GenManip 脚本9 × 200 = 1,800mobile_manip + generalist
simple_pnprule-based10 × 200 = 2,000mobile_manip + generalist
teleop_tasks人类遥操作7 × 400 = 2,800table_top_manip + generalist
合计6,600 episodes（26 tasks，≈9.86M frames）—

读这张表有两个要点。其一，generalist 覆盖全部三个子集——通才看到的数据是专才的并集，这才公平。其二，灵巧的桌面任务（teleop_tasks）来自人类遥操作而非脚本，因此动作里天然带抖动/迟疑/停顿；mobile 类则来自 rule-based 脚本，更干净。子集来源不同，训练时的噪声特性也不同，这会一路影响到你的策略行为。

三个 split：训练/测试隔离的红线

每个 track 又横向切成 3 个 split。它们的角色完全不同：

split角色对你意味着什么
val_train开放验证（贴近训练分布）可随便调，用来确认「我至少学会了训练里见过的东西」。
val_unseen开放验证（留出、未见）可随便调，但它故意放了训练没见过的情形，用来本地估计泛化，不会泄露 test。
test留出测试（驱动 leaderboard）这是记分的那一份。你反复在它上面调参，得到的数字就不再可信。

这条切分背后是一条不容妥协的红线：训练/测试隔离。capability profile 要可信，前提是它反映真实泛化，而不是对评测分布的过拟合。如果开发者能在 test 上反复试错、挑出碰巧得分高的检查点，那张漂亮的雷达图测的就只是「在 test 上调参的能力」，毫无诊断价值。val_unseen 的存在正是为了给你一个可以放心折腾的代理，让你不必去碰 test。

从 RL 看 train / val / test 与泛化

这套切分你在监督学习与 RL 评测里都见过：train 拟合、validation 选模型与调超参、test 只看最终泛化。区别只在「分布偏移」是具身的——val_unseen 的「未见」不是换个随机种子，而是换了物体、背景或指令措辞（这正是下一节 3.4 泛化轴的内容）。把 test 当成你只能查一次的留出集，是这条红线在工程上的全部含义。

口径提醒：split 细分仍是 WIP

3 tracks × 3 splits 的结构是 docs 的可验证事实，提交时也确实用 ebench/<track>/<split> 这样的路径（4.5 会用到）。但每个 split 具体包含多少 episode、怎么细分，当前 docs 标注为 WIP（work-in-progress），尚未给出确定数字。本节表格里的 6,600 是训练数据集总量，不要把它当成某个 test split 的评测条数——二者是不同的东西。

这一节你需要带走的

记住：评测结构是 3 tracks（mobile_manip / table_top_manip / generalist）× 3 splits（val_train / val_unseen / test）；训练子集与 track 对齐，generalist 看全集；val 开放可调、test 留出记分，这条隔离红线守的是 capability profile 的可信度；split 内部细分仍 WIP。

track 与 split 切的是「在哪个分布上测」。下一节我们钻进每个 track 内部——26 个任务到底是哪些、分成哪三个 family，以及关键的：一次 episode 的分数 ∈ [0,1] 是怎么算出来的。
