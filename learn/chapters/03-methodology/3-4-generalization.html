泛化轴：Object / Background / Instruction / Mixed

3.3 讲的是单个任务内部怎么打分。但 capability profile 还想回答一个更尖锐的问题：换了没见过的物体、没见过的背景、换一种说法的指令，模型掉了多少分？掉在哪根轴上？这一节讲 README/paper 愿景里的 4-axis generalization——并且要再次小心：这里大半是愿景语言。

为什么泛化也要分轴

回到 3.2 的红线：val_unseen / test 之所以「未见」，不是换个随机种子，而是制造了真实的分布偏移。但「模型在 unseen 上掉了 0.2」这句话本身又是一个被压扁的标量——你不知道它是看不懂新物体，还是被新背景干扰，还是听不懂换了措辞的指令。这三种失败要用完全不同的方法去修。

所以愿景主张：把 OOD（out-of-distribution）掉分归因到具体的泛化轴。这和 3.1 的能力轴是同一种哲学的延伸——不只问「泛化好不好」，而是把泛化拆成可定位的因子，让掉分变成一份诊断而非一句叹息。

README/paper 愿景的 4 个泛化轴

愿景给出 4 个 generalization 轴，每一根都对应一类受控的扰动：

泛化轴改变了什么这根轴塌了，说明…
Object换成训练没见过的目标物体（新形状/颜色/实例）视觉表征过拟合到了具体物体外观，没学到「这一类东西怎么抓」。
Background换桌面/场景背景、干扰物、布局策略被背景纹理或杂物带偏，目标定位不鲁棒。
Instruction同一任务换一种措辞（指令释义）语言理解贴着训练措辞死记，没真正接住语义。
Mixed以上几种扰动叠加同时出现综合鲁棒性——最接近真实开放世界的难度。

前三轴是单因子受控变量：一次只动一个变量，才能干净地把掉分归因到那一轴。Mixed 则故意叠加，逼近真实部署里「什么都不一样」的处境——它通常是最难、掉分最多的一轴，但单看它无法定位病灶，得配合前三轴一起读。

指令释义：Instruction 轴怎么造

Instruction 轴的扰动叫指令释义（instruction paraphrases）。同一个任务目标，用多套不同措辞表达——「把苹果放进果盘」「pick up the apple and put it in the fruit bowl」「将那个苹果搁到水果碗里去」指的是同一件事。一个真正理解语义的策略应当对这些措辞不变；一个只记住了训练里那句话表面 token 的策略，换个说法就会失手。

这套释义在数据侧是有迹可循的：训练数据集的 meta/tasks.jsonl 里就为每个任务存了多套指令措辞（2.4 提到过）。Instruction 轴本质上是在评测端继续利用这种多措辞结构，去探测语言理解的鲁棒性边界。

从 RL 看 distribution shift 与 OOD 泛化

这整套就是你熟悉的 distribution shift：训练分布 $p_{\text{train}}$ 与测试分布 $p_{\text{test}}$ 不同，策略在 $p_{\text{test}}$ 上的表现就是 OOD generalization。4 个轴等于把「shift 发生在哪个维度」显式参数化——Object 是观测里物体子空间的 shift，Instruction 是语言条件 $\ell$ 的 shift，Background 是观测里场景子空间的 shift。把掉分归因到轴，就是在问「我的 covariate shift 主要来自哪个坐标」，这比一个笼统的泛化间隙信息量大得多。

这是「愿景语言」，docs 细节仍 WIP

4-axis generalization（Object·Background·Instruction·Mixed）来自 README / paper 的愿景描述，它讲清了 EBench 想怎么诊断泛化。但当前线上 docs 站并未把这四个轴落成可提交的 split 或可验证的评测口径——回忆 3.2，docs 的地面真相只有 3 tracks × 3 splits，且 split 细分本身仍标 WIP。所以：把这四个轴当成理解 EBench 设计意图的框架来读，而不是「现在就能在 leaderboard 上分轴打分」的既成事实。一旦 docs 落地，再以 docs 为准。

这一节你需要带走的

记住三点：(1) 泛化也该分轴，把 OOD 掉分归因到具体维度才可操作；(2) README/paper 愿景给出 4 轴 Object·Background·Instruction·Mixed，前三轴单因子受控、Mixed 叠加，Instruction 靠指令释义探测语义鲁棒性；(3) 这是愿景语言，docs 尚未落地对应口径。
