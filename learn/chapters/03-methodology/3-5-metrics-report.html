指标与诊断报告（愿景 vs 现实）

方法论的最后一块拼图：把前面所有 episode 的分数，聚合成一个 leaderboard 数字，再展开成一份可读的诊断报告。聚合的部分有干净的公式与代码事实；展开的部分则横着一道最需要诚实对待的愿景 vs 现实裂缝。这一节我们两件事都讲清楚。

从 per-episode 到 leaderboard：聚合公式

3.3 已经定义了最底层的量：一次 episode 的 task score ∈ [0,1]，由部分分与时序条件算出。往上聚合只有两步，都很朴素：

$$\text{track\_score}(\text{track},\text{split}) = \frac{1}{N}\sum_{i=1}^{N}\text{score}_i \,,\qquad \text{score}_i\in[0,1]$$

其中 $N$ 是该 track / split 下所有 episode 的数量。也就是说，track score = 该 track/split 全部 episode 分数的平均。再往上，leaderboard 把各 track 的 track score 做聚合，得到排行榜上那个对外的数字。

注意这条链路的方向：底层是细腻的（连续部分分、时序门控），越往上越压扁（平均、再聚合）。leaderboard 那个数，正是 0.1 里我们抱怨的「贫穷的标量」——它本身没错，错的是只看它。诊断报告存在的全部意义，就是把这条聚合链反向展开回去，让你看见平均值底下的结构。

愿景里的诊断报告：radar / heatmap / transfer curve

README 与线上平台描绘的诊断报告，正是 3.1–3.4 那套哲学的可视化落点，典型有三种图：

  • radar（雷达图）——沿 3.1 的能力轴（Scene·Atomic Skill·Horizon·Precision·Mobility）展开的 capability profile，一眼看出强弱形状。
  • heatmap（热力图）——任务 × 维度的二维网格，定位「具体哪个任务在哪个条件下塌了」，比雷达更细。
  • transfer curve（迁移曲线）——沿 3.4 的泛化轴，画出从 in-distribution 到 OOD（Object/Background/Instruction/Mixed）的掉分曲线，量化泛化代价。

这三种图合起来，就是把上面那个被平均压扁的标量，沿能力轴和泛化轴重新摊开——从「你考了多少分」变成「你在哪强、在哪崩、换了环境掉多少」。

愿景 vs 现实：必须说清的口径

这里是本节、也是整个 Part 3 最需要你记住的一点。上面那套 radar / heatmap / transfer curve 出现在 README 与线上 leaderboard 平台的描述里，是 EBench 对外展示的产品愿景；但当前线上 docs 的 challenge 页并未列出这些诊断图作为已交付的评测产物。换句话说：

口径澄清：诊断报告，愿景 ≠ docs 现实

能验证的现实（docs / 代码）：评测确实产出 per-episode task score ∈ [0,1]、按 track/split 平均成 track score、再聚合上 leaderboard（提交流程见 4.5）。这条标量聚合链是地面真相。

属于愿景的部分：radar / heatmap / transfer curve 这类多轴诊断可视化，以及 3.1 的 5 能力轴、3.4 的 4 泛化轴，来自 README / paper / 线上平台的描述，docs 的 challenge 页未将其列为当前已交付口径。把它们当成「EBench 想给你的诊断报告长什么样」，而不是「现在 docs 上就能拉到的现成报表」。

一句话：分数和排名是现实，分轴诊断画像是愿景。本书一路这么标注，就是不想让你把蓝图当成已完工的房子。

为什么要这么较真？因为如果你照着 README 的雷达图去预期「提交后能自动拿到一张五轴画像」，再对着 docs 发现只有几个聚合数字，你会以为是自己用错了——其实只是愿景跑在了 docs 前面。讲清这条裂缝，你才能正确地使用 EBench：用它确凿的标量评测，同时理解它正朝着分轴诊断演进。

这一节你需要带走的 · Part 3 收束

记住这条聚合链：per-episode score ∈ [0,1] → track score = 平均 → leaderboard 聚合，这是现实；radar / heatmap / transfer curve 这套分轴诊断报告是愿景，docs challenge 页尚未列出。整个 Part 3 的方法论，就是为「把一个数摊成一张可信的画像」铺设坐标——能力轴（3.1）、track/split 隔离（3.2）、任务与打分（3.3）、泛化轴（3.4）、聚合与报告（3.5）。

方法论讲的是「测什么、怎么算」。可这一切要真的跑起来，得有一个把 Isaac Sim 当黑盒、用 client–server 喂观测收动作的评测引擎。Part 4 我们就钻进那台引擎，从「为什么是 client–server」开始。
