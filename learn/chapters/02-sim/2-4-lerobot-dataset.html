LeRobot 数据格式与 EBench 数据集

仿真栈的最后一块地基是数据。前面 cuRobo / GenManip 生成的轨迹要落盘、要能被千万条地训练，就得有一种标准格式——这就是 LeRobot。这一节我们先看它怎么把一条机器人轨迹拆成「低维 + 视频 + 元数据」，再看 EBench 自己的三个数据子集长什么样，并为 Part 4 的 obs/action 契约埋下伏笔。

LeRobot 格式：解决碎片化

LeRobot（来自 HuggingFace）是机器人学习的一种标准数据格式。它存在的理由很现实：机器人数据极度碎片化——每个实验室一套目录结构、一套字段命名、一套相机编码，互相读不动。LeRobot 给出一个统一约定，让一条轨迹无论来自哪、都能用同一套代码加载。它把一条 episode 拆成三种存储，各取所长：

  • Parquet——存低维、高频的信号：关节角、夹爪、底盘位姿、动作向量、时间戳……这些数值列式存储，读取与切片都快。
  • MP4——存相机视频。图像帧不塞进表里（那会爆掉），而是编码成视频文件，按帧索引对齐到 Parquet 的行。EBench 用 AV1 编码的 MP4。
  • metadata——存结构信息：任务文本、robot type、fps、episode 边界、各字段的形状/含义等。EBench 里体现为 meta/modality.json（modality 定义）与 meta/tasks.jsonl（多套指令释义）。

用起来极简：LeRobotDataset("...") 帮你把三种存储缝合好，按 index 取一帧时自动从 Parquet 取低维、从 MP4 解出对应图像帧，吐出对齐的 PyTorch tensor。你的训练循环看到的就是干净的张量，碎片化在这一层被彻底吸收。

版本：v2.1 与 v3

EBench 的三个子集都是 LeRobot v2.1。格式仍在演进——v3 已支持百万级轨迹 / 十亿级帧，并提供 StreamingLeRobotDataset（边下边训，不必先落全量盘）。你用 EBench 数据时认准 v2.1 即可。

EBench-Dataset：三个子集

EBench 的训练轨迹托管在 🤗 EBench-Dataset，分三个子集。它们的划分逻辑很清晰——按生成方式与任务族分，这直接决定了轨迹的「手感」和它服务的评测 track：

子集来源任务×每任务episodes≈ frameseval track
long_horizon rule-based / GenManip 脚本 9 × 200 1,800 ≈ 3.6M mobile_manip + generalist
simple_pnp rule-based 10 × 200 2,000 ≈ 0.96M mobile_manip + generalist
teleop_tasks 人类遥操作 teleop 7 × 400 2,800 ≈ 5.3M table_top_manip + generalist
合计 26 tasks 6,600 ≈ 9.86M —

注意三件事：(1) 任务总数 9+10+7 = 26，与 benchmark 的 26 task 体系对齐；(2) episode 数 1,800+2,000+2,800 = 6,600；(3) teleop 子集任务最少（7）但 episode 最多（每任务 400），帧数也最大（≈5.3M）——人类示范又长又密。

rule-based vs teleop：两种轨迹「手感」

同样是模仿学习的监督信号，rule-based 与 teleop 轨迹的质地截然不同，这一点会实实在在地影响你训出的策略：

  • rule-based（long_horizon / simple_pnp）由 cuRobo 在脚本下优化生成——还记得上一节的 minimum-jerk 吗？轨迹干净、平滑、一致，像被熨过。它是高质量、低噪声的监督。
  • teleop（teleop_tasks）来自人手遥操作，带着人类操作的全部「毛刺」：抖动、迟疑、停顿，速度不均、偶有回拉。它更接近真实人类行为的分布，但噪声大。
从 GAMES103 看 优化出来的轨迹 vs 人手摇出来的轨迹

cuRobo 的 rule-based 轨迹，就是你做轨迹优化时追求的那条解：连续、可导、jerk 最小——数学上「最优且光滑」。teleop 轨迹则像你没有做平滑、直接采样人手输入得到的折线：含高频噪声、不可导的拐点。一个是优化器的产物，一个是人类感觉运动系统的产物。它们在频域上的差别（teleop 高频能量大），正是下面那个训练坑的根源。

坑：teleop 的抖动会被学进策略

VLA 是模仿学习——它会忠实地复制示范，包括 teleop 里的抖动和迟疑。如果你直接混训，策略可能学出一只「会发抖的手」。想要更平滑的策略，一个实用做法是调高 rule-based（GenManip）子集的采样权重，让干净轨迹占更大比重，把抖动稀释掉。这是数据配比层面的调参，不是模型结构问题。

帧字段与 *_delta：动作的绝对与增量两套

打开一帧，你会看到低维信号分成 observation 与 action 两侧。action 侧每个通道都成对出现——一个绝对值，一个 _delta 增量值：

字段维度含义
action.joints / action.joints_delta12双臂关节目标（绝对 / 增量）
action.gripper4夹爪
action.ee_pose / action.ee_pose_delta14末端位姿 [pos,quat]×2（绝对 / 增量）
action.base / action.base_delta3底盘 x/y/θ（绝对 / 增量）

为什么要存两套？因为不同 VLA 对动作空间的偏好不同：有的模型预测绝对目标位姿，有的预测相对当前状态的增量（delta）。数据集同时备好绝对值和 *_delta，是为了让两类模型都能直接取用、不必自己换算。这正是 Part 4 里 obs/action 契约要细抠的东西——is_rel / base_is_rel 这些开关、各 baseline 怎么映射底盘与夹爪，都是从这里的「绝对 vs 增量」长出来的。我们在 4.3 把动作向量逐位解剖。

埋个伏笔：训练数据 ≠ 评测接口，但同源

这里看到的帧字段（state.* / action.* / video.*_view）和你将来在评测里从 EvalClient 拿到的 obs、要回传的 action，命名与语义是一脉相承的。把这张字段表记在心里，到 Part 4 看评测契约时会有强烈的「似曾相识」——那不是巧合，是同一套 modality 在「离线训练」与「在线评测」两个场景的两副面孔。

下载命令

资产（场景/物体/任务，来自 EBench-Assets）与训练数据（EBench-Dataset）分两个仓库，都用 huggingface-cli 拉：

download_assets.sh
# 场景 / 物体 / 任务资产 → saved/
huggingface-cli download InternRobotics/EBench-Assets --local-dir saved --repo-type dataset

# LeRobot 训练轨迹 → saved/dataset/
huggingface-cli download InternRobotics/EBench-Dataset --local-dir saved/dataset --repo-type dataset

注意两条命令都带 --repo-type dataset（这两个仓库在 HuggingFace 上都注册为 dataset 类型），且 Dataset 落到 saved/dataset 子目录，与 Assets 区分开。拉下来后，saved/dataset 里就是上面三个子集的 LeRobot v2.1 目录，可直接喂给 LeRobotDataset(...)。

这一节你需要带走的

LeRobot 用 Parquet（低维）+ MP4（相机）+ metadata 统一了机器人数据，LeRobotDataset 缝合成 tensor。EBench-Dataset 三子集：long_horizon 1,800 / simple_pnp 2,000 / teleop_tasks 2,800 = 6,600 episodes、≈9.86M frames、26 tasks。rule-based 干净平滑、teleop 抖动会被学进去（调高 rule-based 权重可平滑）。帧字段含绝对 + *_delta 两套，为 Part 4 的 obs/action 契约埋下伏笔。
