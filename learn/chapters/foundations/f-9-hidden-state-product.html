Hidden state 才是产品：Transformer 在优化什么

f-5 到 f-8 讲了 loss 的世界，现在我们回到 Transformer 主线，回答一个总纲问题：这个模型被训练来优化什么？答案可能和你想的不一样——Transformer 本身从不输出答案，它只生产一种中间货物：被上下文充实过的 hidden state。真正的答案，由接在最后的那颗「任务头」从 hidden state 里读出来。

前置

本节假设你读过 F-3 的 Block 主线与 F-4 的 token 来源。f-10 会接着讲「训练怎么更新这些表示」，到时 f-5/f-6 的 MSE 与 Cross Entropy 会派上用场。

先立住总画面：表示工厂 + 读表器

把一个训练好的 Transformer 想成一座工厂。进料口是一排 token（文字、图像 patch、机器人 state），出货口不是「答案」，而是一排 hidden state——每个 token 对应一条被几十层 Block 反复加工过的向量。答案在哪？在工厂外面的读表器上：那是一个叫 task head 的小模块（通常只是一个 Linear 加一点后处理），它从 hidden state 里读出词表分布、类别标签或机器人动作。

所以「Transformer 在优化什么」的准确说法是：训练通过任务头的 loss，倒逼整条流水线生产对这个任务有用的 hidden state。表示是手段也是产品；任务头只是决定了「产品要长成什么样」。这也解释了为什么同一个 Transformer 骨架能服务完全不同的任务——换掉读表器，就换了对表示的要求。

每个 token 都有自己的最终 hidden state

先纠正一个常见画面：很多人把最后一层的输出想成「一个向量」。不对。序列里有多少个 token，最后一层就有多少行 hidden state——输出是一个 [B,n,d] 的矩阵：B 条序列，每条 n 个 token，每个 token 一条 d 维向量。输入「我 喜欢 苹果」，输出就是三行：h我、h喜欢、h苹果，一行不少。

注意 h我 已经不是「我」的字典意思了。它是「我」这个位置在此刻这句话里的状态：里面混进了「喜欢」和「苹果」的信息。同一个「苹果」在「我喜欢苹果」和「苹果发布了新手机」里，最终 hidden state 完全不同——这正是 Transformer 取代静态词向量的原因。

为什么一行能装下整句的信息

机制上你在 F-2 已经见过：每一层 Attention，每个位置都拿自己的 Query 去和所有可见位置的 Key 匹配，再把它们的 Value 按权重取回来。一轮之后，「苹果」那一行就掺进了「喜欢」的内容；几十轮之后，每一行都是整句信息的反复压缩与重组。可以把它想成一场圆桌传话：每个人都只和同桌交换几次纸条，几十轮下来，每个人口袋里都装着全场的摘要。

这回答了一个关键疑惑：hidden state 的「上下文」不是比喻，是字面意义——它是 Value 加权的数学结果。但也不要过度承诺：一行向量容量有限，注意力权重是训练出来的取舍，它不保证保留你关心的全部细节（F-3 对 residual 也有同样的诚实标注）。

四种任务，四种读法：区别在「读哪一行」

F-3 末尾的表给过四种系统的速览。这里不问「读什么」，而问为什么各自恰好读那个位置——答案藏在信息汇聚的方向里。

  • GPT 读最后一个位置。因为它用 causal mask（因果掩码）：每个位置只能看到自己和左边的内容。整句话的信息被一路向右搬运，最后那个位置是唯一「看全了整句」的地方。要预测下一个词，读它一行就够。
  • BERT 读 [CLS] 位置。BERT 是双向的，任何位置都能看到整句，反而没有天然的「汇总点」。于是人们专门在句首放一个 [CLS] token，它的唯一职责就是被读取——整句分类时只用它那一行接 classification head。
  • VLM 读多模态的行。图像 patch 行和文本行在同一个 attention 里互相可见；生成描述时，逐个文本位置读出自己的行，过 LM head 变成文字。
  • VLA 读融合后的多行。视觉、语言、state 的行互相可见之后，action head 再从这些行里读出动作。具体怎么读，是 F-11 的整节主题。

共同点只有一句话：主干负责把信息搬进合适的行，任务头负责读那几行。训练时 loss 从任务头反向流回整条链（f-10 细讲），所有参数一起被塑造。

把全链画出来：推理时的完整主线

到这里，Part F 的所有零件可以拼成一张图。这是推理时的主线（训练时多了 loss 与反向传播，见 f-10）：

一条主线，四种出口
Image + Language + State
  → Embedding / Encoder（一切皆 token，F-4）
  → Transformer Blocks ×N（通信 + 加工 + 保留 + 稳定，F-3）
  → 最终 hidden state 矩阵 [B,n,d]
  → 任务头按读法取行
  → GPT：下一个 token｜BERT：类别｜VLM：文本｜VLA：动作

背下这张图，你就有了看任何 Transformer 论文的第一反应：先找「它的 token 是什么」，再找「它的任务头读哪几行」。剩下的都是细节。

从 RL 看 主干与头，像 representation 与 policy 的分离

RL 里你早就见过这种分工：feature extractor（如 CNN 主干）学表示，actor head 和 critic head 分别从同一份特征里读动作分布和价值。Transformer 只是把「主干」换成了更通用的序列处理器，头的种类也随之变多——但「表示归表示，读取归读取」的思路完全一样。

检查：把这几句话说准确

  1. 「Transformer 输出下一个词。」不准确：输出的是 hidden state 矩阵；任务头把某一行读成词表分布。
  2. 「最后一层只给一个向量。」不准确：每个 token 一行，形状是 [B,n,d]。
  3. 「一行的上下文是修辞。」不准确：它是 Attention 按权重取回 Value 的字面结果。
  4. 「换任务就要换整个模型。」不准确：通常换任务头与训练目标；主干的表示能力可以复用（f-12 会讲怎么复用）。

这一节你需要带走的

Transformer 的产品是 hidden state，不是答案；每个 token 都有一行最终表示，且因 Attention 而携带上下文；GPT/BERT/VLM/VLA 的区别，本质是同一份表示的四种读法。下一节 f-10 把镜头转向训练：这些表示是被谁、按什么规则一步步塑造出来的——从参数清单讲到梯度高速公路。
