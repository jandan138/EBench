一切皆 Token + Embedding

“一切皆 token”这句话很容易误导小白，好像图像、状态、动作都被硬切成文字编号。更准确的说法是：不同模态进入 Transformer 前，都要被包装成同宽向量。离散对象通常查 embedding 表；连续对象通常通过线性层或 encoder 投影。

F-3 里的向量行，在 VLA 里从哪里来？

F-3 把 Transformer 的输入写成 [B,n,d]：B 是 batch，n 是序列位置数，d 是每个位置的向量宽度。那一章先假设这些行已经准备好，专心解释它们如何经过 Block 更新。本章只回答更早的一步：文字、图像和机器人状态怎样各自变成一行 d 维向量，再排进同一条序列。

先固定本章的主线

不同来源的原始数据 → 同宽向量 → Transformer 的输入序列。动作会在这条主线旁边出现，但要先分清它何时是训练目标、何时是推理输出、何时才是某种架构的可选上下文。

从字符串到向量：四个东西不要混

你读 Transformer 时经常会看到 token、token id、embedding、hidden state 混在一起说。它们不是同一个东西，只是在一条流水线上前后相接。

名字是什么例子模型里长什么样
字符串片段人能读的文本片段"她"、"Transformer"还不是模型输入
tokentokenizer 切出的基本单位一个字、一个子词、一个特殊符号概念单位
token id词表里的整数编号3912整数索引
embeddingid 查表得到的初始向量$e_i=E[\text{id}_i]$$d$ 维浮点向量

所以“token 是不是一个 ID？”答案是：日常口语里会混用，但严格说，token 是切分出来的单位，token id 是它在词表里的整数编号，embedding 是这个 id 查表后的向量。模型真正能做矩阵乘法的是向量，不是字符串，也不是中文词本身。

“token”有两种语境，先把它们分开

语境一：文本 token

它是 tokenizer 从字符串里切出的单位，可以再查词表得到 token id。这里的 token 和文字、子词、特殊符号有关。

语境二：sequence slot

它是 Transformer 序列里的一个向量位置。图像 patch、状态片段甚至特定架构里的动作表示，都可以占一个 sequence slot；它们不需要先拥有文本 token id。

因此，“状态 token”通常是简写，准确说法是“由状态投影得到、放在序列中的向量位置”。后文若说“token 在通信”，指的是第二种语境的向量行，不是在说字符串自己有状态或函数。

Embedding、representation、hidden state 的生命周期

先认四个本页会反复出现的符号：i 是位置编号，n 是位置总数，d 是每条向量的宽度，$\ell$ 是层号。第一层 Transformer 看到的通常不是裸 embedding，而是 token embedding 加上位置信息：

$$h_i^0 = x_i = e_i + p_i$$

$e_i$ 是第 $i$ 个 token id 查表得到的 embedding，$p_i$ 是第 $i$ 个位置的位置 embedding 或位置编码。很多论文直接把第一层输入写成 $x_i$，不是因为忘了 $e_i+p_i$，而是因为后面推导只关心“当前层输入的那条向量”。

术语最准确的含义什么时候和别的词相同
embedding输入端查表或投影得到的初始向量刚进入模型时，常是 hidden state 的起点
hidden state某一层、某一位置当前保存的向量$h_i^0$ 可由 embedding 加位置得到；$h_i^1$ 以后已不是 embedding
representation“表示”的泛称，强调这条向量编码了什么信息embedding 和 hidden state 都是 representation

经过一层 Transformer 后，hidden state 会被更新：

$$h_i^{\ell+1} = \text{TransformerBlock}^{\ell}(h_1^\ell,\ldots,h_n^\ell)_i$$

这行式子说明两件事。第一，下一层第 $i$ 个位置的向量来自上一层所有位置的共同计算，不只是原来的第 $i$ 个 embedding。第二，后面的 $h_i^1,h_i^2,\ldots$ 已经包含上下文信息，所以再叫它 embedding 就不准确了，应该叫 hidden state 或 representation。

为什么论文爱写 $x_i$

$x_i$ 常常是“当前正在处理的第 $i$ 个位置向量”的通用符号。第一层时它可能等于 $e_i+p_i$；第 10 层时它可能表示第 10 层输入 hidden state。读论文时要看上下文，不要默认所有 $x_i$ 都是原始 embedding。

两条进场路线：查表和投影

不要把对象硬分成“都是 token”或“都不是 token”。更有用的问题是：它怎样得到一个 d 维 sequence slot？离散对象常查表；连续对象常经投影。

对象怎么变成向量公式例子
离散 token先变成整数 id，再查 embedding 表。$e=E[\text{id}]$文字 BPE token、离散动作 token。
连续输入用线性层或 encoder 投影到模型宽度。$x=Wu+b$机器人状态、图像 patch 特征、连续动作变量。

在 $x=Wu+b$ 里，u 是原始连续输入，W 是学习到的投影矩阵，b 是偏置，x 是投影后的 d 维向量。可以把 sequence slot 想成 Transformer 圆桌上的一个座位：文字、图像 patch、状态片段都能坐上去。

文字：先切词，再查表

语言模型不会直接吃完整句子。它先用 tokenizer 把句子切成 token，再把每个 token 映射成 id。比如“把蓝色杯子放进托盘”可能被切成若干子词 token。每个 id 查 embedding 表，得到一条 $d$ 维向量。

$$\text{id}_1,\ldots,\text{id}_L \quad \rightarrow \quad X_{\text{text}} \in \mathbb{R}^{L \times d}$$

这就是你在 LLM 里熟悉的 next-token prediction 的基础。F-6 会解释为什么离散 token 常用 Cross Entropy 训练。

图像：切成 patch，再变成向量

图像不是天然的一维句子，但可以被切成 patch。ViT 风格的视觉编码会把一张图切成很多小块，每块通过线性投影或视觉 encoder 变成一个向量。四个相机可以产生更多 patch token，最后和文字 token 放到同一序列里。

你可以把图像 patch 想成“视觉词”。某个 patch 可能覆盖蓝色杯子，另一个 patch 覆盖托盘边缘。Attention 后面就能让“蓝色杯子”这几个文字 token 去看对应的图像 patch。

连续状态：投影成 1 或 k 个 sequence slot

机器人状态通常是连续数值，例如关节角、夹爪宽度和底盘速度。它们没有词表 id，不能查 embedding 表。最简单的设计把整条状态向量 $u_{\text{state}}$ 投影成一个位置：

$$x_{\text{state}} = W_{\text{state}}u_{\text{state}} + b_{\text{state}}$$

这里 u 是原始状态数值，W 和 b 是学习到的参数，x 是一个 d 维 sequence slot。也可以让 encoder 输出 k 个位置：

$$u_{\text{state}} \longrightarrow X_{\text{state}} \in \mathbb{R}^{k\times d}$$

k=1 是“整条状态占一个位置”的最简单情况；k 个位置则是模型设计选择，例如按关节组、时间片或 learned queries 拆开。后面的“7 个状态 token”只是本章算例选择 k=7，不是所有 VLA 的固定规则。

动作有三个角色，不是三种 action

发生在什么时候动作在做什么是否属于本例 observation 序列
训练时训练时专家 target：拿示范动作和模型预测比较，交给 MSE、CE 或生成式损失。不是；它是监督答案。
推理时推理时 policy output：模型算出动作，再转换成环境契约需要的 payload。不是；它是模型输出。
特定自回归设计可选上下文输入：已知或先前动作可以作为 sequence slot 进入上下文。可能；这依赖 teacher-forcing 或自回归架构。
“action token”还要再问一句

连续 action slot 只是序列中的向量位置，不是词表 id。只有连续动作先被 FAST 一类 tokenizer 量化后，才会得到可用于 Cross Entropy 的离散 action token id；也不是所有 VLA 都会把动作 token 喂回 Transformer。

把一帧 EBench 观测排成一串

现在只拼 observation 输入。假设某一步有 8 个文字 token、196 个图像 patch，状态 encoder 选择输出 k=7 个状态 slot。它们都被变成 d 维向量后，可以拼成：

$$X = \text{concat}(X_{\text{text}}, X_{\text{image}}, X_{\text{state}}) \in \mathbb{R}^{211 \times d}$$
模态数量示意进入 Transformer 前后文对应章节
指令文字8token id 查 embedding。5b-1 样本里的 language。
图像 patch196视觉 encoder / patch projection。1-2 的 VLM backbone。
本体状态7连续数值投影成 7 个 slot。4-2 的 observation schema。
动作取决于 horizon本例不是 observation 输入；它通常是训练 target 或推理 output，特定自回归设计才可能作上下文。5-8 离散/连续分工。

只变成向量还不够：还要知道位置和类型

如果把 211 个向量随便打乱，模型会丢掉顺序和模态边界。因此真实系统通常还会加入位置编码、模态 embedding、相机 embedding、时间步 embedding 等辅助信息。你不需要在 Part F 里掌握所有细节，只要记住：模型不仅要知道“这个向量是什么内容”，还要知道“它来自哪里、排在第几位”。

从 GAMES103 看 坐标系和属性通道

仿真里一个点不只是三个坐标，还可能有速度、法线、材质、所属刚体。VLA 里的 token 也类似：内容向量只是主体，位置、模态、时间、相机来源会作为额外线索帮助模型解释它。

小检查：拼一个序列

如果一句指令有 12 个文字 token，两张相机图各产生 64 个 patch，状态被拆成 4 个向量，那么总序列长度是 $12 + 2 \times 64 + 4 = 144$。如果模型宽度 $d=1024$，输入形状就是 $144 \times 1024$。

这就是 F-1 那个 $X \in \mathbb{R}^{n \times d}$ 的具体来源。F-2 和 F-3 的 Attention/Block 都是在这排向量上工作。

这一节你需要带走的

“一切皆 token”的准确含义是：不同来源的数据要先变成同宽向量，才能占 Transformer 的 sequence slot。文本 token 常经 id 查 embedding；图像和连续状态常经 projection 或 encoder。动作不是天然固定在输入序列里的东西：训练时常是 target，推理时常是 policy output，只有特定架构才会把它作为上下文。

下一节 F-5 沿着“训练时专家 target”这条线继续：若动作是连续向量，最朴素的学习方式就是回归和 MSE；离散 action token 的 Cross Entropy 路线会在 F-6 出现。
