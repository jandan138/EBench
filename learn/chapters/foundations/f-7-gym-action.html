Gym 契约 + 动作空间（离散 vs 连续）

前面讲了模型内部的 token、loss 和概率。现在回到机器人世界：模型不是在真空里输出数字，它要和环境按固定契约来回交换 observation 与 action。Gym 的 reset/step 心智模型，是理解 EBench EvalClient 的最低门槛。

最小循环：reset 拿第一帧，step 送动作

先只记两个名词：observation 是环境交给策略的当前信息，action 是策略交还给环境的控制命令。Gymnasium 风格环境的核心很小：reset 拿第一帧，再反复“看 observation → 算 action → step”。

Modern Gymnasium mental model
obs, info = env.reset()
episode_done = False

while not episode_done:
    action = policy(obs)
    obs, reward, terminated, truncated, info = env.step(action)
    episode_done = terminated or truncated

terminated 表示任务按自身规则到达终止状态，例如成功或失败；truncated 表示时间上限等外部截断。二者都是一个 episode 的边界。代码重点不是 Python 语法，而是接口边界：环境不管模型内部是 Transformer、Diffusion 还是手写规则；模型也不需要知道仿真器如何跑物理。双方只靠 observation 和 action 对话。

EBench 的 EvalClient 是同一件事的 client-server 版本

EBench 把环境放在 server 侧，把你的模型放在 client 侧。先看不带真实字段的同构关系：

Client-server mental model
observations = client.reset()
eval_finished = False

while not eval_finished:
    worker_obs = select_one_worker(observations)
    if worker_obs["reset"]:
        policy.reset()

    action_payload = policy(worker_obs)
    observations, eval_finished = client.step(action_payload)

这里有三个不同层次的边界：Gymnasium 的 terminated/truncated 是一个 episode 的结束；worker_obs["reset"] 是 EBench 通知“这个 worker 开始新 episode”，此时策略应清缓存；eval_finished 是 client 级整段评测完成。它们不是同一种信号。

再看真实字段，但先只认名字

真实代码会用单个 worker_id 索引 observation，并把 action 按 worker 包装；构造 client 时还需要 endpoint、token、run_id 和 worker_ids。概念上可写成 worker_obs = obs[worker_id]["obs"]，再调用 client.step({worker_id: action_payload})。字段形状和 step/chunk 模式留给 4-2 与 4-4。

Observation 是条件，Action 是承诺

模型每一步都在学一个条件映射：

$$\pi: o_t \mapsto a_t \quad \text{or} \quad \pi: o_t \mapsto A_t$$

$o_t$ 是当前观测，$a_t$ 是当前一步动作，$A_t=[a_t,\ldots,a_{t+H-1}]$ 是未来一段动作。训练时，专家 action 是监督 target；推理时，policy 产生 action payload 并交给环境。它不是随便吐一个数组，而是必须符合环境约定的 shape、单位、范围和语义。F-4 里的 动作三角色 就在这里落地。

动作空间：Discrete、Box、Dict

动作空间数学形态例子常见训练方式
Discrete$a \in \{0,\ldots,K-1\}$左/右/停，选择一个技能。分类、Cross Entropy、categorical policy。
Box$a \in \mathbb{R}^{d_a}$，带上下界。关节速度、末端位姿增量、底盘速度。MSE、高斯策略、flow/diffusion。
MultiDiscrete多个离散维度组合。多个按钮或多个量化 code。多头 CE 或序列化 token。
Dict多个字段组成一个 action。机械臂 + 夹爪 + 底盘 + 控制类型。按字段路由不同 head/loss。
只看结构，不是 EBench 的精确 schema
arm_velocity = Box(low=-1.0, high=1.0, shape=(7,))
action = Dict({
    "arm": arm_velocity,
    "gripper": Box(low=0.0, high=1.0, shape=(1,)),
    "mode": Discrete(2),
})

Box 是一个带范围的连续数组；Dict 不是另一条大向量，而是按字段组织多个子空间的容器。真实系统仍可在网络内部把这些字段拼接、分头预测或序列化。

EBench 的真实 action 更接近 Dict + 连续 Box 的组合，而不是一个简单按钮。比如同一个 action 字典里可能同时包含手臂控制、夹爪控制、底盘运动、绝对/相对控制标志。后文 4-3 会把 16/19-D 与 base_motion 拆开。

Action chunk：一次输出未来一段

很多 VLA 不只输出当前一步动作，而是先预测未来一段：

$$A_t^{\text{pred}}=[a_t,\ldots,a_{t+H_{\text{pred}}-1}]\in\mathbb{R}^{H_{\text{pred}}\times d_a},\qquad K_{\text{exec}}\le H_{\text{pred}}$$

$H_{\text{pred}}$ 是模型一次预测多少步；$K_{\text{exec}}$ 是环境实际执行前 K 步后就重新观察、重新规划多少步。它们不是同一个数。比如 50Hz 下，模型可预测 $H_{\text{pred}}=50$ 步约一秒动作，却只执行 $K_{\text{exec}}=10$ 步约 0.2 秒后再看新 observation。

这种“预测一段、只执行前一段、再重规划”的方式叫 receding horizon。具体执行多少取决于 EBench 的 step/chunk 模式，而不是 action chunk 这个概念本身；后文 4-4 会把它拆开。

把 F-5/F-6/F-8 和动作空间连起来

动作是 Discrete 时，Softmax + CE 很自然。动作是连续 Box 且答案比较单峰时，MSE 很自然。动作是连续 Box 但同一观测有多种合理轨迹时，生成式策略更自然。

单位、范围和 reset 比模型名字更容易出错

小白常以为“模型会输出动作”就结束了。实际上，动作字段的单位和范围极其重要：米、弧度、米/秒、归一化到 [-1,1]，绝对目标还是相对增量，夹爪 1 表示开还是关。这些都不是神经网络能自动猜对的接口细节。

另一个常见坑是 reset。一个 episode 开始时，模型内部缓存、动作队列、历史图像都应该清空。EBench 的 obs 里会有 reset 标志提醒你；不清空就会把上一局的动作记忆带到下一局。

从 RL 看 policy 和 env 的边界

RL 训练里你已经习惯 $\pi(a_t\mid o_t)$ 和 env.step 分离。EBench 的评测也遵守这个边界，只是 env 在远端 server，policy 在你的进程里。理解这个边界，会让后面的 client-server、online challenge、worker 并行都简单很多。

小检查：判断动作类型

  1. “选择 6 个技能中的一个”是 Discrete。
  2. “输出 7 个关节速度”是 Box。
  3. “同时输出左右臂、夹爪、底盘速度和控制模式”更像 Dict。
  4. “一次输出未来 50 步，每步 19 维”是 action chunk，shape 是 $50 \times 19$。
常见误解

连续动作不等于一定要用生成模型。MSE、高斯策略、flow/diffusion 都能处理连续动作。是否需要生成式方法，关键看数据分布是否多峰、控制频率和任务容错度如何。

这一节你需要带走的

Gym 契约的核心是 reset/step；EvalClient 是它在 EBench client-server 评测里的版本。动作空间决定模型输出 shape 和训练 loss：离散动作适合 CE，连续动作可以回归或建模分布。下一节 F-8 会解释为什么“连续动作分布”往往比“一个连续向量”更重要。
