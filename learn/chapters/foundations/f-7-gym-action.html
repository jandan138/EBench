Gym 契约 + 动作空间（离散 vs 连续）

前面讲了模型内部的 token、loss 和概率。现在回到机器人世界：模型不是在真空里输出数字，它要和环境按固定契约来回交换 observation 与 action。Gym 的 reset/step 心智模型，是理解 EBench EvalClient 的最低门槛。

最小循环：reset 拿第一帧，step 送动作

Gym 风格环境的核心很小：先 reset，拿到第一帧 observation；然后反复让模型根据 observation 产生 action，再把 action 交给环境 step。

Gym-style loop
obs = env.reset()
done = False

while not done:
    action = policy(obs)
    obs, reward, done, info = env.step(action)

这段代码的重点不是 Python 语法，而是接口边界：环境不管你的模型里面是 Transformer、Diffusion 还是手写规则；模型也不需要知道仿真器内部怎么跑物理。双方只靠 observation 和 action 对话。

EBench 的 EvalClient 是同一件事的 client-server 版本

EBench 把环境放在 server 侧，把你的模型放在 client 侧，所以接口长得略有不同，但心智模型一样：

EvalClient loop mental model
client = EvalClient(base_url=endpoint, worker_ids=["0"], run_id=run_id)
obs = client.reset()

while True:
    if obs["0"]["obs"]["reset"]:
        model.reset()

    action = model.get_action(obs["0"]["obs"])
    obs, eval_finished = client.step(action)

    if eval_finished:
        break

client.close()

后文 4-2 会细拆真实字段。Part F 里你只需要先记住：EvalClient 本质上就是把 Gym 的 reset/step 搬到网络和多 worker 评测里。

Observation 是条件，Action 是承诺

模型每一步都在学一个条件映射：

$$\pi: o_t \mapsto a_t \quad \text{or} \quad \pi: o_t \mapsto A_t$$

$o_t$ 是当前观测，$a_t$ 是当前一步动作，$A_t=[a_t,\ldots,a_{t+H-1}]$ 是未来 $H$ 步动作 chunk。输出动作不是随便吐一个数组，而是必须符合环境约定的 shape、单位、范围和语义。

动作空间：Discrete、Box、Dict

动作空间数学形态例子常见训练方式
Discrete$a \in \{1,\ldots,K\}$左/右/停，选择一个技能。分类、Cross Entropy、categorical policy。
Box$a \in \mathbb{R}^{d_a}$，带上下界。关节速度、末端位姿增量、底盘速度。MSE、高斯策略、flow/diffusion。
MultiDiscrete多个离散维度组合。多个按钮或多个量化 code。多头 CE 或序列化 token。
Dict多个字段组成一个 action。机械臂 + 夹爪 + 底盘 + 控制类型。按字段路由不同 head/loss。

EBench 的真实 action 更接近 Dict + 连续 Box 的组合，而不是一个简单按钮。比如同一个 action 字典里可能同时包含手臂控制、夹爪控制、底盘运动、绝对/相对控制标志。后文 4-3 会把 16/19-D 与 base_motion 拆开。

Action chunk：一次输出未来一段

很多 VLA 不只输出当前一步动作，而是输出未来一段动作：

$$A_t = [a_t, a_{t+1}, \ldots, a_{t+H-1}] \in \mathbb{R}^{H \times d_a}$$

如果控制频率是 50Hz，$H=50$ 就大约是一秒动作。模型可以每隔几步重新规划一次，这叫 receding horizon。action chunk 的好处是让模型看到短期运动结构；代价是接口和训练 shape 都更复杂。

把 F-5/F-6/F-8 和动作空间连起来

动作是 Discrete 时，Softmax + CE 很自然。动作是连续 Box 且答案比较单峰时，MSE 很自然。动作是连续 Box 但同一观测有多种合理轨迹时，生成式策略更自然。

单位、范围和 reset 比模型名字更容易出错

小白常以为“模型会输出动作”就结束了。实际上，动作字段的单位和范围极其重要：米、弧度、米/秒、归一化到 [-1,1]，绝对目标还是相对增量，夹爪 1 表示开还是关。这些都不是神经网络能自动猜对的接口细节。

另一个常见坑是 reset。一个 episode 开始时，模型内部缓存、动作队列、历史图像都应该清空。EBench 的 obs 里会有 reset 标志提醒你；不清空就会把上一局的动作记忆带到下一局。

从 RL 看 policy 和 env 的边界

RL 训练里你已经习惯 $\pi(a_t\mid o_t)$ 和 env.step 分离。EBench 的评测也遵守这个边界，只是 env 在远端 server，policy 在你的进程里。理解这个边界，会让后面的 client-server、online challenge、worker 并行都简单很多。

小检查：判断动作类型

  1. “选择 6 个技能中的一个”是 Discrete。
  2. “输出 7 个关节速度”是 Box。
  3. “同时输出左右臂、夹爪、底盘速度和控制模式”更像 Dict。
  4. “一次输出未来 50 步，每步 19 维”是 action chunk，shape 是 $50 \times 19$。
常见误解

连续动作不等于一定要用生成模型。MSE、高斯策略、flow/diffusion 都能处理连续动作。是否需要生成式方法，关键看数据分布是否多峰、控制频率和任务容错度如何。

这一节你需要带走的

Gym 契约的核心是 reset/step；EvalClient 是它在 EBench client-server 评测里的版本。动作空间决定模型输出 shape 和训练 loss：离散动作适合 CE，连续动作可以回归或建模分布。下一节 F-8 会解释为什么“连续动作分布”往往比“一个连续向量”更重要。
