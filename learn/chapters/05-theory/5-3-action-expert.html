Action expert：Transfusion 与 MoE

上一节我们用 flow matching 把动作「积」了出来。但这套机制要嫁接到一个预训练好的 VLM（PaliGemma）上——VLM 天生处理离散文本 token、用交叉熵训练，而动作是连续的、用 flow matching 训练。怎么让同一个 transformer同时干这两件事？π0 的答案是 Transfusion 加上一个 action expert。

Transfusion：一个 transformer，两套目标

π0 在结构上 follow Transfusion 的思路。核心一句话：一个 transformer，按 token 类型用不同的训练目标。

  • 离散 token（文本、语言指令）：照常用 next-token cross-entropy（交叉熵）监督——这是 VLM 的母语。
  • 连续 token（动作）：用上一节的 flow-matching loss 监督——回归去噪向量场。

两类 token 在同一个序列、同一次前向里流过 transformer，只是各自挂不同的损失头。这就避免了「把连续动作硬离散成 token」的信息损失（早期 RT-2/OpenVLA 那条路），又让模型保留了 VLM 处理图文的全部能力。Transfusion 原本是为「同一个模型既会画图（连续像素，diffusion）又会写字（离散文本，CE）」设计的；π0 把「画图」换成了「出动作」。

从 RL 看 离散动作头 vs 连续动作头

这其实是你熟悉的 action space 之争的翻版。RL 里离散动作空间（softmax over actions，交叉熵）和连续动作空间（高斯/确定性策略）需要完全不同的策略头。早期 VLA 选了离散——把每维动作分箱成 token，复用语言模型的自回归 + 交叉熵，工程上省事。但 π0 论证：高频灵巧控制需要连续、精细、多模态的输出，离散分箱既丢精度又难表达多模态。于是它在同一个骨干上挂了一个连续动作头（flow matching），而不是改造离散头。Transfusion 让这两种「策略头哲学」在一个网络里共存。

Action expert：给 robotics token 一套独立权重

光有两套损失还不够。π0 更进一步：处理 robotics 相关 token（action + state）时，用一套独立的 transformer 权重。这套独立权重就叫 action expert。

用 π0 自己的话说，这「类似一个两元的 mixture-of-experts（MoE）」：

  • 一元专家处理 image / text 输入输出——用预训练好的 PaliGemma（3B）权重；
  • 另一元专家处理 robotics（state / action） 输入输出——用 action-expert 权重，从零随机初始化，规模约 300M。

为什么要分两套权重，而不是让所有 token 共享同一套？直觉是：图文 token 和动作 token 的统计性质天差地别——前者是离散语义符号、后者是连续物理量。强迫一套 MLP/投影同时拟合两者，会互相拖累；而且我们想保住 PaliGemma 预训练学到的 web 知识，不希望随机初始化的动作部分把它冲乱（这条线索会在 5.4 的 knowledge insulation 里推到极致）。给 robotics token 一套专属权重，让它专门学控制表示，是最自然的隔离。

这是「条件路由」MoE，不是「学习路由」MoE

注意这和你可能听过的稀疏 MoE（用 gating 网络动态决定每个 token 去哪个专家）不同。这里路由是固定且由 token 类型决定的：是图文就走 PaliGemma 权重，是 state/action 就走 action-expert 权重。没有学习出来的 gate，只有一条「按 modality 分流」的硬规则。

关键：两专家共享一次 attention

如果两套专家各算各的、互不通信，那就是两个独立模型，动作根本看不到图像和指令。所以最关键的设计是：两元专家共享同一次 attention。

机制是这样的：所有 token（图文 + state + action）排进同一个序列，做一次联合 self-attention——动作 token 在这一步里 attend 到图像 patch、语言 token、state，从而「读到」当前观测。但在 attention 之外，每类 token 走自己专家的 MLP 和投影层。换句话说：

子层是否共享说明
self-attention共享一次所有 token 在此交互，动作借此读到观测
MLP / 投影 / 输出头各用各的图文走 PaliGemma 权重，state/action 走 action-expert 权重

这正好接上了上一节结尾的 KV cache 巧思：因为观测 token 的 KV 在那一次 attention 里算好就不变，flow matching 的 10 步 Euler 只需反复更新 action token 对这份缓存的 attention，开销极小。

Attention mask：动作 token 是 full bidirectional

还有一个容易忽略但重要的细节——mask 形状。文本 token 通常是 causal（因果）mask：第 $i$ 个 token 只能看前面的，因为文本是自回归一个一个生成的。但动作 token 不是自回归生成的：整段 chunk 是 flow matching 一次性积出来的，chunk 内部没有先后生成顺序。

所以 action token 用 full bidirectional（全双向）mask：chunk 内每个动作 token 都能 attend 到 chunk 内所有其它动作 token。这让这 50 个动作能彼此协调、保持物理连贯（左右臂配合、轨迹平滑），而不是被人为切成单向序列。下面这个部件让你看清：哪些 token 走哪个专家、哪条 attention 边是允许的、动作块内部那个稠密双向连接长什么样。

参数账：3.3B 是怎么凑出来的

把规模算清楚，方便你在读代码时核对：

  • PaliGemma 骨干：约 3B（背景知识：SigLIP 视觉 ≈400M + Gemma LM ≈2.6B，但 paper 只说 3B，我们也只说 3B）。
  • Action expert：约 300M，从零初始化。
  • 合计 π0 约 3.3B 参数。

π0 还做过一个 π0-small 消融，整体约 470M，用来验证方法在小规模下也成立、并隔离各设计的贡献。值得品味的对比是：动作专家只占总参数的不到十分之一（300M / 3.3B），却承担了「把图文理解翻译成连续控制」这件最关键的事——绝大部分容量花在了保留和利用 VLM 的世界知识上，动作头只是一个轻巧但专精的「翻译官」。

这一节你需要带走的

三点：(1) Transfusion = 一个 transformer 混合两种目标，离散 token 走交叉熵、连续动作 token 走 flow-matching loss；(2) action expert = 给 state/action token 一套独立权重（约 300M，类两元 MoE），与图文的 PaliGemma(3B) 权重共享一次 attention、各用各的 MLP，π0 合计 3.3B；(3) action token 用 full bidirectional mask，因为整段 chunk 是一次积出的、内部无生成顺序。

到这里，π0 的「单层」结构已经讲透。但真正的开放世界（进一个全新的家做饭、铺床）还需要更多：分层推理、离散与连续表示的取舍、以及一个保护 web 知识的关键技巧。下一节进入 π0.5、FAST 与 knowledge insulation。
