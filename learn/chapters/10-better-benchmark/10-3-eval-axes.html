评测维度扩张：从 task execution 到 prediction / safety / planning

EBench 问一个问题：这个策略把任务做成了吗？ 这是个好问题，但只是一个轴。一个真正的具身智能体还可能：在动手前预测未来会发生什么、识别哪个动作危险、用想象去规划而不是硬试。EOS 的第三处泛化，是把评测轴从单一的 task execution，扩成一组并列的能力轴——并且让模型自己声明它支持哪些轴。这一节我们看这组轴长什么样、模型怎么声明、以及一个扎心的例子：EBench 的某个 baseline 早就有了一种被打分完全忽略的能力。

9 个 metric family：把「能力」摊成一组轴

我们在 9.8 见过 EOS 的 9 个 MetricFamily。在这里它们不再是抽象列表，而是评测维度本身。把它们并排看，success rate 只是第一格：

metric family它在问什么EBench 评吗
task_outcome成功 / 进度 / 完成质量✅（这就是 EBench 的全部）
procedure步骤顺序、协议是否合法、出错后能否恢复部分（时序前置条件沾边）
state_transition动作真正引起了什么状态变化➖
precision位姿 / 量 / 力 / 时序 / 对位的精度隐含在 task score 里，不单列
hazard危险识别、安全边界、漏报、严重度➖
prediction对未来状态 / 视频 / 隐状态的预测质量➖
planning规划是否有用：imagined-vs-real 差距➖
generalization跨物体 / 场景 / 后端 / 本体的泛化部分（val_unseen split）
resource时间 / 动作数 / 能耗 / 算力 / 人工干预➖

注意这不是九个孤立的数，而是构成一张 CapabilityProfile 的九个分量（9.8）。这正接回我们全书最开头 0.1 的那句话：从一个 success rate，变成一张多轴能力画像——只不过 EBench 把这句话当愿景写在 README，而 EOS 把它落成了 core 里 9 个真实的 family。

让模型声明它支持哪些轴：ModelAdapter

这里有个微妙但关键的设计。要评 prediction / planning / safety，光有 metric family 不够——你还得让模型能产出这些轴需要的东西。EBench 的契约只有一个动作出口 get_action(obs)→action，模型在结构上没有办法告诉你「我还会预测未来」。EOS 的 ModelAdapter 把这扇门开成了一组可选方法：

src/embodied_eval_os/model_adapters/base.py
class ModelAdapter(ABC):
    """Different models may implement one or more methods:
    actor policy, world predictor, planner, safety critic, autonomous agent."""

    @abstractmethod
    def act(self, observation) -> dict | list[float]:
        ...                                   # 必选：actor，等价于 EBench 的 get_action

    def predict_future(self, observation, action_sequence) -> PredictedFuture:
        raise NotImplementedError              # 可选：world model / WAM 轴

    def rank_plans(self, observation, candidate_plans) -> list:
        raise NotImplementedError              # 可选：planning 轴

    def assess_safety(self, observation, candidate_action) -> dict[str, float]:
        raise NotImplementedError              # 可选：hazard / safety 轴

设计的妙处在于「能力即声明」：只实现 act 的模型就是个纯 actor，evaluator 只会评它的 task/precision 轴；如果一个模型同时实现了 predict_future，evaluator 才会去喂 prediction 类指标。这和 9.7 我们读到的双 adapter 思想一脉相承（9.7）：模型不会被强行按一个不支持的轴打分，也不会因为「benchmark 不问」而让一个真实存在的能力被埋没。

从 RL 看 model-free vs model-based 的评测，终于对称了

在 RL 里你早就知道 model-free（只学 policy）和 model-based（还学一个 world model 用来想象 rollout）是两类不同能力。但传统 benchmark 只用 return 评它们，等于把 model-based 智能体的「想象准不准」这一半能力扔掉了。EOS 的 predict_future + prediction family，相当于给 world model 那一半补上了一把尺子：不光看它最后做成没有，还看它对未来的预测和真实 rollout 差多少（这正是 planning family 里 imagined-vs-real gap 的含义）。

prediction 轴的载体：PredictedFuture

prediction 轴需要一个地方装「模型预测的未来」，这就是我们在 9.4 见过的 PredictedFuture。它是 state 层的一等公民：模型通过 predict_future 吐出一个 PredictedFuture，evaluator 把它和实际 rollout 出来的真实未来对齐比对，得到 prediction family 的指标；如果再问「拿这个预测去规划，比不预测好多少」，就进了 planning family 的 imagined-vs-real 评估。换句话说，world model 不再是模型内部的黑箱，它的输出被提到 trace 里、被当作可评测对象。这是 EBench 的 actor-only 契约里根本不存在的一层。

扎心的例子：A1 早就有，却没人给它打分

把这件事讲透最好的例子，恰恰在 EBench 自己的 baseline 里。InternVLA-A1（6.3）是一个 Mixture-of-Transformers，三个专家共享一次 attention：understanding、generation（预测未来视觉状态 = visual foresight 世界模型想象）、action。也就是说——A1 本身就具备 predict-future 能力，它在动作之外还会想象未来的画面。

可是在 EBench 里，A1 和一个纯 actor VLA 拿到的是同一把尺子：只评它最后把任务做成了没有。它那套 visual-foresight 世界模型的预测准不准、有没有帮到决策——EBench 的打分机制完全看不见。这不是 A1 的问题，也不是 EBench「做错了」，而是 actor-only 评测维度结构上就装不下这种能力。同一个 A1 放进 EOS，至少多出两个轴可评：它的 generation 专家产出的预测能进 prediction family，它的预测对规划的帮助能进 planning family。

construct validity 的回声

这正是 8.1 讲的 construct validity 问题：你测的数，真的对应你想测的能力吗？ 当一个具备世界模型的智能体和一个纯反应式策略拿到完全一样的 success rate，这个数对「世界建模能力」的 construct validity 就是零——它根本没在测那个东西。多轴 capability profile 不是为了「更多数好看」，而是为了让每个轴的数都真的对应一种能力。

这一节你需要带走的

(1) EOS 把评测轴从单一 task_outcome 扩成 9 个 metric family（含 prediction / hazard / planning / resource…），合成一张 capability profile，落实了 0.1 的愿景；(2) ModelAdapter 用可选的 predict_future / rank_plans / assess_safety 让模型声明自己支持哪些轴，能力即声明，不会被错配打分；(3) prediction 轴的载体是 PredictedFuture，把 world model 的输出提为可评测对象；(4) InternVLA-A1 的 visual-foresight 世界模型就是一个「EBench 看不见、EOS 能评」的真实能力。下一节，我们面对最难的轴：跨 backend 比较——并用 evidence tier 替代二元的 success/fail。
