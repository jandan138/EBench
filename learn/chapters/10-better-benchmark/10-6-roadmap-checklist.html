对照 10.1：EBench 在「可复现全量 full pass、依赖隔离、强 baseline」这些已落地的工程能力上，现在仍然实打实强于 EOS——它是个发布了的、跑得动的 benchmark。EOS 强的是架构与边界纪律，但它的多场景 / 多后端 / 多轴能力，大部分还是「契约就绪」而非「跑通」。诚实的结论是：EBench 是当下的可用工具，EOS 是更大目标下的早期地基。 别把地基当成楼。

行动清单：如果你要造自己的具身评测系统

这是这本书最实用的一页。无论你最后用不用 EOS，下面这七条是从 EBench 的工程 win 和 EOS 的设计动作里提炼出的、可以直接照做的原则：

  • 1. 把五层显式分开。 spec（评什么）/ state（世界长什么样）/ trace（留下什么证据）/ adapter（怎么接外部）/ evaluator（怎么打分）各占一层，别让它们互相渗透。一开始就分清，比事后拆开便宜一百倍（9.1）。
  • 2. 定义窄契约。 每层之间只暴露最小接口：actor 就一个 act，世界模型多一个可选 predict_future。契约越窄，能换的实现越多，耦合越少（9.7）。
  • 3. core 保持 domain-agnostic，并加一个 leakage check。 core 里不许出现任何具体 benchmark / 仿真器 / 场景的私有名，并写一个 PR 时会失败的脚本来强制它。原则不被编译成检查，就一定会被慢慢违反（10.5）。
  • 4. 每条 trace 都记 seed + versions + claim_boundary。 没有 provenance 的结果不是证据，是孤证；没有 claim_boundary 的结果迟早被人当成它没声明过的东西（9.5、8.4）。
  • 5. 用 capability profile，别用单标量。 把能力摊成一组轴（task / precision / prediction / hazard / planning / generalization / resource…），让两个总分相近的模型也能看出强弱互补。这是全书第一节（0.1）就埋下的主张。
  • 6. 硬问题用 evidence tier，别用二元 success/fail。 跨后端、往真机靠的时候，报「我站在证据阶梯哪一级 + 这级不能 claim 什么」，而不是一个会误导人的 0/1。宁可报 blocked_with_evidence（10.4）。
  • 7. 把 scenario pack 当 anti-corruption layer。 每个领域（家庭 / 实验室 / 工厂…）都是平级插件，定义自己的语义，但绝不反过来改写 core 契约。这样加场景不动引擎，且场景之间天然可比（10.2）。
从 GAMES103 看 评测系统也是一种「数值方法」

GAMES103 教会你一件事：一个数值方法好不好，不只看它在一个 case 上跑出的数，而看它稳定、可复现、误差可界定。评测系统是一模一样的道理——一个好的 benchmark 不是「给出一个排名」，而是「给出一个误差可界定、来源可追溯、不过度声明的能力测量」。你在这本书里学到的每一条护栏，本质上都是在给「能力测量」这个数值过程加上稳定性与误差界。

合上这本书

你是从 0.1 的一个朴素问题出发的：当你说一个机器人策略「做得好」，你到底在测量什么？十个 Part 之后，你已经能从两个层面回答它。

一个层面是深：你彻底读懂了 EBench——它的 client–server 架构、lift2 本体、Isaac Sim 后端、26 个任务的打分、π0 / π0.5 / X-VLA / InternVLA-A1 四条 baseline 背后的 flow matching 与 action chunking、它怎么用 val/test 隔离守住可信度。这些不是道听途说，是你跟着真实代码一行行拆出来的。

另一个层面是广：你现在有了一整套设计词汇，能去评价、乃至去建造一个评测系统。domain-agnostic core、anti-corruption layer、可组合 BackendProfile、capability profile、evidence tier、claim boundary、provenance——这些词不再是名词，而是你手里能用的工具。你知道一个聚焦的 benchmark（EBench）强在哪、一个通用 OS（EOS）想往哪走、以及两者之间那些诚实的、还没填满的空格。

所以这本书真正想留给你的，不是「EBench 是最好的」或「EOS 是未来」这种结论——这两句都太满了。它想留给你的是一种判断力：面对任何一个具身评测系统，你都能问出对的问题——它在测什么？这个数对应什么能力？它声明了什么、又悄悄没声明什么？换个后端还成立吗？三个月后还能复现吗？能问出这些问题，你就已经站在了「造一个更好的评测系统」的起点上。
