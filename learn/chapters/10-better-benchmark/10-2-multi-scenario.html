从 manipulation-only 到 multi-scenario：scenario pack

EBench 把范围收在「室内 mobile manipulation」。这让它干净，但也意味着：想评一个会做实验室操作、会在工厂装配、会在野外采样的具身模型，你得另起一个 benchmark。EOS 的第一处泛化，就是把「场景」从写死的前提，变成一组互相平级、可插拔的 scenario_pack。这一节我们看清楚：一个 scenario pack 拥有什么、core 守住什么、为什么这比「每个领域 fork 一个 benchmark」更好——以及它现在真实的完成度。

五个平级的包，没有一个享有特权

EOS 的 README 直接给出第一批建议的包，并排放在同一个目录下：

README.md · scenario_packs/
scenario_packs/
  home_manipulation/     # EBench-compatible 居家/室内操作场景
  scientific_lab/        # 科学实验室操作场景
  wetlab_safety/         # 湿实验内生安全 / 安全场景
  factory_assembly/      # 未来工厂装配场景
  field_robotics/        # 未来野外机器人场景

关键词是「平级」。在 EOS 里，EBench 风格的居家操作并不比科学实验室更接近 core——它也只是 home_manipulation/ 这一个包，而且它依赖的 EBench/GenManip 运行时细节还要再往外一层、塞进 adapters/ebench/。这正是我们在 9.1 学到的那条硬规则的直接后果：core 里不许出现任何 benchmark / scenario 私有名，由 check_core_leakage.py 在 PR 时强制。没有「主场景 + 几个旁支」，只有「一组对等的包 + 一个对它们一无所知的 core」。

从 RL 看 一个 algorithm，很多个 env

你不会为了跑 Atari 就去改 gym.Env 基类，也不会为了 MuJoCo 再 fork 一份 gym。env 是插件，接口是公共的。scenario pack 就是这个直觉的放大版：CartPole 之于 gym，约等于 scientific_lab 之于 EOS——它定义自己的物理语义和成功条件，但绝不反过来改写评测引擎的契约。EBench 在这个类比里，是一个把 env 和 framework 焊在一起的特例。

一个 scenario pack 拥有什么 vs core 守住什么

scenario-pack-contract.md 把边界划得很清楚。一个 pack 拥有这个场景自己的语义，core 只保留对所有场景都成立的契约。

scenario pack 拥有（domain-specific）core 守住（domain-agnostic）
task suites（tasks/<task_id>.yaml）TaskSpec 的结构本身（9.2）
scene templates、semantic asset 标注SceneSpec / 资产引用契约
process primitives（如 pipette 的取/排液状态更新）Interaction/Process/Device 原语基类（9.6）
domain metric 实例（volume_error_ul、part_alignment_error_mm）9 个 metric family（9.8）
scenario-specific hazard taxonomy、randomizationEpisodeTrace / evaluator 协议（9.5）

规则简洁：每个场景指标都必须挂到一个 core metric family 上。科学实验室的 liquid_volume_error_ul 不是凭空冒出来的，它声明自己 family: precision；湿实验的 dangerous_action_detection_rate 声明 family: hazard。family 是抽象的、跨场景的；instance 是具体的、场景私有的。core 因此能对一个它从没听说过的指标做通用处理——它知道这是个 precision 类指标，该怎么聚合、怎么进 capability profile，而不需要知道「microliter」是什么。

scenario_packs/scientific_lab/metrics/ · 示意
metric:
  name: liquid_volume_error_ul   # 场景私有的指标实例
  family: precision               # 挂到 core 的通用 family
  unit: microliter
  uses_privileged_state: true     # 量液误差需要特权状态，必须显式声明

为什么这比「每个领域 fork 一个 benchmark」更好

最自然的反应是：实验室评测？那就把 EBench clone 一份，改改任务、改改打分，叫它 LabBench。工厂的再 clone 一份。这条路走三次，你就有了三个各自漂移的代码库：同一个 bug 要修三遍，同一个 baseline 要适配三套接口，三套打分逻辑会慢慢长出三种「success」的定义，互相之间根本没法比。

scenario pack 走的是相反方向：引擎只有一个，场景是数据/插件。修一个 trace bug、加一个新 metric family、接一个新 backend，五个场景同时受益。更重要的是——因为所有场景都落到同一个 EpisodeTrace、同一组 metric family、同一份 capability profile，你第一次有了「在 lab 上的 precision 和在 factory 上的 precision 是同一个轴」这种跨场景可比性。fork 永远给不了这个。

「EBench 做到了吗？」

没有，而且无需做到。EBench 的任务定义本就和 GenManip 的 Isaac 场景配置、26-task 文档缠在一起（3.3），它从设计上就只承载一个场景，所以「场景该不该平级」对它根本不是问题。这不是缺点——是聚焦。EOS 的代价则是：要先把引擎和场景彻底分开，才换来「加场景不动引擎」的能力。两者是不同阶段、不同目标的产物。

诚实的完成度：只有一支真的在跑

必须把话说在前面，否则就成了卖愿景。这五个包远不是都实现了：

  • scientific_lab —— 唯一在重度开发的包。有真实 v0 任务（pick/place tube、unscrew cap、pipette 取排液 proxy、autobio-derived pickup），跑通了「MuJoCo 测得的 reset → Genesis apply → 比对」的跨后端 reset 对齐（即 worldstate runtime roadmap 里的 WR-6-2，见 10.4）。
  • home_manipulation —— 主要是兼容元数据：承载 EBench 风格的居家任务与泛化轴（object / background / instruction / layout / mixed），EBench/GenManip 的运行时细节按规矩留在 adapters/ebench/；当前是 adapter smoke 级别，不是全量复现。
  • wetlab_safety —— 主要是 README + hazard schema 草案（hazard 等级、抽象 hazard trace、DDR/FNR-H 等指标 spec），跑的是 proxy / 抽象 trace，明确禁止真实危险数据采集。
  • factory_assembly / field_robotics —— 基本只有 README，列了示例指标（part_alignment_error_mm、rollover_risk_detection 等），属于 WIP / future pack，尚无可跑任务。

所以 scenario pack 现在更像一个已经验证可行、但只填满了一格的承重框架：抽象边界是真的（contract 写定、leakage check 强制、scientific_lab 跑通了端到端证据链），但「五个场景齐头并进」还停在路线图上。这正是 10.6 状态表要逐项摊开的东西——我们绝不把「契约就绪」说成「场景就绪」。

这一节你需要带走的

(1) scenario pack 把「场景」从写死前提变成平级插件，五个包谁都不进 core，由 leakage check 守门；(2) pack 拥有 task/scene/primitive/metric 实例，core 守住 spec/trace/evaluator 契约 与 9 个 metric family，场景指标必须挂到某个 family；(3) 这比 fork-per-domain 好在「引擎只修一遍 + 跨场景可比」；(4) 但现实是只有 scientific_lab 在重度开发，其余多为 README/WIP——契约就绪 ≠ 场景就绪。下一节，我们泛化第二处收窄：评测轴本身——从只看 task execution，扩到 prediction / safety / planning。
