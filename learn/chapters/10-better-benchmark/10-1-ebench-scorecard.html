EBench 记分卡：做到了什么 / 没做到什么

我们读了九个 Part。前八个 Part 把 EBench 从里到外拆了一遍，第九个 Part 又跟着作者读了他自己写的 EmbodiedEval OS（下文仍简称 EOS）的源码。这最后一个 Part 是结账时刻：我们要给 EBench 打一张诚实的记分卡，分清哪些是它实打实做到的工程胜利、哪些是它故意没做或还没做的；然后看 EOS 是怎么把前者保住、把后者往外推的。这一节先立 thesis：留住 EBench 的工程win，泛化它的 scope。

为什么先打记分卡，而不是先夸 EOS

有一种很省事但很坏的写法：把自己的新系统说成救世主，把被对比的旧系统说成一团糟。我们不这么干，原因很简单——EBench 是一个已经发布、可复现、工程上相当扎实的 benchmark，而 EOS 现在只是一个雄心更大、但还处在 MVP 阶段的早期系统。把 EBench 贬低，等于否定我们前八个 Part 学到的所有东西，也会让「EOS 更好」这句话变成空话。

正确的姿势是：先承认 EBench 在它自己的目标下做得多好，再问一个不同的问题——「如果目标更大（多场景、多后端、不止评 task execution），需要补哪些设计？」EBench 的「没做到」里，绝大多数不是 bug，而是scope 选择：它本来就只想做「Isaac 上的室内 mobile manipulation」。记分卡的价值，是把「做得好的工程」和「故意收窄的范围」分开标注。

Part 10 的 thesis：保住 win，泛化 scope

把记分卡两栏并排看，结论自然浮现。EBench 的 ✅ 全是工程层面的胜利：它们与「只做 manipulation」无关，换成任何场景都该保留——可复现、依赖隔离、partial credit、训练/测试隔离，这些是所有好评测系统的公共底座。而 ➖ 全是scope 层面的收窄：单后端、单本体、隐式 spec、actor-only 契约、单标量。

于是 EOS 的设计动作可以一句话概括：把 EBench 的工程 win 当作不可退让的底线继承下来，同时把每一处 scope 收窄都「泛化」成一个可插拔的维度。

  • 单后端 → 可组合的 BackendProfile（physics / renderer / sensor / control 各自可换，9.3）；
  • 单场景 → 平级的 scenario_pack（home / lab / wetlab / factory / field 都不进 core，9.1）；
  • actor-only → 多能力 ModelAdapter（可选 predict_future / rank_plans / assess_safety，9.7）；
  • 单标量 → CapabilityProfile（9 个 metric family 分轴报告，9.8）；
  • 二元 success/fail（尤其跨后端时）→ 显式 evidence tier + claim boundary（10.4 详谈）。

注意：泛化 scope 不等于「全都做完了」。EOS 现在大部分能力是契约 + smoke 级实现，真正跑通的场景只有 scientific_lab 一支，跨后端也只在一个 pickup 任务上做到 reset 对齐。这一点我们在 10.6 的状态表里会如实摊开——本 Part 不卖愿景，只分清「已实现 vs WIP」。

从 RL 看 benchmark 也是一种「环境设计」

你在 RL 里很熟悉一件事：换个 reward、换个 observation、换个 termination 条件，同一个算法的排名能完全翻转。benchmark 本身就是一组被冻结的环境设计决策。EBench 把这组决策收得很紧（一个本体、一个仿真、一个标量），好处是排名干净、可比；代价是它只能回答一类问题。EOS 想做的是把这些设计决策从「写死」变成「声明式可换」，让同一套引擎能承载很多个「EBench」。

这一节你需要带走的

(1) EBench 的 ✅ 是工程胜利（可复现 30min/8×4090、client–server 依赖隔离、强 baseline、partial credit、训练/测试隔离），这些任何评测系统都该继承；(2) 它的 ➖ 是 scope 收窄（单后端 / 单本体 / 隐式 spec / actor-only / 单标量），是取舍而非缺陷；(3) 唯一真正的缺口 ⚠️ 是 README 愿景与 docs 机制的口径落差；(4) Part 10 的 thesis = 保住 win，泛化 scope。下一节，我们从第一处泛化开始：怎么把「manipulation-only」变成一组平级的 scenario pack。
