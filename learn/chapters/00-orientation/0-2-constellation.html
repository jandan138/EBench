The repo constellation

你 clone 的这个 repo 不是 EBench 的全部，它只是前门。EBench 真正的样子是一群仓库组成的constellation——每个仓库管一摊事，彼此分工明确。这一节我们站到高处，把这张星图认全，你之后读任何一段代码、跑任何一条命令，都能立刻知道「它属于哪颗星」。

为什么不是一个仓库

一个自然的疑问是：为什么不把所有东西塞进一个 repo？答案藏在 EBench 的运行方式里。EBench 跑在 NVIDIA Isaac Sim 上，这是个又重又挑剔的仿真栈——它对 CUDA、PyTorch、各种 GPU 库的版本有强约束（2.1 会细讲）。而你要评测的 VLA 模型，自己也带一整套依赖：可能是某个版本的 transformers、某套 flow-matching 的库、特定的 CUDA。把仿真和模型塞进同一个 Python 环境，依赖几乎必然打架。

EBench 的解法是 client–server：让仿真当一个黑盒服务器独占自己的环境，模型当客户端住在自己的环境，两者只通过网络协议说话。这个切分是整套星座结构的根本理由，我们在 4.1 会专门拆它的契约。先记住：仓库被拆成这么多颗星，本质上是为了依赖隔离。

从 RL 看 environment 与 agent 的老分工

这其实是你早就熟悉的结构。RL 里 environment 和 agent 天然分离：env 负责 step(action) → obs, reward, done，agent 负责 obs → action。EBench 只是把这条边界物理化成进程/机器的边界——server 是 env，client 是 agent。你会发现 reset() / step() 这套接口几乎原封不动地出现在评测协议里。

下面这张表把 constellation 的每颗星列清楚——它是什么、归谁管、和别的星是什么关系。你现在不用记住细节，知道「有这么几颗星、各管一摊」就够了。

EBench 由 Shanghai AI Laboratory / InternRobotics 出品，全称 Elemental Mobile Manipulation Benchmark。把这七颗星连起来看，它们分成三层：资产层（Assets / Dataset 提供素材）、引擎层（GenManip server + genmanip-client 跑评测）、门面层（本 repo + docs + leaderboard 是你和别人的入口）。

client–server 这条缝切在哪

把刀切在 GenManip（server）和 genmanip-client（client）之间，是个深思熟虑的设计。注意 client 包极简——它几乎不带重依赖，只负责按协议收发 obs/action。重的东西（Isaac Sim、USD、PhysX、RTX 渲染）全锁在 server 那一侧。

好处很直接：你给模型搭环境时，完全不必装 Isaac Sim；server 升级 CUDA 也不会逼你的模型跟着改。两侧各自演化，靠一个稳定的网络契约对齐。这也是为什么验证一次 full pass 能干净地跑在 8× RTX 4090 上约 30 分钟——server 那侧把 GPU 并行的活儿全包了，client 只管推理。

这一节只画地图，不教操作。gmp submit / gmp eval 这些命令、EvalClient 的 reset/step/close 用法、动作向量长什么样，全都留给 Part 4（评测引擎）。现在你只要能在脑子里画出这张星图：本 repo 是前门，GenManip 是黑盒 server，genmanip-client 是住进模型 env 的瘦客户端。

这一节你需要带走的

EBench 不是单个 repo，而是一个分工明确的 constellation：本 repo 当前门，GenManip 当黑盒 server，genmanip-client 当瘦客户端，Assets/Dataset 供素材，docs/leaderboard 当对外门面。这一切的根本理由是 client–server 依赖隔离——让仿真和模型各活在自己的环境里。

地图认清了，下一节我们聊点更贴你的事：这本书怎么读。它有自己的结构、两座专门为你（带着 RL 和 GAMES103 底子）搭的桥、以及一套交互玩法。
