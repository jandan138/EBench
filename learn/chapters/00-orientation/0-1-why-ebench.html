从 success rate 到 capability profile

我们先不碰任何代码。先想清楚一个更基本的问题：当你说一个机器人策略「做得好」，你到底在测量什么？这个问题的答案，决定了 EBench 长什么样。

一个数字的贫穷

大多数 manipulation benchmark 给你一个数：success rate。模型跑 100 个 episode，成功 63 个，于是 0.63。排行榜上 0.63 排在 0.61 前面，故事结束。

但 0.63 是个被压扁的数字。它把「这个模型能把长流程任务一步步做完」「它的手够不够稳能拧上螺母」「它会不会推着底盘走到桌子另一边」「换一句没见过的指令它还听不听得懂」——全部揉成了一个标量。两个都拿 0.63 的模型，可能强在完全不同的地方。你拿到这个数，什么也学不到，既不知道该往哪改,也不知道它会在什么情况下崩。

从 RL 看 标量回报的信息瓶颈

这其实是你熟悉的问题。在 RL 里，把一个复杂行为压缩成单一标量 reward，会丢掉大量结构信息——这正是 reward shaping、multi-objective RL、以及各种 diagnostic / probing 方法想补回来的东西。EBench 做的事，类比过来就是：不要只看 return，要看 return 在各个能力轴上的分解。

把一个数摊成一张画像

EBench 的全称是 Elemental Mobile Manipulation Benchmark。它的核心主张写在 README 的副标题里：

“From a single success rate to a multi-axis capability profile.”

也就是把黑盒的 success rate，变成一张可读的 capability profile——一张沿多个能力轴展开的雷达图。下面这个（示意）雷达图就是这个想法的样子：同样的总分，画像可以完全不同。

为什么是「mobile manipulation」

EBench 想覆盖三类通常被分开研究的操作范式，放进同一个 benchmark：

  • long-horizon——多步、有先后依赖的长流程任务（开微波炉 → 放两个蛋挞 → 关门）。
  • dexterous & precise——需要精细对位的灵巧操作（把齿轮套上轴并与另两个齿轮啮合）。
  • mobile——带移动底盘，机器人要先走过去再操作。

这三类背后是完全不同的能力短板。把它们塞进一个 success rate 里尤其浪费信息；分轴诊断的价值在这里最大。所有任务都跑在同一个本体上——lift2：双臂 + 移动底盘 + 4 个 480×640 相机（我们会在 1.3 里仔细拆它）。

还有一件事：别在测试集上自欺

capability profile 要可信，前提是数字反映真实泛化，而不是对评测分布的过拟合。EBench 因此严格区分 val_train / val_unseen（开放，可调）与留出的 test（驱动 leaderboard）。这条「训练/测试隔离」的红线，我们在 3.2 详谈。

一个口径提醒（会贯穿全书）

EBench 的 README / 即将发布的 paper 用「5-axis atomic diagnostic」「794 task instances」「Test-Mini」这类愿景语言来描述能力画像；而当前线上 docs 站给出的可验证事实是「3 tracks × 3 splits、26 tasks、数据集 6,600 episodes」，且 split 细分仍标注 WIP。本书会把愿景（为什么要多轴诊断）和地面真相（代码与 docs 里确实存在的东西）分开标注，不让你混淆。

这一节你需要带走的

EBench 的出发点不是「又一个 benchmark」，而是一个测量哲学：把能力拆开来看。记住这三件事就够进入下一节了——(1) 单一 success rate 信息太少；(2) EBench 用 capability profile 沿多轴展开；(3) 它横跨 long-horizon / dexterous / mobile 三种范式，并用训练/测试隔离守住可信度。
