Primitive 层：把物理态翻译成场景语义

9-4 留了一个悬念：TaskWorldState 有三个 schema-free 通道 process / hazards / procedure，由 scenario pack 自己填。可怎么填？总不能让每个 pack 手写一坨 if-else 去读 CanonicalPhysicalState 里的物体位姿。EOS 给出的答案是 Primitive 层——三个极简的抽象基类，每个只有一个方法 update(physical_state) -> dict。它们就是「把后端归一化的物理量，翻译成场景能懂的语义」这件事的标准插槽。整层 core 代码只有一个文件、三个 ABC。

问题：物理态太低级，任务语义太高级

设想一个倒液体的任务。CanonicalPhysicalState 告诉你的是：杯子的位姿、机械臂的关节角、有哪些接触点。但 evaluator 想知道的是更高一层的事实——「现在杯里有多少微升」「瓶盖拧上了没」「门铰链开到多少度」「流程走到第几步」。这两层之间隔着一道翻译：从「几何/接触的原始数字」推出「场景的离散或连续语义」。

这道翻译不能进 core。还记得 9-1 那条硬规则吗——core 里不许出现 liquid_volume_ul、pipette 这类场景私名。所以 core 只能定义翻译的形状（一个抽象基类、一个签名），而把具体翻译逻辑全部推到 scenario_packs/。Primitive 层就是这个「只定形状」的契约。

三个原语基类

core 按翻译的对象类型把原语分成三类，每个都是一个 ABC，每个都只强制一个抽象方法 update：

core/primitives/base.py
class InteractionPrimitive(ABC):
    """Generic primitive for contact/interaction semantics.

    Examples across scenarios: door hinge, threaded cap, plug insertion,
    part alignment, terrain contact, tool-object contact.
    """
    primitive_id: str

    @abstractmethod
    def update(self, physical_state: CanonicalPhysicalState) -> dict[str, Any]:
        """Return scenario-specific state inferred from physical state."""


class ProcessPrimitive(ABC):
    """Generic primitive for process/material/task dynamics.

    Examples: material transfer proxy, field state, part assembly status,
    terrain traversability, crop state, conveyor process.
    """
    primitive_id: str

    @abstractmethod
    def update(self, physical_state: CanonicalPhysicalState) -> dict[str, Any]:
        """Return process state."""


class DevicePrimitive(ABC):
    """Generic state machine for instruments, appliances, factory devices, etc."""
    primitive_id: str

    @abstractmethod
    def update(self, physical_state: CanonicalPhysicalState) -> dict[str, Any]:
        """Return device state."""

三类的分工，从 docstring 的例子里读得很清楚：

原语类翻译什么docstring 给的通用例子
InteractionPrimitive接触 / 交互语义（两个东西之间发生了什么）门铰链、螺纹瓶盖、插头插入、零件对位、地形接触、工具-物体接触
ProcessPrimitive过程 / 物料 / 任务动态（一个量随时间怎么变）物料转移代理、场状态、装配进度、地形可通过性、作物状态、传送带过程
DevicePrimitive器械 / 家电 / 工厂设备的状态机仪器、家电、工厂设备的离散状态

注意例子横跨了科学实验室、工厂、野外、居家——这是 9-1 「Core Entry Test」的体现：一个概念只有对至少两个场景有用，才配进 core。「门铰链开了多少」「瓶盖拧没拧上」「液体转移了多少」「设备在哪个 mode」——这些结构在无数场景里反复出现，所以它们的 ABC 进 core；而「液体到底叫 liquid_volume_ul 还是别的」「门叫什么 id」，是 pack 自己的事。

update：唯一的动作，就是「读物理、吐语义」

三个 ABC 长得几乎一样，核心都是那个 update(physical_state) -> dict。它的契约简单到一句话：输入是 core 认识的 CanonicalPhysicalState，输出是一个场景私有的字典。这个字典正好就是要塞进 9-4 那三个 schema-free 通道的内容——InteractionPrimitive / DevicePrimitive 的产物多半进 procedure 或 hazards，ProcessPrimitive 的产物多半进 process。

于是一个 scenario pack 计算 TaskWorldState 的完整链路是这样的（概念示意，具体 pack 自己实现）：

scenario_packs/<name>/...（概念示意）
# 一个具体过程原语，住在 scenario pack 里，不进 core
class LiquidVolumeProxy(ProcessPrimitive):
    primitive_id = "liquid_volume_proxy"

    def update(self, physical_state):
        # 从杯子位姿 / 倾角等几何量，推一个体积代理值
        tilt = _estimate_tilt(physical_state.objects["beaker"])
        return {"liquid_volume_ul": _proxy_volume(tilt)}

# pack 用一组原语，拼出 schema-free 通道
def compute_task_world_state(physical):
    process = {}
    for prim in self.process_primitives:        # 各 ProcessPrimitive
        process.update(prim.update(physical))
    procedure = {}
    for prim in self.interaction_primitives:    # 各 InteractionPrimitive / DevicePrimitive
        procedure.update(prim.update(physical))
    return TaskWorldState(physical=physical, process=process, procedure=procedure)

要强调的是：上面的 LiquidVolumeProxy 是示意。core 里只有三个抽象基类，没有任何一个具体原语——具体的液体体积代理、螺纹瓶盖判定、离心机状态机，全部住在 scenario_packs/<name>/ 下。这正是 9-1 那条红线在 Primitive 层的样子：scripts/check_core_leakage.py 会确保 core 的 primitives 目录里永远不会冒出 pipette 或 liquid_volume 这种名字。

从 GAMES103 看 从原始 contact / 位姿里「读出」高层谓词

你在物理引擎里拿到的是最原始的东西：两个 collider 的接触法向力、刚体的位姿和速度。但「抓稳了吗」「瓶盖咬合了吗」「门到位了吗」这些是高层谓词，引擎不会直接告诉你——你得自己从几何和接触里推。InteractionPrimitive.update 干的就是这件事：吃一帧物理量，吐一个「接触级别的语义判断」。这跟你在 GAMES103 里从 penetration / contact 推断「碰撞是否发生、约束是否满足」是同一种思维——只不过这里推的是任务语义，不是数值约束。

从 RL 看 这是可插拔、可复用的 state abstraction / feature extractor

RL 里你常给 raw observation 加一层特征工程或 state abstraction，把高维原始量压成对决策/评估有意义的低维语义（「球过了网没」「门开了没」）。Primitive 层就是把这层特征抽取标准化、可插拔化了：core 定签名，scenario pack 提供具体 extractor，同一套 evaluator 因此能复用在无数场景上。区别在于它的产物不是喂给 policy，而是喂给评测——它是 evaluator 的「语义传感器」。

为什么值得专门做一层抽象

你可能会问：每个 pack 直接写个函数读物理态不就行了，何必搞三个 ABC？价值在于统一契约带来的复用。因为所有原语都遵守 update(physical) -> dict 这一个签名，runtime 和 evaluator 就能不认识具体场景地驱动它们：「把这一帧物理态喂给 pack 注册的所有原语，收集它们的输出，填进 TaskWorldState」——这段逻辑可以写在通用层，对实验室、工厂、野外一视同仁。换句话说，Primitive 层把「场景语义计算」从一堆散落的私有函数，收敛成一个有名字、有契约、可被通用代码调度的扩展点。这就是 9-1 那句「场景全是可插拔的 pack」在状态计算上的兑现。

「EBench 做到了吗？」

没有这一层抽象。EBench 的成功判定、子目标判定，是直接烤进任务脚本的——每个任务在 GenManip 的场景配置里写死自己的 goal 条件和部分分逻辑（见 3.3 的 26 任务与打分），score∈[0,1] 由这些脚本算出。这对一个聚焦的 manipulation benchmark 完全够用、甚至更直接。但它没有一个可复用的「物理态 → 场景语义」原语抽象层：没有 InteractionPrimitive / ProcessPrimitive / DevicePrimitive 这种跨场景共享的契约，于是「门铰链判定」「螺纹咬合判定」这类逻辑在不同任务间无法以统一接口复用，换一个全新领域（比如湿实验安全）就得从头写一套判定。EOS 要承载很多领域，才需要把这层翻译标准化成可插拔原语。

这一节你需要带走的

(1) Primitive 层解决「物理态太低级、任务语义太高级」之间的翻译，是 9-4 三个 schema-free 通道的填充器；(2) core 只定义三个 ABC——InteractionPrimitive（接触/交互）、ProcessPrimitive（过程/物料）、DevicePrimitive（设备状态机），每个只有一个 update(physical_state) -> dict；(3) 具体原语（如液体体积代理）一律住在 scenario_packs/，core 永远不出现场景私名，由 leakage checker 守住；(4) 统一签名让通用 runtime/evaluator 能不认识场景地调度原语、复用同一套评测逻辑。下一节进入 Adapter 层：BackendAdapter 与 ModelAdapter 这两道窄契约，如何让 EOS 插上任意 simulator、插上任意模型族。
