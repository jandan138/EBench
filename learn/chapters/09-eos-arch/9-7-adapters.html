Adapter 层：BackendAdapter 与 ModelAdapter

到此 core 的「评什么 / 什么状态 / 什么证据」都齐了，但还差最后两扇门：外面那个具体的 simulator 怎么进来？外面那个具体的 model 怎么进来？EOS 的回答是两道窄契约——BackendAdapter 与 ModelAdapter。窄，是它们的全部美德：契约越窄，能插进来的东西就越多。这一节我们逐方法读这两个 ABC，看它们如何分别成为「通往任意仿真」和「通往任意模型族」的防腐门。

BackendAdapter：通往任意 simulator 的防腐门

先看后端。BackendAdapter 是一个 ABC，它的 docstring 把边界写得很硬：「This must not contain scenario-specific benchmark logic. 它只负责加载场景、执行动作、暴露归一化的状态/观测。」——这正是 9-1 说的 anti-corruption layer：把外部仿真的脏假设挡在门外，只让归一化的东西进来。

backends/base.py
@dataclass
class BackendStepResult:
    observation: dict[str, Any]
    reward: float = 0.0
    terminated: bool = False
    truncated: bool = False
    info: dict[str, Any] = field(default_factory=dict)


class BackendAdapter(ABC):
    backend_profile_id: str

    @abstractmethod
    def load_scene(self, scene_spec: SceneSpec) -> None: ...

    @abstractmethod
    def reset(self, seed: int | None = None,
              initial_state: dict[str, Any] | None = None) -> dict[str, Any]: ...

    @abstractmethod
    def step(self, action: dict[str, Any] | list[float]) -> BackendStepResult: ...

    @abstractmethod
    def get_observation(self) -> dict[str, Any]: ...

    @abstractmethod
    def get_canonical_state(self) -> CanonicalPhysicalState: ...

    def set_state(self, state: CanonicalPhysicalState) -> None:
        raise NotImplementedError("This backend does not support set_state().")

    def close(self) -> None:
        pass

五个必须实现的方法，把一个仿真器对评测唯一需要暴露的能力压到了最小：load_scene（按 SceneSpec 摆好场景）、reset（按 seed 复位）、step（执行一个动作、回一个 BackendStepResult）、get_observation（吐模型能看的观测）、get_canonical_state（吐 9-4 那个后端归一化的 CanonicalPhysicalState）。注意这个 ABC对任务一无所知——它不知道什么叫成功、不算分、不读 task spec。它是「无状态于任务」的：成败判定属于 evaluator，它只管把世界往前推一步、如实报告状态。

两个非抽象方法暴露了设计的取舍。set_state 默认直接 raise NotImplementedError——因为「把世界设到任意状态」是很多真实仿真（尤其真机）做不到的强能力，所以它不是必需，而是「能做就重写、做不到就如实拒绝」。这种「默认抛 NotImplemented、可选覆盖」的写法，是 EOS 表达「可选能力」的统一手法，等下在 model 那边会再见到它。

来看最小实现 DummyBackend——一个确定性的 smoke 后端，它一行 Isaac、一行 MuJoCo 都没有，却能跑通整条评测链路：

backends/dummy.py
class DummyBackend(BackendAdapter):
    """Minimal deterministic backend for smoke tests."""
    backend_profile_id = "dummy_backend"
    backend_components = {
        "platform": "dummy_backend", "physics": "semantic_proxy",
        "renderer": "none", "sensors": "none",
        "control": "direct_delta", "process_runtime": "none",
    }

    def step(self, action):
        self._t += 1
        delta = float(action.get("delta", 0.0)) if isinstance(action, dict) else (float(action[0]) if action else 0.0)
        self._position += delta
        obs = self.get_observation()
        return BackendStepResult(observation=obs,
                                 reward=1.0 if self._position >= 1.0 else 0.0,
                                 terminated=self._position >= 1.0,
                                 info={"position": self._position})

    def get_canonical_state(self) -> CanonicalPhysicalState:
        return CanonicalPhysicalState(time_s=float(self._t),
                                      objects={"dummy_object": {"position": self._position}},
                                      metadata={"scene_id": self._scene_id})

它把「世界」简化成一个标量 position，动作就是给它加 delta，到 1.0 就算 terminated。但请注意它已经把契约填全了：它声明了 backend_profile_id，声明了 9-3 的六件套 backend_components（这些会被 9-5 的 trace 原样记下），并且 get_canonical_state 真的吐出了归一化的物理态。这就是窄契约的力量——一个真实仿真器要接入，无非是把这五个方法换成真的 Isaac / MuJoCo 调用，core 一行不改。DummyBackend 本身是已实现的；真实后端 provider（MuJoCo / Genesis / USD-Isaac）的接入则属于 9-8 会讲的 WorldState runtime 主线工作，目前多为 WIP / smoke 阶段。

ModelAdapter：一个模型可以同时是好几种角色

另一道门是模型。ModelAdapter 的 docstring 一句话点题：「不同模型可以实现一个或多个方法：actor policy、world predictor、planner、safety critic、autonomous agent。」——也就是说，模型族不是用「继承哪个类」来区分的，而是用「实现了哪几个方法」来区分的。

model_adapters/base.py
class ModelAdapter(ABC):
    model_id: str

    def reset(self, task_context: dict[str, Any]) -> None:
        pass

    @abstractmethod
    def act(self, observation: dict[str, Any]) -> dict[str, Any] | list[float]:
        pass

    def predict_future(self, observation, action_sequence) -> PredictedFuture:
        raise NotImplementedError

    def rank_plans(self, observation, candidate_plans) -> list[Any]:
        raise NotImplementedError

    def assess_safety(self, observation, candidate_action) -> dict[str, float]:
        raise NotImplementedError

只有 act 是 @abstractmethod——必须实现。因为「给观测、出动作」是 actor 的最低门槛，是所有被评模型的公约数。其余三个方法用的还是那招「默认 raise NotImplementedError、能力可选」：实现了 predict_future，你这个 adapter 就额外是个世界模型，可以被 prediction 指标族评（它返回的正是 9-4 那个 PredictedFuture）；实现了 rank_plans，就额外是个 planner；实现了 assess_safety，就额外是个 safety critic。一个模型可以同时是 actor + world model + planner + safety critic——四顶帽子戴在同一个 adapter 上。model-adapter-contract.md 把这条说死：「a model adapter should expose only the methods it supports，不支持的就抛清晰的 NotImplementedError 并上报 capability 元数据。」

最小实现 RandomPolicyAdapter 只实现了 act——它就是个纯 actor，恒定吐一个 delta：

model_adapters/random_policy.py
class RandomPolicyAdapter(ModelAdapter):
    model_id = "random_policy"

    def __init__(self, delta: float = 0.25) -> None:
        self.delta = delta

    def act(self, observation: dict) -> dict[str, float]:
        return {"delta": self.delta}

它没实现 predict_future / rank_plans / assess_safety，于是这三个能力自动落回基类的 NotImplementedError——runtime 据此就知道「这个模型只能当 actor，别拿 prediction 指标族去评它」。这套「方法即能力声明」的设计，让加一个新模型族（比如未来的 autonomous agent）不需要改 core：只要它能写进 EpisodeTrace 且至少映射到一个指标族即可（model-adapter-contract.md §10 的入场条件）。

从 RL 看 model-free 和 model-based 共用一套接口

RL 里 model-free（只学 policy）和 model-based（还学 dynamics model）通常是两套完全不同的代码。EOS 用一个 ModelAdapter 把它们统一了：act 是 policy（model-free 的全部），predict_future 是 dynamics model（model-based 多出来的那块）。同一个对象可以两者都有。这恰好回应了 9-4 的 PredictedFuture——评测系统这边给世界模型留好了被评的位置，模型这边就给世界模型留好了产出预测的方法，两端对齐。

为什么「窄」是优点

把两道门并排看：BackendAdapter 五个方法、ModelAdapter 一个必需方法。契约窄到这个程度，好处是组合爆炸地省事——任意一个满足 5 方法契约的后端，配上任意一个满足 act 契约的模型，就是一次合法评测；而它们之间的桥梁（spec、canonical state、trace）全是 core 已经定好的。这就是 9-1 「domain-agnostic core + backend-composable」最终在接口层的兑现：core 不认识 Isaac、不认识 π0，它只认识「能 step 的后端」和「能 act 的模型」。

「EBench 做到了吗？」

一半做到了，而且做得很好；另一半结构上没有。先说做到的：EBench 有清晰的模型契约——EvalClient + ModelClient 那套 reset / get_action / step / close（见 4.2），这本身就是一道漂亮的模型侧防腐层，让模型装在自己的 env 里、不和 server 抢依赖。但两点关键差异：其一，EBench 没有可替换的 BackendAdapter——它的后端焊死在 GenManip / Isaac Sim 上（server 就是把 Isaac 当黑盒，见 4.1、2.1），换不了 MuJoCo、换不了真机。其二，它的模型契约是 actor-only：只有 get_action / get_action_chunk，没有 predict_future / assess_safety——因为 EBench 评的就是 actor 的任务成败，不评世界模型、不评安全 critic。再次强调：对一个 Isaac-only、评 actor 的 benchmark，这套设计精准且够用；EOS 要插任意 sim、评任意模型族，才必须把后端也抽成可换的 adapter、把模型契约扩成多能力可选。

这一节你需要带走的

(1) BackendAdapter 用五个方法（load_scene/reset/step/get_observation/get_canonical_state）把仿真器对评测的暴露压到最小，对任务一无所知，是通往任意 simulator 的防腐门；set_state 默认抛 NotImplemented 表达「可选强能力」；DummyBackend 是已实现的最小后端。(2) ModelAdapter 只强制 act，predict_future/rank_plans/assess_safety 用「默认抛 NotImplemented、可选覆盖」表达「一个模型可同时是 actor / world model / planner / safety critic」；RandomPolicyAdapter 是纯 actor。(3) 契约越窄越能组合：任意后端 × 任意模型 = 一次合法评测，core 不认识具体 sim 或模型。下一节是 Part 9 的收束：Evaluator 与 9 个 MetricFamily，加上 Runtime 的分层——从简单同步的 EpisodeRunner，到正在推进的 WorldState runtime 主线，并用 run_smoke.py 把整条链路真的跑一遍。
