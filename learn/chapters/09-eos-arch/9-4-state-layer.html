State 层：从物理量到任务世界态

上一节我们把后端拆成了可换的组件——但这立刻带来一个问题：MuJoCo 吐 qpos，Isaac 吐 articulation joint state，两边的数组布局、命名、单位都不一样。如果 evaluator 直接读后端原生量，那「换 physics 引擎」就会连带逼着 evaluator 改写。State 层就是为了切断这条耦合：它定义一套后端归一化的世界态，让同一个 evaluator 通吃所有后端。整层只有一个文件——core/state/world_state.py，三个 dataclass。

CanonicalPhysicalState：后端归一化的物理量

最底层是 CanonicalPhysicalState——「canonical」就是「标准/正典」的意思：不管哪个引擎跑出来的物理，最终都要被翻译成这一套结构。

core/state/world_state.py
@dataclass
class CanonicalPhysicalState:
    """Backend-normalized physical state.
    Examples: object poses, robot joints, velocities, contacts, sensor states.
    """
    time_s: float
    objects: dict[str, Any] = field(default_factory=dict)
    robots: dict[str, Any] = field(default_factory=dict)
    joints: dict[str, Any] = field(default_factory=dict)
    contacts: list[dict[str, Any]] = field(default_factory=list)
    metadata: dict[str, Any] = field(default_factory=dict)

它只装普适的物理概念：时间、物体、机器人、关节、接触。注意它不是 frozen——和 spec 层的 frozen dataclass 不同，state 是运行时随每一步变化的测量结果，所以可变。也注意字段全是 dict / list[dict]：core 不规定关节叫什么名、物体怎么编号，那些是后端私有的，由 adapter 在翻译时填进来。core 只保证形状——「有一个叫 joints 的字典」——而不保证内容的命名空间。

从 GAMES103 看 qpos 是引擎私有编码，世界态是物理事实

你写过物理求解器，知道 qpos/qvel 只是某个引擎对「广义坐标」的一种打包方式——关节顺序、四元数约定、根关节是否含在内，每个引擎都不一样。但「这个杯子在世界系的位姿」是一个不依赖引擎的物理事实。CanonicalPhysicalState 存的就是后者：把引擎私有的编码，归一到事实层。这样 evaluator 关心的是「杯子到没到位」，而不是「第 7 个 qpos 分量是多少」。

TaskWorldState：物理 + 三个 schema-free 通道

纯物理量还不够描述一个任务的世界。一次倒液体的任务，evaluator 想知道的不止是杯子位姿，还有「现在杯里有多少微升」「有没有触发溢出危险」「流程走到第几步」。但这些语义是场景特定的——core 不能认识「液体体积」。于是 TaskWorldState 把它们做成三个schema-free 字典，由 scenario pack 自己填：

core/state/world_state.py
@dataclass
class TaskWorldState:
    """Canonical physical state plus scenario-specific process/domain state.
    Core does not assume any one scenario, setting, simulator, or benchmark.
    """
    physical: CanonicalPhysicalState
    process: dict[str, Any] = field(default_factory=dict)    # 如 liquid_volume_ul
    hazards: list[dict[str, Any]] = field(default_factory=list)
    procedure: dict[str, Any] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

这是整个 State 层最体现 9-1 哲学的地方：physical 是 core 认识的归一化物理；而 process / hazards / procedure 三个通道，core 只规定它们存在、不规定里面有什么。科学实验室 pack 往 process 里塞 {"liquid_volume_ul": 250}，湿实验安全 pack 往 hazards 里塞危险事件——而 core 永远不会出现 liquid_volume_ul 这个名字（否则 check_core_leakage.py 不会因为它报错，但它会违背「core 不认识场景语义」的精神，这类语义按约定只能从 pack 填入）。这就是「schema-free」的力量：用一个开放字典，把无穷多种场景语义挡在 core 之外，又让它们有处可放。

PredictedFuture：让世界模型也能被评

到这里 State 层还只描述「现在」。但 EOS 的野心（README §0）不止评 task execution，还要评 world prediction——世界模型 / WAM 对未来的预测能力。这需要一个表示「模型想象出来的未来」的结构，就是 PredictedFuture：

core/state/world_state.py
@dataclass
class PredictedFuture:
    """Model-predicted future for WAM/WM/planner evaluation."""
    horizon_s: float
    predicted_states: list[TaskWorldState] = field(default_factory=list)
    predicted_observations: list[dict[str, Any]] = field(default_factory=list)
    success_probability: float | None = None
    risk_scores: dict[str, float] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

它的字段透露了 EOS 想问世界模型的几类问题：你预测的未来世界态序列（predicted_states，注意类型就是一串 TaskWorldState——预测和真实用同一套表示，才能直接对齐比对）准不准？你预测的未来观测（predicted_observations，比如想象出的未来帧——还记得 6.3 InternVLA-A1 的 visual foresight 吗）像不像？你给的成功概率和风险分校准吗？

从 RL 看 评 model，不只评 policy

RL 里你分得清 model-free（直接学 policy/value）和 model-based（学一个 dynamics model 再用它规划）。绝大多数 manipulation benchmark 只评 actor——给观测、收动作、看成败。PredictedFuture 让 EOS 能评另一类对象：dynamics model 本身。架构文档里专门画了一条 WAM 数据流——模型 predict_future() 出一个 PredictedFuture，backend 真跑出 ground-truth 未来 TaskWorldState，两者相减喂给 prediction / safety / planning 指标族。这正是 9.2 那 9 个 MetricFamily 里 prediction / planning 两族存在的理由。

归一化为什么是地基

把这三层串起来看：CanonicalPhysicalState 抹平后端差异，TaskWorldState 用 schema-free 通道接住场景语义，PredictedFuture 复用同一套表示来评世界模型。归一化在这里不是洁癖，而是让上一节的可组合性真正可用的前提：因为物理量被归一了，「换 physics 引擎、evaluator 不动」才成立；因为预测和真实共用 TaskWorldState，「同一个指标既能评 actor 又能评 world model」才成立。State 层是 spec 与 evaluator 之间那块承重的地板。

「EBench 做到了吗？」

没有归一化状态层，也没有 predicted-future 通道。EBench 的观测直接来自 Isaac：client 收到的就是 state.joints(12) / state.gripper(4) / state.base(3) / state.ee_pose 这些后端原生量（见 4.2 的 obs 契约），打分逻辑也直接读这些 Isaac-native 量。它没有一个像 CanonicalPhysicalState 的归一化抽象——因为它本来就只有一个后端，不需要抹平差异；也没有 PredictedFuture 这种通道，因为 EBench 评的是 actor 的 task 成败（score∈[0,1]，track 取 mean），不评世界模型的未来预测。再次强调：单后端、评 actor 的 benchmark 这样做完全够；EOS 要跨后端、还要评 world model / planner，才必须先把状态归一、再给「想象的未来」留一个一等公民的位置。

这一节你需要带走的

(1) CanonicalPhysicalState 把各后端的私有物理编码（qpos / Isaac joint state…）归一成事实层的世界态，让 evaluator 不再绑后端；(2) TaskWorldState = 归一物理 + process / hazards / procedure 三个 schema-free 字典，把场景语义（如 liquid_volume_ul）挡在 core 外又有处可放；(3) PredictedFuture 用同一套表示承载「模型想象的未来」，让 world model / planner 也能被评，对应 prediction/planning 指标族；(4) 归一化是让上一节可组合性、跨 actor/world-model 评测真正落地的地基。下一节进入 Trace 层：所有这些 spec、state、prediction、metric 最终怎么沉淀成唯一的证据 EpisodeTrace，以及 privileged state 如何被审计。
