Evaluator、9 个 MetricFamily 与 Runtime 分层

spec、state、trace、primitive、adapter 都齐了——现在到了把它们跑起来、打成分的最后一环。这一节有三块：Evaluator（怎么从 trace 算指标）、9 个 MetricFamily（指标怎么分类）、以及 Runtime 分层（怎么把一切驱动起来）。Runtime 这块要特别诚实：EOS 有一个简单、已实现的同步 EpisodeRunner，和一条正在推进的、远更宏大的 WorldState runtime 主线——两者并存，我们会把它们分清楚。这是 Part 9 的收束，我们最后用 run_smoke.py 把整条链路真的跑一遍。

Evaluator：从 trace 读证据，吐指标

9-5 说过 trace 是唯一证据。那么打分就只能是一件事：从 trace 里读证据，算出指标。Evaluator 这个 ABC 把它定成两个粒度——evaluate_step（逐步，可选）和 evaluate_episode（整集，必需）：

core/evaluators/base.py
@dataclass
class MetricResult:
    name: str
    value: float | int | bool | str
    family: str
    unit: str | None = None
    details: dict[str, Any] = field(default_factory=dict)

@dataclass
class EpisodeResult:
    episode_id: str
    task_id: str
    metrics: list[MetricResult] = field(default_factory=list)
    summary: dict[str, Any] = field(default_factory=dict)

class Evaluator(ABC):
    evaluator_id: str

    def evaluate_step(self, state: TaskWorldState, **kwargs: Any) -> list[MetricResult]:
        return []

    @abstractmethod
    def evaluate_episode(self, trace: EpisodeTrace) -> EpisodeResult:
        pass

关键在 evaluate_episode 的签名：它吃的就是一个 EpisodeTrace，吐一个 EpisodeResult（一组 MetricResult + 一份 summary）。注意每个 MetricResult 都带一个 family 字段——这就是它归属的那个指标族。core 自带三个通用 evaluator：TaskOutcomeEvaluator、PredictionEvaluator、HazardEvaluator，而领域特定的（如「液体体积误差」）由 scenario pack 添加。看 HazardEvaluator 怎么不认识任何具体危险地工作：

core/evaluators/hazard.py
class HazardEvaluator(Evaluator):
    """Scenario packs define hazard event types and severity scales."""
    evaluator_id = "hazard"

    def evaluate_episode(self, trace: EpisodeTrace) -> EpisodeResult:
        hazard_events = []
        for step in trace.steps:
            if step.world_state:
                hazard_events.extend(step.world_state.hazards)   # 9-4 的 hazards 通道
        max_severity = max((int(e.get("severity", 0)) for e in hazard_events), default=0)
        return EpisodeResult(
            episode_id=trace.episode_id, task_id=trace.task_id,
            metrics=[MetricResult("hazard_event_count", len(hazard_events), family="hazard"),
                     MetricResult("max_hazard_severity", max_severity, family="hazard")],
            summary={"hazard_event_count": len(hazard_events),
                     "max_hazard_severity": max_severity})

它只是遍历每个 TraceStep 的 world_state.hazards（9-4 那个 schema-free 通道，由 9-6 的原语填），数一数、取个最大 severity——什么算危险、severity 怎么定，全是 pack 的事，core 只做通用聚合。TaskOutcomeEvaluator 同理：它读 trace.episode_metrics 或最后一步的 metrics 里那个 success 谓词，core 不规定成功的具体判据。PredictionEvaluator 在 core 里目前是个 placeholder——只数「有多少步带了 prediction」，真正的 state/video/hidden-state 预测指标留给 scenario pack 注册。这三个都是已实现但刻意通用的骨架。

9 个 MetricFamily：core 定族，pack 实例化

指标怎么分类？metric-system.md 的原则很干脆：core 定义 metric family，scenario pack 实例化具体 metric。一个 family 是抽象的、绝不提具体场景；具体的 liquid_volume_error_ul 这种实例属于 pack。core 的 family 是一个 str 枚举：

core/specs/metric_spec.py
class MetricFamily(str, Enum):
    TASK_OUTCOME = "task_outcome"
    PROCEDURE = "procedure"
    STATE_TRANSITION = "state_transition"
    PRECISION = "precision"
    HAZARD = "hazard"
    PREDICTION = "prediction"
    PLANNING = "planning"
    GENERALIZATION = "generalization"
    RESOURCE = "resource"


@dataclass(frozen=True)
class MetricSpec:
    """Core defines metric families. Scenario packs instantiate concrete metrics."""
    name: str
    family: MetricFamily
    scope: str = "episode"
    target: str | None = None
    unit: str | None = None
    parameters: dict[str, Any] = field(default_factory=dict)
    weight: float = 1.0
    metadata: dict[str, Any] = field(default_factory=dict)

九个族各管一个维度：

MetricFamily评什么
task_outcome成功、进度、完成质量
procedure步骤顺序、协议合法性、恢复行为
state_transition动作引起的状态变化
precision位姿/量/力/时序/对位的精度
hazard危险、安全边界、漏报、严重度
prediction未来状态/视频/隐藏态的预测质量（评 world model）
planning规划有用性、想象 vs 真实的差距
generalization跨对象/背景/指令的泛化
resource时间、动作数、能耗、计算、干预

注意 prediction / planning 两族正是 9-4 PredictedFuture、9-7 predict_future / rank_plans 存在的理由——它们让评测能对准 world model 和 planner，而不只是 actor。

一个源码 vs 文档的口径差异（如实标注）

枚举里第 8、9 个是 generalization 和 resource。但设计文档 metric-system.md §1 列的是 restoration（失败后的恢复修复）和 resource，没列 generalization。也就是说文档与代码此处对不齐：以真实枚举为准，当前 core 里的九族是上表那九个，含 generalization、不含 restoration。这类「文档愿景 vs 代码地面真相」的漂移，本书一贯如实标注、以代码为准。

从 RL 看 不是一个 scalar return，是九维向量

RL 习惯把行为压成一个标量 return。这九个 family 等于明确拒绝了这种压缩：把「成败」「过程合法性」「精度」「安全」「预测」「规划」「泛化」「资源」拆成九个正交维度分别记录——这正是 0.1 那张 capability profile 的思想，被 EOS 做成了 core 里写死的指标分类法。metric-system.md §7 甚至明令报告「避免单一 scalar 作为主结果」，要出一张多轴 profile。

Runtime（一）：简单同步的 EpisodeRunner（已实现）

谁来驱动「reset → act → step → 记 trace」这个循环？最直接的答案是 EpisodeRunner——它把 backend + model + task_spec + scene_spec 攒在一起，run(seed) 跑出一个 EpisodeTrace：

core/runtime/episode_runner.py
@dataclass
class EpisodeRunner:
    backend: EpisodeBackend
    model: EpisodeModel
    task_spec: TaskSpec
    scene_spec: SceneSpec
    max_steps: int = 100

    def run(self, seed: int | None = None) -> EpisodeTrace:
        self.backend.load_scene(self.scene_spec)
        obs = self.backend.reset(seed=seed)
        self.model.reset({"task": self.task_spec})

        trace = EpisodeTrace(
            episode_id=str(uuid.uuid4()), task_id=self.task_spec.task_id,
            scenario_pack=self.task_spec.scenario_pack,
            backend_profile_id=self.backend.backend_profile_id,
            model_profile_id=getattr(self.model, "model_id", None), seed=seed,
            versions={"repo_commit": _current_repo_commit(),       # 真去 git rev-parse HEAD
                      "task_spec": self.task_spec.task_id,
                      "scene_spec": self.scene_spec.scene_id},
            backend_components=_backend_components(self.backend))

        for step_index in range(self.max_steps):
            action = self.model.act(obs)
            result = self.backend.step(action)
            canonical = self.backend.get_canonical_state()
            trace.append(TraceStep(
                step_index=step_index, time_s=canonical.time_s,
                observation=obs, action=action,
                canonical_physical_state=canonical,
                task_world_state=TaskWorldState(physical=canonical),
                metric_results={"terminated": result.terminated,
                                "truncated": result.truncated, "reward": result.reward},
                privileged_state_visible_to_model=False))
            obs = result.observation
            if result.terminated or result.truncated:
                trace.episode_metrics["task_success"] = result.terminated
                break

这段代码把前几节全串起来了：它驱动 9-7 的 act / step / get_canonical_state，把 9-4 的 canonical state 包成 TaskWorldState，逐步写进 9-5 的 trace，并且老老实实把 privileged_state_visible_to_model=False 标在每一步。尤其值得点名的是 _current_repo_commit()——它真的去 git rev-parse HEAD 把当前 commit 记进 versions，这就是 9-5 那条「没有版本号就不是有效证据」的兑现。

但要诚实：EpisodeRunner 是单进程、同步的。它没有 EBench 那种内建的 client–server / 多 worker 分布式机制——这块被推迟到 adapter 去做（一个外部分布式后端可以包成一个 BackendAdapter）。它是「happy path」的最小可跑骨架，已实现、被 smoke 覆盖，但不是为大规模并发设计的。

Runtime（二）：WorldState runtime 主线（WIP）

这里必须把话说清楚：上面那个 EpisodeRunner 虽然已实现，却不是当前的开发主线。EOS 真正在推进的活，是一条叫 WorldState Runtime Foundation 的 roadmap（WR-1…WR-7），目标是让每一个后端（MuJoCo / Genesis / USD-Isaac / 真机 / Ansys 式多物理）都通过同一套运行时基座来 report / apply / measure / project 世界态，并带显式的 coverage 和 claim boundary。它分成多层（L0 资源 → L2 WorldSession → L3 RuntimeMeasurementSnapshot → L4 投影 → L5 任务语义 → L6 证据账本）。我们看其中两块最能体现其「anti-overclaim」纪律的核心类型。

L2 是 WorldSession——一个运行时信封（runtime envelope），不是「实时世界镜像」。它记的是后端 profile、生命周期、时钟、拓扑引用、domain channel 声明、端口、能力，以及 apply/measure 协议钩子；它不是逐步物理真值的所有者，那是 L3 的事。它的 claim_boundary 默认就写着 "L2 runtime envelope only."。

L3 是 RuntimeMeasurementSnapshot——不可变、artifact-backed 的某个时间/事件边界上的测量证据。它的精髓在 MeasurementCoverageLevel 这个枚举：每一项测量都要标自己的覆盖等级——

core/runtime/runtime_measurement.py
class MeasurementCoverageLevel(str, Enum):
    EXACT = "exact"
    APPROXIMATE = "approximate"
    PARTIAL = "partial"
    READ_ONLY = "read_only"
    OPAQUE_ONLY = "opaque_only"
    UNSUPPORTED = "unsupported"
    BLOCKED = "blocked"

# 内建的「反过度宣称」正则：禁止把 measured evidence 说成 task success / parity / restorability
OVERCLAIM_PATTERN = re.compile(
    r"\b(proves?|claims?|asserts?|guarantees?|certifies?|establishes?)\b"
    r"[^.;,\n]{0,80}\b(task success|parity|restorability)\b" r"|" r"...")

这套设计的偏执之处在于：它在构造 dataclass 时就用 OVERCLAIM_PATTERN 这个正则去扫描 claim_boundary 文本，一旦发现有人想把「测到的证据」吹成「任务成功 / 跨后端 parity / 可还原性」，直接 raise ValueError 让对象建不出来。9-5 那句默认 claim_boundary 是「人写的纪律」，到这一层就升级成了「机器强制的纪律」。还有 ProjectionLinkage——它把「L3 测量 → canonical 投影」做成显式、可门控的：只有当一堆条件（源快照强标识、必需 packet 齐全、没有 blocking 字段……）都满足，ready_for_projection 才为 True，否则这份测量不允许被当成 canonical 真值往下用。

这条主线的真实状态要如实说：roadmap 上 WR-1 到 WR-7 的契约/证据 schema多已 closed，但它们大多是「证据契约」层面的完成——真正的 backend provider 接入仍是 backend-specific 的后续工作；最近的跨后端动态对比（WR-7D，MuJoCo vs Genesis）收尾时是 gate_status=failed，明确不主张 task success、不主张 backend 等价。换句话说：这是一台一任务、一后端对的「诊断显微镜」，是 active 的 WIP 主线，和上面那个简单的 EpisodeRunner 是两条不同的轨道。

从 GAMES103 看 覆盖等级 = 诚实标注「这个量我到底测准了没」

你跑过不同求解器就知道：同一个「接触力」，有的引擎给精确解析接触、有的只给近似、有的干脆不暴露。MeasurementCoverageLevel 就是把这种「我对这个量的把握程度」变成一等数据：EXACT / APPROXIMATE / PARTIAL / OPAQUE_ONLY…… 不再假装所有后端都测得一样准。这正是 GAMES103 训练出的物理直觉——数值结果带着误差与适用范围——被升格成评测系统里强制携带的元数据。

Happy path：run_smoke.py 把整条链路跑一遍

讲了这么多类型，最后用 examples/run_smoke.py 看它们怎么真的串起来跑通（这是已实现、可直接执行的端到端示例）：

examples/run_smoke.py
scene = SceneSpec(scene_id="dummy_scene",
                  asset_instances=[AssetInstance(instance_id="obj", asset_id="dummy_object")])
task = TaskSpec(task_id="smoke.move_to_threshold", scenario_pack="smoke",
                scene_id="dummy_scene",
                instructions=["Move the dummy object to position >= 1.0."],
                success_conditions=["dummy_object.position >= 1.0"])

runner = EpisodeRunner(backend=DummyBackend(),
                       model=RandomPolicyAdapter(delta=0.4),
                       task_spec=task, scene_spec=scene, max_steps=10)
trace = runner.run(seed=42)
result = TaskOutcomeEvaluator().evaluate_episode(trace)
trace.episode_metrics.update(result.summary)
print(f"episode_id={trace.episode_id}")
print(f"steps={len(trace.steps)}")
print(f"metrics={result.summary}")
if args.trace_dir:
    trace_path = TraceStore(args.trace_dir).save_json(trace)   # 落盘成 JSON 证据

读这十几行，你会看到 Part 9 的每一块各就各位：SceneSpec / TaskSpec（spec 层）→ DummyBackend（9-7 后端 adapter）+ RandomPolicyAdapter（9-7 模型 adapter）→ EpisodeRunner（本节 runtime）跑出 EpisodeTrace（9-5 证据）→ TaskOutcomeEvaluator（本节 evaluator）算出 EpisodeResult → TraceStore 落盘成可审计 JSON。这条 happy path 不碰任何真实仿真，却把 domain-agnostic core 的全套契约验证了一遍——这也正是 9-1 提到的 PR 前必跑命令之一。

「EBench 做到了吗？」

这是个互有长短的对照。Runtime 上 EBench 强：它有成熟的分布式 client–server 评测——server 跑 Isaac、client 多 worker 并行、结果落 saved/eval_results/（见 4.1），这是 EOS 目前还没有的（EOS 的 EpisodeRunner 是单进程同步，分布式被推迟到 adapter；WorldState 主线还在打证据基座）。Metric 上 EOS 强：EBench 最终聚到一个 task score∈[0,1]、track 取 mean、leaderboard 聚合的偏标量口径；EOS 则把评测劈成九个正交的 MetricFamily、且能对准 actor / world model / planner / safety critic 多种对象。一句话：EBench 把一件事（Isaac 上评 actor）做得又快又分布式；EOS 把评测的维度与对象铺得很宽，但运行时的工程成熟度还在路上。

这一节、以及整个 Part 9 你需要带走的

(1) Evaluator 从 EpisodeTrace 读证据吐 MetricResult，core 自带通用的 TaskOutcome / Prediction / Hazard，领域指标由 pack 加；(2) core 定义 9 个 MetricFamily（task_outcome / procedure / state_transition / precision / hazard / prediction / planning / generalization / resource，注意与文档的 restoration 漂移、以代码为准），pack 实例化具体 metric，拒绝单 scalar 主结果；(3) Runtime 有两条轨道——已实现的简单同步 EpisodeRunner（无内建分布式，记 repo_commit/seed），与WIP 主线 WorldState runtime（L2 WorldSession 信封、L3 不可变 artifact-backed RuntimeMeasurementSnapshot + MeasurementCoverageLevel + 正则反过度宣称 + 门控的 ProjectionLinkage）；(4) run_smoke.py 把全套契约端到端跑通。

逐层读完 EBench 与 EOS 两套系统，你应该已经能看出它们各自的「为什么这样设计」与「代价是什么」。下一个 Part 我们换一种姿势：给 EBench 一张诚实的记分卡，并认真谈谈——拿着这两套系统的经验，怎么造一个更好的具身评测基准。
