设计立场：domain-agnostic core + anti-corruption layers

前面八个 Part 我们一直在读 EBench——一个具体的、绑死 Isaac Sim 的 manipulation benchmark。现在镜头切换：作者自己又写了一套通用具身评测系统 EmbodiedEval OS（下文简称 EOS）。它的第一行 README 就把立场摆得很硬——"A domain-agnostic, backend-composable evaluation OS for embodied intelligence." 这一节我们先不看任何 dataclass，先想清楚一件事：为什么它要刻意不做成 EBench 的 fork？

它不是什么

EOS 的 CLAUDE.md 里有一段近乎宣言的 "It is not" 列表，把所有最容易掉进去的坑逐条钉死：

CLAUDE.md · Project Identity
It is not:
- an EBench fork;
- a GenManip wrapper;
- an Isaac-only benchmark;
- a scientific-lab-only benchmark;
- a VLA-only evaluator.

为什么要专门写这五条？因为它们正是这个项目当下真实承担的活：作者既要兼容 EBench / GenManip 老管线，又要交付科学实验室、湿实验安全的评测，还想长期覆盖居家、工厂、野外、仓储、真机。三股需求互相拉扯——最省事的写法就是「在 EBench 上加几个 if」，但那样三个月后 core 就会被某一个 benchmark 的假设腌透。EOS 的回答是：科学实验室、居家、工厂、野外、真机，全部都是可插拔的 scenario_pack 或 adapter，没有一个享有进入 core 的特权。

从 RL 看 把 env 接口和具体 env 分开

你在 RL 里早就接受了一条边界：gym.Env 的 reset/step 协议是通用的，而 CartPole、Atari、MuJoCo 各自的物理是实现，绝不会反过来污染 Env 抽象基类。EOS 把这条边界提升成了整个系统的第一原则——core 只定义契约，EBench 只是众多 env 之一，从 adapter 这道门进来。

那条硬规则：core 里不许出现 benchmark 名字

立场要落地，得有一条机器能检查的规则。EOS 的规则非常具体：src/embodied_eval_os/core/ 里不许出现任何 benchmark / scenario / simulator 的私有名字。CLAUDE.md 的 Hard Rules 直接列出黑名单——EBench、GenManip、Lift2、Isaac41、AutoBio、Lab、WetLab、Pipette、Centrifuge、TubeRack、Contamination…… 而它们的去处也写得明明白白：

  • 场景特定概念（pipette、hazard taxonomy、ebench 兼容表）→ 进 scenario_packs/<name>/；
  • 外部 benchmark 概念（GenManip runtime、Lift2 动作、Isaac 4.1 lock-in）→ 进 adapters/<name>/；
  • core 只保留 scenario-neutral 的契约：TaskSpec、MetricSpec、BackendProfile、EpisodeTrace……

这不是口头约定，而是用脚本在 PR 阶段强制执行的。scripts/check_core_leakage.py 会遍历 core/ 下所有 .py，对黑名单逐条正则匹配，命中就 return 1 让 CI 挂掉：

scripts/check_core_leakage.py
FORBIDDEN_PATTERNS = [
    ForbiddenPattern("ebench", _contains_pattern("ebench")),
    ForbiddenPattern("genmanip", _contains_pattern("genmanip")),
    ForbiddenPattern("lift2", _contains_pattern("lift2")),
    ForbiddenPattern("isaac41", _contains_pattern("isaac41", "isaac_41", "isaac 4.1")),
    ForbiddenPattern("autobio", _contains_pattern("autobio", "auto_bio")),
    ForbiddenPattern("mujoco", _contains_pattern("mujoco", "mu_joco")),
    ForbiddenPattern("evalclient", _contains_pattern("evalclient", "eval_client")),
    ForbiddenPattern("lab", _word_pattern("lab", "laboratory")),
    ForbiddenPattern("pipette", _word_pattern("pipette")),
    ForbiddenPattern("centrifuge", _word_pattern("centrifuge")),
    # ... tube_rack / contamination / pcr_plate / vortex_mixer ...
]

def main() -> int:
    bad = find_forbidden_tokens(ROOT)   # ROOT = src/embodied_eval_os/core
    if bad:
        for path, token in bad:
            print(f"FORBIDDEN token {token!r} found in {path}")
        return 1
    print("OK: no forbidden benchmark/scenario leakage in core")
    return 0

注意它甚至区分两种匹配：_contains_pattern（子串即命中，用于 ebench 这种独一无二的串）和 _word_pattern（带词边界，用于 lab 这种短词，避免误伤 collaborate 之类）。CLAUDE.md 把这条命令和 pytest -q、python examples/run_smoke.py 一起列进 "Required Commands Before PR"——泄漏检查是合并前的硬门槛，不是建议。

三层可插拔结构 = 三道 anti-corruption layer

有了这条红线，整个系统就自然分成三个同心层，每一层都是一道防腐层（anti-corruption layer，这个词直接来自 DDD）：外层的脏假设不许渗进内层。

层放什么例子（真实路径）
core/scenario-neutral 契约：spec / state / metric family / tracecore/specs/task_spec.py、core/state/world_state.py
scenario_packs/某个场景自己的语义：过程原语、危险分类、指标实例scientific_lab/process_primitives/pipette.py、wetlab_safety/hazard_taxonomy.yaml
adapters/把外部 benchmark 翻译进来，只读不污染adapters/ebench/、adapters/autobio/

依赖方向是单向的，core-boundaries.md 写得斩钉截铁：core ← scenario_packs、core ← adapters、core ← backends，never 反过来。core 不许 import adapters.*，也不许 import scenario_packs.*。换句话说，EBench 可以认识 core，core 永远不认识 EBench。

那「什么概念配进 core」也有一道测试题——core-boundaries.md 的 "Core Entry Test"：一个概念只有在它对至少两个 scenario pack、或两个 backend profile、或两个 model family 有用时，才允许进 core；只服务于单一 benchmark 或单一 simulator 的东西，必须留在外面。这就是为什么 TaskSpec 能进 core，而 Lift2Action 不能。

interactivecore-leakage-sorter：这个概念该放哪一层？
把每个概念拖到 core / scenario_pack / adapter 三个桶里。放错了——比如把 Pipette 丢进 core——就会触发和 check_core_leakage.py 一样的红色报错。体会一下那条红线在 PR 时长什么样。
「EBench 做到了吗？」

没有，而且是设计使然。EBench 从骨子里就是「manipulation + Isaac」专用：它固定本体 lift2、固定仿真 Isaac Sim 4.1.0、固定一套 score schema，server 端就是把 GenManip 当黑盒跑（见 4.1 的 client–server 架构、2.1 的 Isaac 绑定）。它的 core 概念和 scenario 细节是缠在一起的——任务定义散落在 GenManip 的场景配置和 26-task 文档里，没有一道「core 不许出现 Isaac/Lift2」的红线，也没有 leakage checker。这不是 EBench 的缺点：它本就是一个聚焦的 benchmark，而 EOS 想做的是承载很多个 EBench 的 OS。两者的目标不同，于是边界纪律也完全不同。

这一节你需要带走的

(1) EOS 刻意不是 EBench fork / GenManip wrapper / Isaac-only——它把所有场景降级为可插拔的 scenario_pack 或 adapter；(2) 硬规则是 core/ 不许出现任何 benchmark/scenario/simulator 私有名，由 check_core_leakage.py 在 PR 时强制；(3) 三层 core / scenario pack / adapter 是三道 anti-corruption layer，依赖严格单向 core ← 外层。下一节我们打开最内层第一块砖：Spec 层——它如何只用纯声明式的 dataclass 说清「评什么」，而把「怎么跑」推到外面。
