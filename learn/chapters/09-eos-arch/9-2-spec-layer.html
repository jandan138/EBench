Spec 层：声明『评什么』而非『怎么跑』

上一节我们立了规矩：core 只放 scenario-neutral 契约。这一节就看最纯粹的那一类契约——Spec 层。它住在 core/specs/，全是 @dataclass(frozen=True)，没有一行执行逻辑。它的任务只有一个：用结构化、可冻结的数据，把一次评测「评什么」说清楚，而把「怎么跑」整个甩给后面的 runtime 和 backend。这就是「声明式」三个字的全部含义。

什么叫「声明式」

翻开 core/specs/，你会看到 9 个文件，没有一个里面有 for 循环、step() 或对 simulator 的调用。它们全部是 frozen dataclass——一旦构造就不可变，本质上是一份契约文档，只是用 Python 类型写的。TaskSpec 不知道怎么 reset 环境，ActionSpec 不知道怎么发关节指令，MetricSpec 不知道怎么算分。它们只是声明这些东西应该是什么样子。

从 RL 看 spec ≈ env 的 observation_space / action_space

你熟悉 gym 里的 env.observation_space 和 env.action_space：它们是描述，不是实现——一个 Box(shape=(3,)) 不会帮你 step，它只告诉你形状和范围。EOS 的 Spec 层就是把这套「空间声明」扩展到了整个评测：不仅声明动作/观测空间，还声明任务流程、成功条件、指标、泛化轴。先声明清楚，再谈执行。

TaskSpec：一次评测的「目录」

核心是 TaskSpec。它不存任何场景几何或物理，只是把一次评测需要的所有引用和条件聚到一起：

core/specs/task_spec.py
@dataclass(frozen=True)
class ProcedureStep:
    step_id: str
    instruction: str
    preconditions: list[str] = field(default_factory=list)
    success_conditions: list[str] = field(default_factory=list)
    allowed_actions: list[str] = field(default_factory=list)
    metadata: dict[str, Any] = field(default_factory=dict)


@dataclass(frozen=True)
class TaskSpec:
    task_id: str
    scenario_pack: str
    instructions: list[str]
    scene_id: str
    embodiment_id: str | None = None
    procedure: list[ProcedureStep] = field(default_factory=list)
    success_conditions: list[str] = field(default_factory=list)
    observation_spec: ObservationSpec | None = None
    action_spec: ActionSpec | None = None
    metrics: list[MetricSpec] = field(default_factory=list)
    generalization_axes: dict[str, Any] = field(default_factory=dict)
    allowed_backend_profiles: list[str] = field(default_factory=list)
    metadata: dict[str, Any] = field(default_factory=dict)

逐字段读，能读出整个系统的世界观：

  • scenario_pack：一个字符串，指明这个任务属于哪个 pack（如 "scientific_lab"）。注意——core 不 import 那个 pack，只用名字引用它。这正是 9-1 那条单向依赖的体现。
  • scene_id / embodiment_id：又是引用，不是内嵌。场景几何在 SceneSpec，本体在 EmbodimentSpec，这里只留 id。
  • procedure：一串 ProcedureStep，每步带 preconditions / success_conditions。这就是「前置满足后才评后续」的时序结构——但这里只是声明，判定逻辑在 evaluator。
  • generalization_axes：泛化轴是个 dict，open-ended，不写死「object/background/instruction」那套。哪个 pack 要测什么泛化，自己填。
  • allowed_backend_profiles：一个任务可以声明它允许在哪些 backend 上跑——这是为下一节的可组合 backend 埋的接口。

整个 TaskSpec 像一本书的目录：它告诉你这次评测涉及哪个 scene、哪个 embodiment、哪些 metric、允许哪些 backend，但每一项的内容都在别处。

ActionSpec / ObservationSpec：声明空间，不声明数值

ActionSpec 最值得看的是它的时间粒度枚举 ActionTemporalSpec——EOS 不假设动作就是「每步一个」，而是把粒度做成一等公民：

core/specs/action_spec.py
class ActionTemporalSpec(str, Enum):
    SINGLE_STEP = "single_step"
    CHUNKED = "chunked"
    SKILL_LEVEL = "skill_level"
    PROCEDURE_LEVEL = "procedure_level"


@dataclass(frozen=True)
class ActionSpec:
    action_space_id: str
    temporal: ActionTemporalSpec = ActionTemporalSpec.SINGLE_STEP
    control_mode: str = "unspecified"
    shape: tuple[int, ...] | None = None
    schema: dict[str, Any] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

还记得 5.1 讲的 action chunking 吗？在 EBench 里，chunk 是 client/server 协议里的一个 chunk_size 参数；在 EOS 里，「单步 / chunk / 技能级 / 流程级」被提升为任务声明的一部分——一个任务可以明说「我这一关评的是 procedure-level 决策」。ObservationSpec 同理，用一串 CameraStreamSpec 声明相机，再用一组布尔开关声明是否含 depth / segmentation / force-torque / proprioception。特别留意 include_privileged_state 这个字段——它默认 False，把「会不会偷看特权状态」变成一个显式、可审计的声明，呼应 9-1 里「不许悄悄给模型喂特权状态」的禁令（细节在 9.5）。

MetricSpec 与 SemanticAsset：family 通用，实例下放

MetricSpec 把 9-1 的「概念分层」做成了类型。core 只定义指标族 MetricFamily（9 个：task_outcome / procedure / state_transition / precision / hazard / prediction / planning / generalization / resource），具体指标由 scenario pack 实例化：

core/specs/metric_spec.py
class MetricFamily(str, Enum):
    TASK_OUTCOME = "task_outcome"
    PROCEDURE = "procedure"
    STATE_TRANSITION = "state_transition"
    PRECISION = "precision"
    HAZARD = "hazard"
    PREDICTION = "prediction"
    PLANNING = "planning"
    GENERALIZATION = "generalization"
    RESOURCE = "resource"


@dataclass(frozen=True)
class MetricSpec:
    name: str
    family: MetricFamily          # 通用：core 定义
    scope: str = "episode"
    target: str | None = None
    unit: str | None = None
    parameters: dict[str, Any] = field(default_factory=dict)
    weight: float = 1.0
    metadata: dict[str, Any] = field(default_factory=dict)

于是 core 永远不会出现 contamination_rate 这种字段；湿实验安全 pack 想要它，就实例化一个 MetricSpec(name="contamination_event_detection_rate", family=HAZARD, ...)。core-boundaries.md 把这个例子写成了「Wrong vs Right」对照——family 留在 core，event_type="contamination" 落到 pack。

最后一块是 SemanticAsset，它体现了一条精妙的切分：语义归 core，几何归 backend。

core/specs/asset_spec.py
@dataclass(frozen=True)
class SemanticAsset:
    """Geometry lives in backend-specific files. Core only stores semantic
    sites, joints, affordances and provenance."""
    asset_id: str
    category: str
    source_uri: str | None = None
    backend_uris: dict[str, str] = field(default_factory=dict)   # 各后端的几何文件
    sites: dict[str, Any] = field(default_factory=dict)          # 语义位点
    joints: dict[str, Any] = field(default_factory=dict)
    affordances: dict[str, Any] = field(default_factory=dict)
    provenance: dict[str, Any] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

core 不存网格、不存 USD、不存 MJCF——那些 backend 私有几何文件的路径全塞进 backend_uris 这个字典（key 是 backend 名，value 是路径）。core 只保留跨后端都成立的语义：这个物体有哪些 sites（如「瓶口」「把手」）、哪些 joints、哪些 affordances，以及它从哪来（provenance）。这样同一个 SemanticAsset 可以在 Isaac 用 USD、在 MuJoCo 用 MJCF，而 evaluator 看到的「瓶口」始终是同一个语义位点。

「EBench 做到了吗？」

没有独立的、带类型的 Spec 层。EBench 的任务定义是隐式的：它散落在 GenManip 的场景配置文件里，加上 docs 上那份 26-task 的清单（见 3.3 任务与打分）。没有一个像 TaskSpec 这样的 frozen dataclass 把 instructions / scene_id / procedure / metrics / allowed_backend_profiles 聚成一份可校验的声明，也没有把动作时间粒度（single_step / chunked / skill / procedure）或「是否含特权状态」显式声明出来——这些在 EBench 里要么写死在 server 端，要么靠协议约定。再次强调：EBench 是一个聚焦的 benchmark，把任务硬编码进场景配置完全够用；EOS 要让很多种场景共享同一套评测引擎，才需要把「评什么」抽成显式、可冻结、可比对的 Spec。

这一节你需要带走的

(1) Spec 层是一组 frozen dataclass，纯声明、零逻辑——它说「评什么」，不说「怎么跑」；(2) TaskSpec 是评测目录，用 id 引用 scene/embodiment/pack，把 procedure、metrics、泛化轴、允许的 backend 聚到一起；(3) ActionTemporalSpec 把动作粒度、include_privileged_state 把特权状态都做成显式声明；(4) MetricFamily 在 core、具体指标在 pack，SemanticAsset 让语义留 core、几何下放 backend。下一节我们追问那个反复出现的 allowed_backend_profiles：EOS 怎么把「Isaac vs MuJoCo」拆成可组合的组件？
