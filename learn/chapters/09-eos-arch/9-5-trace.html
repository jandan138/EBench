Trace 即唯一证据：EpisodeTrace 与 privileged state

上一节我们把 spec、canonical state、predicted future 都准备好了。但它们都是过程——一次评测真正留下来、能被复查的，只有一样东西：EpisodeTrace。EOS 的设计文档把这件事写得几乎是教条——"EpisodeTrace is the source of truth for every evaluation run." 这一节我们就拆开这个「唯一证据」：它带哪些字段、为什么每个字段都关乎可复现，以及它如何用一条 privileged_state 红线，把「模型偷看了答案」这种作弊从结构上挡住。

为什么是「唯一」证据

回想 8.4 我们反复强调的那条标准：一个评测结果只有在能被别人原样重跑、逐步审计时才算数。EOS 把这条标准物化成了一个 dataclass。trace-and-state.md 列出 trace 的职责是「保存足够信息，以便跨场景、跨模型、跨 backend profile 去 reproduce / audit / score / compare 一个 episode」——四个动词，对应四种用途，全压在同一个对象上。

这意味着 trace 不能只是「一串 observation 和 action」。它得同时是：复现所需的配方（seed、版本、后端组成）、审计所需的逐步明细（每一步的状态、动作、指标）、以及一张写死的免责声明（这串证据到底证明了什么、没证明什么）。我们一层层看。

TraceStep：一步里发生的全部事实

最小单元是 TraceStep——评测循环里走的每一步，都凝结成一个它。字段直接对应 9-4 那三层状态：

core/state/trace.py
@dataclass(init=False)
class TraceStep:
    step_index: int
    time_s: float
    observation: dict[str, Any]
    action: dict[str, Any] | list[float] | None
    canonical_physical_state: CanonicalPhysicalState | None = None
    task_world_state: TaskWorldState | None = None
    prediction: PredictedFuture | None = None
    metric_results: dict[str, Any] = field(default_factory=dict)
    privileged_state_visible_to_model: bool = False
    metadata: dict[str, Any] = field(default_factory=dict)

注意这一步同时记下了三种东西：模型看到了什么（observation）、模型做了什么（action）、以及世界真实是什么（canonical_physical_state / task_world_state）。这三者分离是关键——因为「模型看到的」和「世界真实的」往往不一样，而这个差距正是评测想测量的核心。prediction 字段则留给世界模型：如果被评的是个 WAM，它在这一步想象的未来 PredictedFuture 也会被原样存下，留待和真实未来对账（这正是 9-4 PredictedFuture 的落地处）。

这里有一个值得留意的工程细节：TraceStep 用了 @dataclass(init=False) 加手写 __init__，就是为了向后兼容。旧代码可能用 world_state= / metrics= 这两个老参数名，新代码用 task_world_state= / metric_results=；构造函数把老名映射到新名，还提供了 world_state / metrics 两个 property 别名。更妙的是，只要你给了 task_world_state，它会自动把 canonical_physical_state 设成 task_world_state.physical——保证「物理态」永远和「任务世界态」对得上，不会出现两者打架的脏 trace。

从 RL 看 不只是 transition，是带审计字段的 transition

你在 RL 里熟悉的 transition 是 $(s_t, a_t, r_t, s_{t+1})$——够训练用。但评测要的更多：它要分清「agent 观测 $o_t$」与「特权真值 $s_t$」（部分可观下两者不同），还要记下「这一步给了哪些指标」。TraceStep 就是把 RL transition 扩成一个可审计版本：observation / action / 真值 state / metric / 以及一个布尔「模型这步是否看到了特权态」。它不是为了学习，是为了事后没人能赖账。

EpisodeTrace：一整集 = 一个自包含证据单元

把许多 TraceStep 串起来、再裹上整集级别的元数据，就是 EpisodeTrace。它的头部字段才是「可复现」真正的载体：

core/state/trace.py
@dataclass
class EpisodeTrace:
    episode_id: str
    task_id: str
    scenario_pack: str
    backend_profile_id: str
    physics_profile_id: str | None = None
    renderer_profile_id: str | None = None
    seed: int | None = None
    model_profile_id: str | None = None
    versions: dict[str, Any] = field(default_factory=dict)        # repo_commit / task_spec / scene_spec
    backend_components: dict[str, Any] = field(default_factory=dict)
    privileged_state_policy: str = "model_input_excludes_privileged_state"
    claim_boundary: str = "Trace evidence only; no simulator fidelity or real-world safety claim."
    steps: list[TraceStep] = field(default_factory=list)
    episode_metrics: dict[str, Any] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

逐条看这个头部，你会发现它就是一份「重跑这次评测需要的全部坐标」：scenario_pack + task_id（评什么）、backend_profile_id + physics_profile_id + renderer_profile_id（在哪个后端、哪套物理/渲染上跑——对应 9-3 的可组合后端）、seed（随机性的钥匙）、model_profile_id（哪个模型）。而 versions 字典里塞的是 repo_commit / task_spec / scene_spec 三个版本号——没有它们，三个月后代码改了，这串 trace 就成了无法对照的孤证。trace-and-state.md 把话说死：「没有这些元数据，trace 不被视为有效证据。」

而 backend_components 是对 9-3 那个六件套（platform / physics / renderer / sensors / control / process_runtime）的快照——它记下「这次评测的后端是由哪些组件拼出来的」，这样换了组件、结果变了，你能立刻定位是哪一件变了。

interactivetrace-anatomy：把一份 EpisodeTrace 拆开看
点开头部元数据与逐步明细。注意「复现配方」（versions / seed / backend_components）、「逐步审计」（每个 TraceStep）、「免责声明」（claim_boundary / privileged_state_policy）三组字段，如何分别服务 reproduce / audit / claim 三种用途。

两条写进字段的红线：privileged state 与 claim boundary

这是整个 trace 设计里最体现「评测纪律」的两个字段，它们都是默认值就是安全值。

第一条是 privileged_state_policy，默认 "model_input_excludes_privileged_state"。它对应一条具体规则（trace-and-state.md §8）：TaskWorldState / CanonicalPhysicalState 里可能含特权信息（比如真实液体体积、隐藏过程变量）——这些 evaluator 和 recorder 可以读，但默认绝不喂给模型。如果哪个 PR 想把特权字段暴露给 model adapter，必须显式说明理由并把该 benchmark 标成「state-conditioned」模式。每个 TraceStep 上那个 privileged_state_visible_to_model 布尔，就是这条策略落到逐步粒度的见证：它逐步记录「这一步模型有没有被允许看真值」，事后任何人都能核查模型是不是偷看了答案。

第二条是 claim_boundary，默认那句话掷地有声："Trace evidence only; no simulator fidelity or real-world safety claim." 它把这份证据的「证明力边界」直接焊死在数据里——这串 trace 只证明「在这个仿真后端里，这次跑成了这样」，它不证明仿真有多真、不证明能迁移到真机、不证明真实世界安全。EOS 在很多更底层的运行时类型里也复用了同一个 anti-overclaim 思路（我们 9-8 会看到 RuntimeMeasurementSnapshot 甚至用正则去拦截过度宣称的措辞）。

从 GAMES103 看 仿真跑通 ≠ 物理为真

你写过求解器就懂：一个刚体在你的积分器里稳稳落进槽里，不代表真实世界的摩擦、形变、接触会这么走。仿真只是某套数值假设下的结果。claim_boundary 把这份谦逊变成了一个不可省略的字段——任何拿 trace 去说「我的策略真机安全」的人，都会被这句默认免责声明当场拦下。这正是把 GAMES103 的「数值结果 ≠ 物理真相」上升成了评测系统的硬约束。

序列化：to_dict / from_dict 与 TraceStore

证据要能存、能传、能被别的工具读，所以 trace 全程可 JSON 化。EpisodeTrace.to_dict() 用一个递归的 _to_plain 把所有嵌套 dataclass（canonical state、task world state、prediction）压平成纯字典；from_dict() 再反向重建，并且体贴地接受旧别名——读 step 时 task_world_state 取不到就退回 world_state，metric_results 取不到就退回 metrics。这保证老 smoke trace 仍能被新代码读出来，而新产物一律写规范名。

落盘则交给 TraceStore，薄到只有两个方法：

core/recorders/trace_store.py
class TraceStore:
    def __init__(self, root: str | Path) -> None:
        self.root = Path(root)
        self.root.mkdir(parents=True, exist_ok=True)

    def save_json(self, trace: EpisodeTrace) -> Path:
        path = self.root / f"{trace.episode_id}.json"
        path.write_text(
            json.dumps(trace.to_dict(), ensure_ascii=False, indent=2, sort_keys=True),
            encoding="utf-8",
        )
        return path

    def load_json(self, path: str | Path) -> EpisodeTrace:
        payload = json.loads(Path(path).read_text(encoding="utf-8"))
        return EpisodeTrace.from_dict(payload)

注意 sort_keys=True 这个小动作：键有序的 JSON 让两份 trace 可以直接 diff，也让内容哈希稳定——又是「可审计」的细节。设计文档把这一层定位为 MVP-0.5 产物：trace 是 JSON 文件，必须能无损往返 save_json() / load_json()，不丢 provenance、backend component、privileged-state、claim-boundary、state、action、prediction、metric 任何一项。这块是已实现并被 smoke 测试覆盖的（下一节、以及 9-8 的 run_smoke.py 都会真的写出一份）。

「EBench 做到了吗？」

部分做到，但形态完全不同。EBench 确实会按 task / run_id 把评测结果落盘到 saved/eval_results/<task>/<run_id>/（见 4.1），也能 --frame_save_interval 存过程帧——它有结果记录。但它没有一个像 EpisodeTrace 这样、把 versions{repo_commit/task_spec/scene_spec}、backend_components、privileged_state_policy、claim_boundary 全部做成一等字段的类型化证据单元，更没有逐步的 privileged_state_visible_to_model 这种「模型有没有偷看答案」的结构化审计位。EBench 的复现靠的是固定本体 + 固定 Isaac + 固定 split 这套外部约定；EOS 因为要跨后端、跨模型族，必须把复现配方和免责声明刻进 trace 本身，否则换了后端就无从对照。再说一次：单后端 benchmark 用外部约定足矣；通用 OS 才需要让每一份证据自带全部坐标。

这一节你需要带走的

(1) EpisodeTrace 是每次评测的唯一证据，同时承担 reproduce / audit / score / compare 四种用途；(2) TraceStep 逐步分离「模型观测 / 模型动作 / 世界真值 / 模型预测 / 指标」，是带审计字段的 RL transition；(3) 头部的 seed / versions / backend_components 是复现配方，缺一即「无效证据」；(4) privileged_state_policy 默认排除特权态、逐步用 privileged_state_visible_to_model 见证，claim_boundary 默认把证明力焊死在「仅 trace 证据」；(5) to_dict/from_dict + TraceStore 让证据无损 JSON 往返、键有序可 diff——这一层已实现并被 smoke 覆盖。下一节进入 Primitive 层：scenario pack 究竟怎么把归一化的物理态，翻译成 9-4 里那三个 schema-free 的任务语义通道。
