BackendProfile：把『Isaac vs MuJoCo』拆成可组合组件

上一节 TaskSpec 里有个字段叫 allowed_backend_profiles。这一节就揭开它指向的东西：BackendProfile。它要解决一个看似简单、实则要命的问题——当你说「我在 Isaac 上跑的」「他在 MuJoCo 上跑的」，你到底说了什么？EOS 的回答是：「Isaac」「MuJoCo」是太粗的标签，一个 backend 必须拆成可组合的组件，否则跨后端比较根本无效。

「Isaac vs MuJoCo」是一句没信息的话

设想两篇论文都报告「我们的模型在 Isaac 上 0.7，在 MuJoCo 上 0.5」。这 0.2 的差距是物理引擎不同造成的？还是渲染器不同（RTX 光追 vs 简易 rasterizer，导致视觉输入分布漂移）？还是接触模型、传感器噪声、控制接口的差异？你无从判断，因为「Isaac」这个名字同时打包了平台、物理、渲染、传感、控制、过程运行时六样东西。把六个变量焊死成一个名字再去比，得到的差值无法归因。

从 GAMES103 看 integrator ≠ renderer ≠ contact model

你在 GAMES103 里早就把这些当成正交的轴：时间积分器（显式/隐式 Euler、symplectic）是一回事，碰撞/接触求解（penalty、impulse、LCP）是另一回事，而把场景画成像素的 renderer 跟物理八竿子打不着。换 renderer 不改变刚体怎么运动；换 integrator 不改变画面长相。EOS 只是把你脑子里这套「可分离的轴」写进了数据结构——一个 backend 不是一个名字，是这些轴的一次组合。

ComponentProfile + CapabilityTier

BackendProfile 把一个后端拆成六个 ComponentProfile：platform、physics、renderer、sensors[]、control、process_runtime。每个组件自带一组 capabilities，而每项能力的保真度用一个枚举 CapabilityTier 标注——这是整套设计的灵魂：

core/specs/backend_profile.py
class CapabilityTier(str, Enum):
    """Capability fidelity declaration for a backend component."""
    NATIVE = "native"                  # 原生、真实实现
    REDUCED = "reduced"                # 降级近似
    SEMANTIC_PROXY = "semantic_proxy"  # 语义代理（不是真物理）
    EMULATED = "emulated"              # 仿造
    UNSUPPORTED = "unsupported"        # 不支持


@dataclass(frozen=True)
class ComponentProfile:
    name: str
    component_type: str          # "physics" / "renderer" / "control" ...
    version: str | None = None
    capabilities: dict[str, CapabilityTier | str] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)


@dataclass(frozen=True)
class BackendProfile:
    profile_id: str
    platform: ComponentProfile
    physics: ComponentProfile | None = None
    renderer: ComponentProfile | None = None
    sensors: list[ComponentProfile] = field(default_factory=list)
    control: ComponentProfile | None = None
    process_runtime: ComponentProfile | None = None
    metadata: dict[str, Any] = field(default_factory=dict)

    def component_summary(self) -> dict[str, str | None]:
        return {
            "platform": self.platform.name,
            "physics": self.physics.name if self.physics else None,
            "renderer": self.renderer.name if self.renderer else None,
            "control": self.control.name if self.control else None,
            "process_runtime":
                self.process_runtime.name if self.process_runtime else None,
        }

CapabilityTier 是把「Claim boundary」编码进类型的关键。一个后端可以诚实地说「我的刚体是 native，但液体只是 semantic_proxy」——这正是 CLAUDE.md 那条「semantic proxy is not real CFD」禁令的落地：不许把语义代理冒充真物理。component_summary() 是这个 dataclass 上唯一的方法，而且它只做聚合展示，不做任何执行——再次印证 spec 层零逻辑。

一份真实的 profile：物理用 MuJoCo，渲染用 Blender

profile 以 YAML 存在 configs/backend_profiles/。下面这份 mujoco_blender.yaml 把「可组合」体现得最直白——它故意把物理和渲染拆给两个不同的引擎：MuJoCo 算物理，Blender Cycles 离线渲高质量图。

configs/backend_profiles/mujoco_blender.yaml
profile_id: mujoco_physics_blender_render
platform:
  name: split_backend           # 平台本身就声明：这是一个拼装后端
  component_type: platform
physics:
  name: mujoco
  component_type: physics
  capabilities:
    rigid_body: native
    articulation: native
    contact: native
renderer:
  name: blender_cycles
  component_type: renderer
  capabilities:
    rgb: offline_high_quality
    depth: emulated
    segmentation: emulated
metadata:
  role: split_physics_rendering_profile

注意 platform.name 直接写 split_backend——平台层坦白「我不是某个一体化 simulator，我是把物理和渲染拼起来的」。对比一下 isaac_physx_rtx.yaml：它的 physics 是 physx（liquid 标 semantic_proxy、deformable 标 reduced），renderer 是 isaac_rtx（rgb/depth/segmentation 全 native）。两份 profile 一摆，差异不再是「Isaac vs 拼装」这种模糊话，而是逐组件、逐能力、带 tier的精确对照。

interactivebackend-composer：自己拼一个 BackendProfile
分别给 physics / renderer / sensors / control 选不同引擎，看 component_summary() 怎么变。试试「固定 physics、只换 renderer」——这正是下面说的唯一有效的跨后端对照方式。

回报：跨后端比较只能逐组件做

把 backend 拆开之后，一条强约束自然浮现，backend-composability.md 写成了 Bad / Better 对照：

docs/design/backend-composability.md · §5
Bad:
  Compare Isaac vs MuJoCo.

Better:
  Compare PhysX vs MuJoCo physics while keeping
  task spec and metric spec fixed.
Or:
  Compare Isaac RTX vs Blender Cycles rendering while
  keeping physics rollout fixed.

这就是为什么文档反复强调「same physical rollout + different renderer」「same task + different physics engine」这些组合——只有固定其余组件、只动一个组件，差值才可归因。它让你能把视觉泛化和动力学泛化分开测，把「渲染器导致的失败」和「物理导致的失败」分开诊断。文档还特意泼了盆冷水：这种逐轴的 readiness matrix 不是 benchmark score——它回答的是「哪个组件有终态证据、哪个被什么 blocker 卡住、相对 baseline 改了哪个组件」，不回答「模型成功了吗、两个引擎物理等价吗、像素等价吗」。把组件级证据误读成成绩单，是这层最大的诱惑，也是被明令禁止的。

「EBench 做到了吗？」

没有可组合的 backend 抽象——它固定一个后端。EBench 锁死 Isaac Sim 4.1.0（PhysX 物理 + RTX 渲染 + GenManip runtime，见 2.1），server 端把整个 Isaac 当黑盒。这对一个聚焦 benchmark 是合理的：固定后端才能保证 leaderboard 公平可比。但代价是你无法在 EBench 内部问「换个 renderer 模型还行不行」「物理引擎对这个失败贡献多少」——这些变量被焊死了。在 EOS 里，这套老栈被降级成一份带标签的兼容 profile ebench_genmanip_legacy.yaml，metadata.role: ebench_compatibility_only、must_not_drive_core_design: true，并明写 anti_corruption_layer: adapters/ebench——它能被复现，但绝不许反过来定义 core 的设计。

这一节你需要带走的

(1)「Isaac vs MuJoCo」是粗到无法归因的标签；EOS 用 BackendProfile 把后端拆成 platform/physics/renderer/sensors/control/process_runtime 六个 ComponentProfile；(2) 每项能力用 CapabilityTier（native→reduced→semantic_proxy→emulated→unsupported）诚实标注保真度，禁止用 proxy 冒充真物理；(3) mujoco_blender.yaml 这种「物理 MuJoCo + 渲染 Blender」的拼装 profile 说明组件真的可换；(4) 跨后端比较只能逐组件做——固定其余、只动一个，差值才可归因，而这是 readiness 证据、不是 score。下一节进入 State 层：这些不同后端吐出的物理量，怎么被归一成同一套世界态，让同一个 evaluator 通吃 MuJoCo 的 qpos 和 Isaac 的 joint state。
