泛化怎么评：train/test isolation 与 leakage

8.4 的过度宣称清单里有一条：「在测试分布上高分 ≠ 泛化」。这一节把它正面回答清楚——泛化到底怎么测才算数。核心只有一句话：要测泛化，被测的东西就必须是模型真正没见过的。听起来理所当然，但「没见过」这件事，比绝大多数人以为的更难保证——一道细微的裂缝，就能让一个其实只会背书的模型，在你的榜上拿到泛化高分。

泛化是「对没见过的东西」的能力

先把概念钉死。泛化（generalization）不是「在训练分布上做得好」——那叫拟合。泛化是「面对训练时没出现过的情况，依然做得好」。所以测泛化的逻辑前提，是存在一批模型在训练中从未接触的样本，我们专门拿它们来考。这批样本一旦在训练里露过脸，它测出来的就只是记忆，不是泛化。整个泛化评测的可信度，都压在「考题确实是新的」这一个支点上。

从 RL 看 对评测分布的过拟合

你在 RL 里反复警惕的 overfitting，搬到这里就是「过拟合到评测分布」。一个 agent 可以把某个固定关卡的每一帧背得滚瓜烂熟、刷出惊人回报，换一张新地图却原地撞墙——它学到的是这张图的查找表，不是可迁移的策略。评测泛化要做的，正是不让模型靠查找表蒙混过关：把考场换成它没背过的地图。你对「在训练环境上 reward 高不代表学到了策略」的警惕，原封不动就是这一节的全部动机。

held-out split 与 train/test isolation

实现「考题是新的」的标准做法，是 held-out split：把数据切成互不重叠的几份——可调的训练/验证集，和一个被锁起来、只用于最终评测的留出测试集。这就是 train/test isolation（训练/测试隔离）。

EBench 的做法（细节见 3.2）是三个 split：val_train 与 val_unseen 是开放的、可以随便调参看反馈的；而 test 被留出来驱动 leaderboard。隔离的关键纪律是：test 上的信息绝不能回流到训练或调参里。一旦你开始「在 test 上看一眼、回去改改、再看一眼」，你就在用 test 的梯度悄悄训练，test 也就不再是 held-out 的了——它退化成了又一个训练集，只是退化得很隐蔽。

隔离不是「分开存两个文件夹」

真正的隔离是信息流上的隔离，不只是文件上的分开。常见的隐蔽泄漏：反复用 test 结果做模型选择（早停、调超参、挑 checkpoint）——每做一次，test 的信息就漏进了你的决策一次。所以 leaderboard 才要把 test 锁在服务器端、限制提交频次：它防的不是你看不到答案，而是防你用答案来反向优化。

沿不同的轴去考泛化：OOD axes

「没见过」其实有很多种没见过，而模型可能在某些轴上泛化、在另一些轴上彻底失灵。所以好的泛化评测会把它分轴来考，而不是笼统问一句「泛化吗」。EBench 设想的几条 OOD（out-of-distribution，分布外）轴（详见 3.4）很有代表性：

  • object：换没见过的物体（同任务，新道具）。
  • background：换没见过的背景 / 场景布置。
  • instruction：换没见过的指令措辞（同任务，新说法）。
  • mixed：以上几种同时变——最接近真实开放世界的考法。

分轴的价值和 8.5 是一脉相承的：一个笼统的「泛化分」会把这几种能力压扁；分轴诊断才能告诉你「它认得新物体但听不懂新说法」这种可操作的结论。注意，按 8.1 提过的口径区分：这套多轴 OOD 是 README/paper 的愿景语言，当前线上 docs 的可验证机制是三 split 的隔离；本书把愿景和地面真相分开标注。

data leakage：泄漏怎样悄悄毁掉一切

train/test isolation 听起来简单，但 data leakage（数据泄漏）是出了名地难根除——它指任何让「测试信息」偷偷进入训练的途径，而它往往藏在你没注意的地方。几种典型的失效模式：

  • 近重复泄漏：test 里的物体/场景，是 train 里同一个的换皮版本（同一 3D 资产、只改了贴图或角度）。看着是新的，实质模型早就见过。
  • 共享底层资产：train 和 test「任务名不同」，但底层用的是同一批 3D 物体库——模型在物体层面其实没遇到新东西。
  • 调参泄漏：上面说过的，反复用 test 反馈做模型选择，test 信息经由你的决策回流。
  • 统计量泄漏：用全量数据（含 test）算归一化统计、词表、数据增强参数，test 的分布信息混进了训练管线。
泄漏的危险在于它抬高的是你最想信的那个数

data leakage 不会让结果看起来出错——恰恰相反，它让泛化分变得漂亮。一个有泄漏的榜单，会奖励那些「最善于（无意中）利用泄漏」的模型，而不是真正泛化最强的模型，于是整个领域被往错误方向牵引。更糟的是它常常无声无息：除非你主动去审计 train/test 之间的资产重叠、措辞重叠、统计量来源，否则你不会发现自己报的「泛化能力」其实是「记忆能力」。可信的泛化评测，必须把「我们如何保证没有泄漏」当成结果的一部分写出来——这又回到了 8.4 的 claim boundary：说清你的隔离覆盖到了哪一层。

Part 8 收尾，与 Part 9 的桥

这一节，也是这一部分的收束：(1) 泛化 = 对没见过的东西的能力，整套评测压在「考题确实是新的」这一个支点上；(2) 用 held-out split 实现 train/test isolation，隔离的是信息流而不只是文件夹；(3) 分轴考 OOD（object / background / instruction / mixed），别压成一个笼统的泛化分；(4) 警惕 data leakage 的各种隐蔽形态——它专门抬高你最想信的那个数。

到这里，Part 8 的「评测系统通论」就讲完了：从「分数是什么主张」（construct validity），到五层解剖，到契约即架构，到可复现与 claim boundary，到聚合陷阱，再到泛化与泄漏。这些都是第一性原理——不依赖任何具体框架。
