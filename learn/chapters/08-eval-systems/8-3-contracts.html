契约即架构：abstraction boundary 决定可扩展性

上一节我们把系统切成五层。但真正决定一个系统命运的，不是层本身，而是层与层之间那条缝——它们靠什么契约（contract）对话。这一节的核心主张听起来有点极端，但它是对的：架构不在盒子里，架构在盒子之间的接口里。接口划好了，你能不断往里加新东西；接口划砸了，加一样东西都要重写半个系统。

为什么接口比模块更重要

设想你要给评测系统加一件新东西——一个新模型、一个新仿真后端、一个新任务。一个健康的系统里，这应该是个局部动作：写一个新实现，塞进既有的插槽，别处纹丝不动。能不能做到这一点，几乎完全取决于插槽（也就是契约）当初是怎么定义的。

契约就是一组约定：「我向你保证我会提供 A、B、C 这几个方法，输入输出长这样；至于我内部怎么实现，与你无关。」当 Metric 层只通过「读 State」这个契约和上游对话，它就不知道、也不需要知道底下是 Isaac Sim 还是 MuJoCo。于是换后端这件事，对 Metric 层是透明的。契约把「会变的东西」（具体实现）关在边界内侧，把「稳定的东西」（接口形状）暴露在外侧——这正是 abstraction boundary 这个词的字面意思。

为什么说契约定义了「可扩展性的形状」

你的系统能沿哪个轴低成本扩展，等于问：你在哪个边界上定义了干净的契约。在「模型」边界上有契约 → 加新模型便宜；在「后端」边界上有契约 → 换新仿真器便宜；在「任务」边界上有契约 → 加新任务便宜。没有契约的那个轴，扩展成本就是「重写」。所以做架构，本质就是选择在哪几条缝上预先付出抽象的成本。

耦合 vs 可组合：两种相反的命运

契约的反面是耦合（coupling）。当 Metric 层为了打分，直接 reach 进仿真器的私有字段——比如读 sim.bodies[3].contact_force——它就和那个具体后端的内部结构焊死了。表面上少写了一层抽象、省事；实际上你埋了一颗雷：换后端、甚至后端升级改了字段名，打分逻辑就崩。耦合的代价不在写的时候，在改的时候，而且它会沿着每一条「偷偷伸进去的手」四处传染。

反过来，可组合性（composability）的意思是：每个组件只认契约、不认实现，于是它们能像乐高一样自由拼装。{后端 A, 后端 B} × {模型 X, 模型 Y, 模型 Z} 这 2×3 种组合，你不需要写 6 份代码——你写 2 个后端实现、3 个模型实现，让契约把它们正交地拼起来。组合数随实现数相乘，工作量却只随实现数相加，这就是可组合系统的复利。一个评测系统要服务很多模型 × 很多任务 × 可能多个后端，可组合性几乎是它能不能长大的唯一决定因素。

要守住这条线，有个很实用的技巧叫 anti-corruption layer（防腐层）：当你必须接入一个混乱的外部系统（比如某个仿真器古怪的 API），不要让它的混乱渗进你的核心；而是写一层薄薄的适配，把外部世界翻译成你内部干净的契约。外部怎么变，脏只脏在防腐层这一处，核心永远说自己那套干净的语言。这个想法 Part 9 的 9.1 会展开成 EOS 的核心设计立场，这里你先记住这个名字和直觉。

一个范本：EBench 的 EvalClient 契约

抽象讲完，看一个真实的、划得极漂亮的窄契约——EBench 的 EvalClient（我们在 4.2 拆过它的内部）。它要解决的难题很现实：仿真器（Isaac Sim）有一套沉重、挑剔的依赖；各家 VLA 模型又有各自一套同样挑剔、还互相冲突的依赖。把它们装进同一个 Python 环境，是一场依赖地狱。

EBench 的答案是在二者之间划一条窄到不能再窄的契约，只有三个动作：

genmanip_client/eval_client.py（契约用法）
from genmanip_client.eval_client import EvalClient

client = EvalClient(base_url="http://127.0.0.1:8087",
                    worker_ids=["0"], run_id="my_eval")

obs = client.reset()            # ① reset：开一个新 episode，回首帧观测
eval_finished = False
while not eval_finished:
    action = model.get_action(obs)
    obs, eval_finished = client.step(action)   # ② step：喂 action，回下一帧观测
client.close()                  # ③ close：收尾

就这三件事：reset / step / close。契约的两侧因此被彻底解耦：仿真这一侧（server）跑在它自己的重环境里，把 Isaac Sim 当黑盒；模型这一侧只需要在自己的环境里 pip install 一个极薄的 genmanip-client 包，就能通过这三个方法对话。模型作者完全不需要知道 Isaac Sim 的存在，更不必和它的依赖共处一室——依赖冲突在契约这条缝上被一刀切断。

这就是「窄契约」的威力：接口越窄，两侧能各自变化的自由度越大。任何能实现 get_action(obs) 的模型都能接入，任何能在 server 端跑起来的仿真都能被换掉——而这条 reset/step/close 的缝一动不动。它是「契约即架构」最干净的一个实证。

从 RL 看 这就是 gym 的 env API

你对这个契约其实早就熟得不能再熟了。Gymnasium（旧称 gym）整个生态能繁荣，靠的就是一条窄契约：env.reset() → obs, reward, done, info = env.step(action)。正因为每个 env 都遵守这条缝，你写的任何 agent 都能跑在任何 env 上，CartPole 和 Atari 共用同一份训练循环。EvalClient 的 reset/step/close 就是同一个设计哲学搬到评测里：把「环境」和「策略」用一条标准接口隔开，两边各自爆炸式生长而互不打扰。你在 RL 里享受过的那种「随便换 env、随便换 agent」的自由，正是一条好契约给的礼物。

这一节你需要带走的

(1) 架构的本质在层与层之间的契约，不在层本身——可扩展性的形状，就是你在哪几条缝上划了干净契约的形状；(2) 耦合（伸手抓别人的实现）让改动沿缝传染，可组合（只认契约）让工作量随实现数相加、组合数相乘；(3) anti-corruption layer 把外部的混乱挡在核心之外；(4) EBench 的 EvalClient（reset/step/close）是一条窄契约的范本，它把模型依赖和仿真依赖一刀切开，正如 gym 的 env API 隔开了 agent 与 environment。下一节，我们追问下一个问题：契约划好了，跑出来的那个数，凭什么可信、凭什么可复现？
