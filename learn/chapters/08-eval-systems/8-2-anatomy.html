评测系统的解剖学：Spec / State / Runtime / Metric / Report

上一节我们说，一个分数是关于能力的主张。这一节回答：这个主张是怎么被造出来的？答案是——任何评测系统，不管它写得多随意，都在做同样的五件事。把这五层认出来，你就有了一把通用的解剖刀，可以拆开你见到的任何 benchmark，也包括你自己将要写的那个。

任何评测系统都有的五层骨架

从「声明评什么」到「产出给人看的报告」，中间一定要走过五个环节。它们可以是五个干净的模块，也可以挤在一个 1000 行的脚本里纠缠不清——但逻辑上它们一定都在：

  • Spec（规格层）—— 声明「评什么」：有哪些 task、哪些 scene、action / observation 接口长什么样、跑在什么 embodiment 上、要算哪些 metric。它应当是纯数据、不含逻辑的：一份冻结的声明。
  • State（世界状态层）—— 把仿真器吐出来的原始物理量，归一化成一个统一的世界状态表示，再叠上任务语义（流程到了第几步、有没有触发危险、过程量是多少）。评分逻辑读的是它，而不是后端的原始字段。
  • Runtime（运行时层）—— 真正驱动一个 episode：reset → 取 obs → 模型出 action → step → 记录。它的产物是一条可复现的证据。
  • Metric（指标层）—— 把一条证据翻译成「分」。不只是 success / fail，而是若干家族的诊断量。
  • Report（报告层）—— 把指标聚合成给人看的诊断：多轴画像、热力图、迁移曲线，并写清这份结果不能证明什么（claim boundary）。

注意这五层的数据流向是单向的：Spec 定义了世界 → Runtime 在这个世界里跑出 State 的时间序列 → Metric 读 State 打分 → Report 聚合。每一层只依赖它上游产出的抽象，而不该伸手去碰更上游的实现细节。这条单向链，是后面所有「可复现」「可扩展」讨论的地基。

成熟系统让每一层显式，临时脚本让它们纠缠

这套骨架的关键不在于「有没有这五层」——每个系统都有——而在于这五层是显式还是隐式。

一个临时拼起来的评测脚本，往往把五层揉在一起：任务定义硬编码在 if-else 里（Spec 藏进了逻辑），打分时直接 reach 进仿真器的私有字段（Metric 越过 State 抓 Runtime 的实现），最后 print 一个平均数（Report 退化成一行 stdout）。它能跑，但你没法单独替换任何一块：想换个仿真后端，得重写打分；想加个新 metric，得读懂整个 episode 循环；想复现半年前的结果，发现 Spec 根本没被记录下来。层与层之间没有边界，改动的代价就会沿着纠缠四处传染。

一个成熟的系统反过来：让每一层成为显式、独立、可命名的东西。Spec 是一组冻结的 dataclass，State 有自己的归一化类型，Runtime 产出一个结构化的 trace，Metric 是一族可注册的 evaluator，Report 是一个多轴容器。代价是前期要多写一些「看起来没用」的抽象；回报是每一层都能被单独替换、单独测试、单独复现。这一部分接下来几节（契约、可复现、聚合、泛化）讲的，本质上都是「怎么把某一层的边界划干净」。

从 GAMES103 看 这就是仿真循环的分层版

你在 GAMES103 里写物理仿真时，其实已经在做同一件事。Spec 对应「场景与材料参数的声明」（质量、刚度、约束）；State 对应「每一步的位置/速度状态向量」；Runtime 对应「时间积分循环」本身；把它当 benchmark 看，Metric 就是「能量是否守恒、约束误差多大」这类诊断；Report 是你画出来给助教看的收敛曲线。当年你若把这几件事搅在一个 update() 函数里，换一个积分器就得重写一切；分层之后，换 explicit / implicit Euler 只动 Runtime 那一层。评测系统的分层动机，和你写好一个仿真器的动机是同一个。

一条光谱：从轻量 benchmark 到显式框架

「显式 vs 隐式」不是非黑即白，而是一条光谱。你可以在一个聚焦 benchmark 里把很多层保持轻量，也可以在一个长期演进的评测框架里把每一层都做成独立对象。关键不是哪边天然更高级，而是目标和成本是否匹配。

EBench（Part 4 我们拆过它的引擎）坐在偏「隐式」的一端——这不是缺点，而是工程取舍。它的 Spec 散落在 GenManip 场景配置和 docs 的 26 task 列表里，没有独立的 Spec 类型；它的 State 直接就是 Isaac Sim 吐出的 obs 字典，没有「归一化物理态」这层抽象；它的强项在 Runtime——那套 client-server 把仿真当黑盒、让模型装进自己的 env，是一个划得极漂亮的边界（下一节专门讲）。它优先把能跑通、能并行、能上榜这条主干打通，其余层保持轻量。

光谱另一端的系统会把五层逐一提升为一等公民：独立的 Spec 数据类型、归一化 State、结构化 Trace、多族 Metric、显式 claim boundary。它用更多的前期抽象，换「任何一层都能换、任何结果都能复现」的可扩展性。两端没有对错，只是服务于不同优先级：一个面向发布和复现一个具体 benchmark，一个面向长期承载多任务、多模型、多后端的演进。

这一节你需要带走的

(1) 任何评测系统都由 Spec → State → Runtime → Metric → Report 五层组成，数据单向流动；(2) 区别不在「有没有这五层」，而在它们是显式还是隐式——隐式会让改动沿纠缠传染，显式让每层可换、可测、可复现；(3) EBench 把重心放在能跑通、能并行、能上榜的主干上，其他系统可以按需求付出更多抽象成本。下一节我们放大其中最关键的东西：层与层之间那条契约（contract）——它才是真正决定一个系统能不能长大的地方。
