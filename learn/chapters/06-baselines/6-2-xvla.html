X-VLA：soft prompt + domain_id

π0 用一个 3B 的 PaliGemma 当骨干，靠庞大的预训练知识硬扛。X-VLA 走了一条相反的路：把骨干做得极小极纯，把"跨本体"的差异塞进几个可学习的向量里。理解它，等于理解一个问题——当你想让一个策略同时服务很多种机器人时，是该把差异写进权重，还是写进 prompt？

模型层：soft-prompted Transformer 与 domain_id

X-VLA（arXiv 2510.10274）是一个 flow-matching VLA：和 π0 一样用 flow matching 出 action chunk（5.2）。但它的骨干非常素——一个纯 soft-prompted 的标准 Transformer encoder，24 层、hidden 1024，没有 PaliGemma 那种庞大 VLM。视觉语言编码交给 Florence-Large，骨干只负责把多模态 token 融合成动作。

它真正的招式是 soft prompt。"硬 prompt"是你写给模型的文字；"软 prompt"是一组可学习的 embedding 向量，直接拼进 token 序列、随训练优化。X-VLA 给每个数据源（embodiment）一套自己的 soft prompt：

  • 这套 embodiment-specific 的 embedding 负责吸收"这个机器人/数据源特有"的差异；
  • 剩下的共享骨干则学一套 embodiment-agnostic 的策略——跨本体通用的"怎么操作"。

domain_id 就是"选哪套 soft prompt"的索引。训练时不同数据源用不同 id；推理时你告诉它"我现在是哪个 domain"，它就取对应的那套 embedding。EBench 的 run.py 里这个值恒为 0——用默认那套 prompt。

从 RL 看 这是 multi-task 的 task embedding，搬到了 embodiment 上

在 multi-task RL 里，你常给策略喂一个 task id / task embedding，让一个网络条件化地解多个任务。X-VLA 把同样的思路用在本体上：domain_id 就是 embodiment 的条件变量，soft prompt 就是它对应的可学习 embedding。骨干学共性，prompt 记差异——和 multi-task RL 里"shared backbone + per-task head"的分工是同一个直觉，只是把 head 换成了拼在输入端的几个向量。

两阶段训练与 PEFT：~9M 参数接一个新机器人

这个设计的回报在迁移阶段。X-VLA 分两阶段：

  • 预训练：在 0.9B 帧 / 290K episodes / 7 个平台的混合数据上训共享骨干 + 各 domain 的 soft prompt。
  • 迁移到新机器人：不重训骨干。加一套新的 soft prompt，冻结骨干，只 PEFT（parameter-efficient fine-tuning）那一小撮参数——论文报告仅 ~9M（约 1%）可训练参数就能逼近 SOTA。

对照 π0.5 的 knowledge insulation（5.4）：两者都在回答"怎么不破坏预训练知识地学新东西"，但答案相反——π0.5 用 stop-gradient 隔离 action expert 的梯度，X-VLA 干脆冻结整个骨干，只动 prompt。一个软隔离，一个硬冻结。

代码层：AutoModel + inference_api

现在打开 baselines/X-VLA/run.py。第一个和 π0 截然不同的地方是模型加载方式：X-VLA 没有独立 model server，模型直接加载进 eval client 进程，用 HuggingFace 的 trust_remote_code 机制：

baselines/X-VLA/run.py
model = AutoModel.from_pretrained(
    args.model_path, trust_remote_code=True, torch_dtype=torch.float32
).to(device).eval()
processor = AutoProcessor.from_pretrained(args.model_path, trust_remote_code=True)

trust_remote_code=True 意味着模型的自定义代码（包括那个 inference_api 方法）随 checkpoint 一起分发，AutoModel 动态加载执行。没有 WebSocket，没有第二个进程——推理就是一次本地方法调用。这让 X-VLA 的部署比 π0 简单一截，代价是模型 env 和 client env 必须能共存（X-VLA 的依赖恰好兼容 genmanip-client）。

prep 进：proprio 缩放

X-VLA 把所有低维状态拼成一个 proprio 向量喂进去，但拼之前对底盘和夹爪做了缩放：

baselines/X-VLA/run.py
c_joints = np.array(o["state.joints"], dtype=np.float32)
c_base   = np.array(o["state.base"],   dtype=np.float32)
p_base = c_base.copy()
p_base[:2] *= 10.0     # x, y 放大 10 倍
p_base[2]  /= 10.0     # theta 缩小 10 倍
proprio = np.concatenate(
    [c_joints, p_base, np.array(o["state.gripper"], dtype=np.float32) * 20.0]
)
action_chunk = model.inference_api(
    payload={
        "proprio": proprio,
        "language_instruction": o["instruction"],
        "image0": o["video.overlook_camera_view"],
        "image1": o["video.left_camera_view"],
        "image2": o["video.right_camera_view"],
        "domain_id": 0,
        "steps": 10,
    },
    processor=processor,
)

这些 magic number（base xy ×10、theta ÷10、gripper ×20）是为了把各通道的数值量级对齐到训练时 X-VLA 期望的尺度——底盘位移以米计是小数，乘 10 拉到和关节角可比的范围；夹爪开度同理乘 20。注意 payload 里那两个关键参数：domain_id: 0（用默认 soft prompt）和 steps: 10（flow matching 的 Euler 积分步数，和 π0 一样 10 步）。同样只用 overlook/left/right 三路图，丢弃 top 视角。

prep 出：反缩放、gripper 阈值门控、base_is_rel=False

出来的 chunk 要把刚才的缩放反着除回去，然后处理夹爪和底盘：

baselines/X-VLA/run.py
action_chunk[:, 12:14] /= 10.0     # base x, y 反缩放
action_chunk[:, 14]    *= 10.0     # base theta 反缩放
action_chunk[:, 15:19] /= 20.0     # gripper 反缩放

for act in action_chunk[:args.used_chunk_size]:
    grip = ( ([act[15], act[16]] if act[15] >= 0.001 else [-0.01, -0.01])
           + ([act[17], act[18]] if act[17] >= 0.001 else [-0.01, -0.01]) )
    deploy_action_chunk.append({
        "action": act[:6].tolist() + grip[:2] + act[6:12].tolist() + grip[2:],
        "base_motion": act[12:15].tolist(),
        "control_type": "joint_position",
        "is_rel": False,
        "base_is_rel": False,
    })

(1) gripper 阈值门控。夹爪不直接用模型输出，而是过一道判断：若 act[15] >= 0.001 就用预测值，否则强制成 [-0.01, -0.01]（明确"闭合/不动"）。这是把连续输出在很小的开度上钳到一个确定状态，避免夹爪在接近 0 的噪声里抖动。这是 X-VLA 特有的后处理风格——后面 InternVLA-A1 也门控夹爪，但用的是不同阈值和不同离散值。

(2) base_motion 的排布。关节也要像 π0 那样重排成 [左臂6, 左夹2, 右臂6, 右夹2]；底盘则直接取 act[12:15]。

X-VLA 独一份：base_is_rel = False（绝对底盘）

三个 baseline 里，只有 X-VLA 用 base_is_rel: False——它的 base_motion 是绝对底盘目标，而不是逐步增量。π0 和 InternVLA-A1 都是 base_is_rel: True（相对）。这意味着 X-VLA 的模型直接预测"底盘该到的绝对位姿"，仿真端不再做累积；而另两者预测的是相对位移，要靠仿真把 $\Delta$ 积回去。同一个 base_motion 字段，语义被一个布尔值彻底改写——这是 4.3 那个 action 向量解剖里最容易踩的坑。

step_mode：chunk 还是逐步

X-VLA 的 run.py 用 --step_mode 显式支持两种部署节奏，默认 step：

baselines/X-VLA/run.py
if args.step_mode == "chunk":
    obs, done = eval_client.step({wid: deploy_action_chunk})   # 整段一次发回
else:
    for action in deploy_action_chunk:                          # 逐步发，每步一次 step
        obs, done = eval_client.step({wid: action})
        if done or obs[wid]['obs'].get('reset', False):
            break

chunk 模式把整段 used_chunk_size（默认 30）一次性发回，让仿真内部执行，省往返；step 模式逐个动作发，每步都能看到新 obs（但每次都用同一段已推好的 chunk，并不重新推理——重新推理发生在外层 while 循环的下一轮）。--used_chunk_size 30 控制每次推理用前多少步。和 π0 一样，每次 step 都裹在 try/except 里，失败就走 recreate_eval_client，按 --client_reinit_retries / --client_reinit_backoff 退避重连。

run_xvla_eval.sh：多 worker、GPU round-robin

启动脚本比 π0 简单（没有 server 那一层），核心是按 WORKER_IDS 起多个进程，并用 GPU_IDS 做 round-robin 绑卡：

scripts/run_xvla_eval.sh
worker_ids=(${WORKER_IDS//,/ })
gpu_ids=(${GPU_IDS//,/ })
for i in "${!worker_ids[@]}"; do
    wid="${worker_ids[$i]}"
    env CUDA_VISIBLE_DEVICES=${gpu_ids[$((i % ${#gpu_ids[@]}))]} \   # round-robin 绑 GPU
        python -u "$RUN_PY" \
        --model_path "$MODEL_PATH" --base_url "$BASE_URL" \
        --run_id "$RUN_ID" --token "$TOKEN" \
        --worker_id "$wid" --step_mode "$STEP_MODE" &
    pids+=($!)
done

i % ${#gpu_ids[@]} 是经典的 round-robin：worker 数多于 GPU 数时，多个 worker 轮流分到同一块卡。env 变量一目了然：MODEL_PATH / BASE_URL / RUN_ID / TOKEN / WORKER_IDS / GPU_IDS / STEP_MODE / LOG_DIR。脚本还带一个完整的 cleanup trap，会连同 CUDA 子进程一起 TERM/KILL——长跑评测里防僵尸进程的实用细节。多 host 时各 host 共享 RUN_ID、切分 WORKER_IDS，就能拼出更大的并行规模。

小结：同一个契约，另一种性格

X-VLA 用极小骨干 + soft prompt 解跨本体，用 trust_remote_code 把模型塞进 client 进程，用一堆缩放/反缩放对齐量级，用阈值门控稳住夹爪，并且独自选择了绝对底盘 base_is_rel=False。它和 π0 共享同一个 EvalClient 契约，性格却完全不同。下一节的 InternVLA-A1 会引入第三种性格——有状态的 action 队列和一个会"想象未来画面"的世界模型。
