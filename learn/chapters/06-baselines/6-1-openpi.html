π0 / π0.5（openpi）

Part 5 把 flow matching、action chunking、action expert 一行行公式推完了，Part 5.5 又把训练流水线拆成了账本。现在到了兑现承诺的时刻：这些理论，在真实的 openpi baseline 里到底长成什么样的代码？我们从最"教科书"的那个开始——π0 / π0.5。它会让你看到，理论与契约之间，只隔着一个 prep_input 和一个 prep_output。

两个进程：model server 与 eval client

π0 baseline 不是一个脚本，而是两组进程。这是它和后面两个 baseline 最大的结构差异，先记牢：

  • model server：跑 scripts/serve_policy.py，把训练好的 π0/π0.5 权重加载进显存，开一个 WebSocket 服务端口等推理请求。它只负责"给我 obs，还你 action chunk"，不碰 EBench 仿真。
  • eval client：跑 scripts/pi_eval_client_online.py，里面装着 genmanip-client（连 EBench 仿真服务器的那条线）和一个 WebsocketClientPolicy（连 model server 的那条线）。它是中间人：从仿真拿 obs，转发给 model server，把回来的 action 翻译成 EBench 契约格式再 step 回仿真。

为什么要拆两层 WebSocket？因为 π0 的推理 env（JAX/PyTorch + openpi 依赖）和 EBench 的 client env（genmanip-client）天然冲突——这正是 4.1 讲的依赖隔离哲学，在 baseline 内部又套了一层。model server 跑在 openpi 的 venv 里，eval client 跑在 GenManip-client 的 conda env 里，靠 WebSocket 解耦。

从 RL 看 policy 就是个无状态函数

把 model server 想成一个纯函数 $\pi_\theta: o \mapsto A$：给它观测，它返回动作 chunk。它没有 env、没有 reward、没有 rollout 循环——所有"环境交互"的状态都在 eval client 和仿真那一侧。π0 这里把"策略"和"环境循环"在进程级别切开了，这种洁癖在 RL 代码里很少见，但对部署极其友好。

prep_input：把 EBench obs 喂成 π0 的语言

eval client 的核心是 Pi0Client 类。它的 get_action 只有三步：把 obs 转成 model server 认得的字典、调 infer、再把动作翻译回契约。先看进的方向：

baselines/openpi/scripts/pi_eval_client_online.py
def prep_input(self, obs):
    observation = {
        "images/head":      obs[self.ids]["obs"]["video.overlook_camera_view"],
        "images/hand_left": obs[self.ids]["obs"]["video.left_camera_view"],
        "images/hand_right":obs[self.ids]["obs"]["video.right_camera_view"],
        "states/joint":     obs[self.ids]["obs"]["state.joints"],
        "states/gripper":   obs[self.ids]["obs"]["state.gripper"],
        "states/base":      obs[self.ids]["obs"]["state.base"],
        "prompt":           obs[self.ids]["obs"]["instruction"],
    }
    return observation

注意这里的相机映射：EBench lift2 有 4 个视角（1.3），但 π0 只吃三路图像——一路第三人称 + 两路手腕。openpi 把 EBench 的 overlook 当 head（第三人称），left / right 当两只手腕。top 视角直接丢弃。这是 π0 架构（PaliGemma 三图槽位）写死的限制，不是 EBench 的选择。

这个 dict 进了 model server 后，会经过 EBenchInputs 这个 transform，把字段塞进 π0 真正认的键名：

baselines/openpi/src/openpi/policies/ebench_policy.py
state = torch.cat([state_joint, state_gripper], dim=-1)   # 12 + 4 = 16
inputs = {
    "state": state,
    "image": {
        "base_0_rgb":        base_image,    # = head = overlook
        "left_wrist_0_rgb":  left_image,
        "right_wrist_0_rgb": right_image,
    },
    "image_mask": { "base_0_rgb": np.True_, "left_wrist_0_rgb": np.True_,
        "right_wrist_0_rgb": np.True_ if model_type == PI0_FAST else np.False_ },
}
一个细节：state 只有 16 维，没有 base

注意 EBenchInputs 拼的 state 是 joint(12) + gripper(4) = 16——base 不进 state。prep_input 传了 states/base，但 transform 没用它。底盘位置不喂给策略当观测；底盘的运动完全由输出侧的 base_motion 通道处理。这一点等下在 prep_output 里会再次印证。

prep_output：joint 重排、base 相对增量、base_is_rel=True

现在看出的方向。model server 吐出的是一个 (horizon, 19) 的 action chunk（EBenchOutputs 用 data["actions"][:, :19] 截出 19 维：12 关节 + 4 夹爪 + 3 底盘）。但这 19 维的排布顺序和 EBench 契约要的不一样，所以要重排：

baselines/openpi/scripts/pi_eval_client_online.py
def prep_output(self, action_chunk):
    actions = []
    last_base_motion = np.zeros(3)
    for i in range(self.horizon):
        action = action_chunk[i]
        gripper_joint = action[12:16]
        base_motion = action[16:19] - last_base_motion   # chunk relative to delta
        last_base_motion = action[16:19]
        joint = np.concatenate([
            action[:6],          # joints 0-5
            gripper_joint[0:2],  # left gripper
            action[6:12],        # joints 6-11
            gripper_joint[2:4],  # right gripper
        ])
        action_dict = {
            "action": joint, "base_motion": base_motion,
            "control_type": "joint_position",
            "is_rel": False, "base_is_rel": True,
        }
        actions.append({self.ids: action_dict})
    return actions

两件值得品的事：

(1) joint 重排。模型输出是 [6 左臂关节, 6 右臂关节, 2 左夹, 2 右夹]（关节集中、夹爪集中）；而 EBench 契约要的 16 维是 [6 左臂, 2 左夹, 6 右臂, 2 右夹]（每只手臂的关节+夹爪挨在一起，见 4.3）。这段 np.concatenate 就是把夹爪插回各自手臂后面。这种"同一个 19/16 维向量，不同 baseline 排布不同"的麻烦，是整个 Part 6 反复出现的主题。

(2) base 的相对增量。模型预测的 action[16:19] 是相对 chunk 起点的累积位移（一条逐步增长的底盘轨迹）。但 EBench 的 base_motion 期待的是每一步的增量 $\Delta$。于是用 base_motion = action[16:19] - last_base_motion 做差分，把"累积相对"还原成"逐步相对"，并打上 base_is_rel=True。

从 GAMES103 看 累积 vs 增量，就是位置与速度

模型输出的累积相对位移像一条位置轨迹 $x(\tau)$，而 EBench 要的逐步 $\Delta$ 像速度 $\dot x$。action[16:19] - last 正是对位置做一阶差分得到速度——你在数值积分里反过来做（速度积成位置）的同一套运算，只是方向相反。底盘控制器在仿真里再把这些 $\Delta$ 积回绝对位置。

主循环：契约的最小骨架

把这两个 prep 接上 EvalClient，主循环就是 Part 4 那张契约图的字面实现：

baselines/openpi/scripts/pi_eval_client_online.py
eval_client = EvalClient(base_url=base_url, worker_ids=worker_ids, run_id=run_id, token=token)
pi0_client  = Pi0Client(args.model_host, args.model_port, ids, args.horizon)
obs = eval_client.reset()
eval_finished = False
while not eval_finished:
    action_chunk = pi0_client.get_action(obs)
    try:
        obs, eval_finished = eval_client.step(action_chunk)
    except Exception as e:
        eval_client.close()
        eval_client = EvalClient(base_url=base_url, worker_ids=worker_ids, run_id=run_id, token=token)
        obs = eval_client.reset()

每一轮 get_action 推一整个 horizon 长的 chunk，step 把整段一次发回去——这是 chunk 模式（4.4），省网络往返。注意那个 try/except：传输一旦失败，就 close() 旧 client、重建、重新 reset()。三个 baseline 都有这套"出错就重建 client"的错误恢复，是 EBench 长跑评测下的标配。

launch 脚本：多 server、端口与 GPU 分配

怎么把这两组进程拉起来？看 launch_pi_onlineeval.sh。它先按 NUM_CLIENTS 起 N 个 model server，再起 N 个对应的 eval client：

scripts/launch_pi_onlineeval.sh
POLICY_CFG="pi05_ebench_all"   # TrainConfig.name of your model
Horizon=30                     # replan horizon
# ---- start N model servers ----
for ((i=0; i<NUM_CLIENTS; i++)); do
    export CUDA_VISIBLE_DEVICES=$(($i / 2))     # 每个 server 占一块（或共享）GPU
    PORT=$((8000 + 2*i))                        # 端口 8000, 8002, 8004 ...
    python scripts/serve_policy.py --port $PORT \
        policy:checkpoint --policy.config=$POLICY_CFG --policy.dir=$POLICY_PATH &
done
sleep 60
# ---- start N eval clients ----
for ((i=0; i<NUM_CLIENTS; i++)); do
    python scripts/pi_eval_client_online.py --horizon $Horizon \
        --model_host 0.0.0.0 --model_port $((8000 + 2*i)) \
        --worker_ids "$i" --run_id "$RUN_ID" --token $TOKEN --eval_host $URL &
done

几个要点。端口步长是 2（8000, 8002, 8004…），给每个 server 留出余量。GPU 用 i / 2 整除映射——意味着默认两个 server 共享一块 GPU（π0 推理显存够），你按自己卡的显存改这条。第 i 个 client 用 --model_port 8000+2i 精确连到第 i 个 server，并用 --worker_ids i 在 EBench 仿真上认领第 i 个 worker。这样 N 条 (server, client, worker) 流水线并行跑，互不干扰。

脚本顶部那一排环境变量就是你要填的全部配置：GENMANIP_PATH / NUM_CLIENTS / RUN_ID / POLICY_CFG / POLICY_PATH / Horizon / TOKEN / URL。其中 Horizon=30 是 replan horizon：每推一次用前 30 步，对应 Pi0Client(horizon=30) 的循环长度。

三个配置：all / mobile / tabletop

POLICY_CFG 不止一个。openpi 为 EBench 的三条 track 准备了对应的 TrainConfig，π0 和 π0.5 各一套：

config 名对应 track输出动作维度action_horizon
pi05_ebench_allgeneralist（全部）[:, :19]（含 base）50
pi05_ebench_mobilemobile_manip[:, :19]（含 base）50
pi05_ebench_tabletoptable_top_manip[:, :16]（无 base）50

关键区别在最后一列。all 和 mobile 用 EBenchOutputs，截 [:, :19]，带底盘；而 tabletop 用另一个 ebench_fixedbase_policy.EBenchOutputs，截 [:, :16]——桌面任务底盘不动，干脆把 base 通道去掉。这就解释了为什么同一个 baseline 要分三套配置：track 决定了机器人要不要走路，也就决定了动作向量是 19 维还是 16 维。（π0 系列还有平行的 pi0_ebench_* 三套，区别只是模型用 π0 而非 π0.5。）

把理论落回代码

到这里，Part 5 的每个概念都在 π0 baseline 里找到了落点：flow matching 与 action expert 藏在 model server 的权重里（serve_policy.py 加载、infer 用 10 步 Euler 积分采样）；action chunking 是那个 horizon=30 的 replan；契约则由 eval client 的 prep_input/prep_output 两个翻译层兜住。π0 是三个 baseline 里最"标准"的——下一节的 X-VLA 会用完全不同的模型加载方式和底盘约定，把这套契约推到另一个极端。
