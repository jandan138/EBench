InternVLA-A1：MoT + visual foresight

前两个 baseline，模型都是"看一眼，出动作"。InternVLA-A1 多了一件事：它会先想象接下来画面应该长什么样，再据此决定怎么动。这个"先想象后行动"的世界模型直觉，加上它在代码里那个有状态的 action 队列，让它成为三个 baseline 里部署逻辑最丰富的一个。

模型层：Mixture-of-Transformers 三专家

InternVLA-A1（arXiv 2601.02456，InternRobotics）的骨架是一个 Mixture-of-Transformers（MoT）。注意它和 5.3 讲的 π0 那种 action expert 不是一回事：π0 是"两元 mixture"（图文一套权重、机器人 token 一套权重）；InternVLA-A1 把三种能力各做成一个 Transformer 专家：

  • understanding expert——理解视觉与语言（基于 InternVL3 / Qwen3-VL 系视觉语言模型）；
  • generation expert——预测未来的视觉状态，也就是 visual foresight：像一个世界模型，先在脑子里把"做完这步画面会变成什么样"想象出来；
  • action expert——用 flow matching 出 action chunk（和 π0 同源的连续动作头）。

三个专家共享一次 unified masked self-attention——它们不是三个独立网络，而是在同一次 attention 里通过 mask 互相交互（理解的结果能影响想象，想象能约束动作）。3B 版本基于 Qwen3-VL-2B，代码里对应 QwenA1Policy。它在 692M 帧（真实 + 合成 + 人类视频）上预训练，开 torch.compile 后约 ~13Hz。

从 RL 看 generation expert 是个 model-based 的世界模型

在 model-based RL 里，你学一个 dynamics model $\hat{s}_{t+1} = f(s_t, a_t)$，用想象的 rollout 帮助决策。InternVLA-A1 的 generation expert 干的正是这件事，只不过它预测的"下一状态"是未来的图像帧，而不是低维 state。它把世界模型的想象作为一路条件，喂给 action expert——这是 model-based 思想和 imitation/flow-matching 策略的一次缝合。是否真去解码那张想象的图，由代码里的 decode_image 开关控制。

代码层：QwenA1Policy 与有状态 wrapper

打开 baselines/InternVLA-A1/inference.py。和 X-VLA 一样，模型直接加载进 client 进程（无独立 server），但走的是 LeRobot 的 policy 接口而非 HuggingFace AutoModel：

baselines/InternVLA-A1/inference.py
config = PreTrainedConfig.from_pretrained(ckpt_path)
assert isinstance(config, QwenA1Config)
policy = QwenA1Policy.from_pretrained(config=config, pretrained_name_or_path=ckpt_path)
policy.cuda().to(dtype).eval()

真正特别的是包在外面的 QwenA1PolicyWrapper。它不是无状态的——它持有一个 action 队列和几条图像历史，这是前两个 baseline 都没有的。

有状态 action 队列：队列空了才推理

核心逻辑藏在 get_action 里：每步先看队列空不空，空了才推一整段，否则直接从队列头取一个：

baselines/InternVLA-A1/inference.py
def get_action(self, obs: dict) -> dict:
    if not self.action_plan:          # 队列空 → 推一段新的
        self._plan_action(obs)
    if self.action_plan:
        action = np.asarray(self.action_plan.popleft(), dtype=np.float32)   # 否则弹一个
        ...

这是 receding horizon replan（5.1）在客户端的显式实现。对比一下三者的触发节奏：π0 每轮 while 循环都重推一整段；X-VLA 在 step 模式下逐步发但每轮重推；而 InternVLA-A1 是队列驱动——一段 chunk 一步步 popleft 消耗完，才触发下一次推理。action_plan 是个 deque(maxlen=action_horizon_size)，默认装 30 步。

从 GAMES103 看 队列就是一个预算好的轨迹缓冲

想象你的积分器一次解出未来 30 帧的轨迹，存进缓冲，然后逐帧播放；播完了再解下一段。这正是离线轨迹生成 + 在线回放的模式。队列 popleft 是回放指针，_plan_action 是重新解算。比每帧都重解省算力，但缓冲越长，对突发扰动越迟钝——这就是 5.1 那个 chunk 长度的经典 tradeoff，在这里是 action_horizon_size 这个数字。

temporal image pair 与 base 置零

因为有 visual foresight，InternVLA-A1 喂模型的不是单帧，而是 [过去帧, 当前帧] 一对——让世界模型看到画面在怎么变。它维护三条图像历史 deque，按 image_history_interval（默认 15）回看一帧：

baselines/InternVLA-A1/inference.py
past_idx = max(len(self.head_color_history) - self.args.image_history_interval - 1, 0)
image_head_with_history = torch.stack(
    [self.head_color_history[past_idx], self.head_color_history[-1]], dim=0)   # [past, current]

state 这一侧有个容易忽略的处理：拼 state 时把 base 三维置零，只保留维度、抹掉数值：

baselines/InternVLA-A1/inference.py
self.chunk_start_base = np.asarray(base_state, dtype=np.float32).copy()   # 记下 chunk 起点底盘
zero_base = np.zeros_like(base_state)
new_state = np.concatenate([joints, zero_base, gripper_state])            # base 置零再拼

注释说这是为了和训练时的预处理一致——训练时底盘 state 就被置零了，所以推理也要置零，否则分布不匹配。但它在这之前先把当前底盘 chunk_start_base 存下来，因为出来的底盘预测是相对这个起点的，待会儿要加回去。

推理后处理：unnormalize、delta 加回、gripper 离散化

_plan_action 里推理 + 后处理是这一节最密的一段，拆开看：

baselines/InternVLA-A1/inference.py
action_pred, _ = self.policy.predict_action_chunk(inputs, decode_image=self.args.decode_image)
action_pred = action_pred[0, : self.args.infer_horizon, :19]          # 取前 infer_horizon 步、前 19 维
action_pred = self.unnormalize_fn({"action": action_pred})["action"]  # 反归一化（stats_key 如 lift2）

if self.args.action_mode == "delta":
    init_action = torch.from_numpy(new_state)[None].float().cuda()
    action_pred[:, :12] += init_action[:, :12]                        # 关节 delta 加回 init state

action_pred_joints  = action_pred[:, :12]
action_pred_gripper = action_pred[:, 15:]
action_pred_base    = action_pred[:, 12:15]
action_pred_gripper[action_pred_gripper < 0.005] = 0.0                 # gripper 离散化
action_pred_gripper[action_pred_gripper > 0.005] = 0.044
action_pred = torch.cat([action_pred_joints, action_pred_gripper, action_pred_base], dim=-1)
self.action_plan.extend(action_pred[:self.args.action_horizon_size].cpu().numpy())

四个动作连起来：(1) 从原始输出切 [0, :infer_horizon, :19]（默认 infer_horizon=50，但队列只装前 action_horizon_size=30）。(2) unnormalize_fn 用 stats.json[stats_key]（默认 lift2）按 mean/std 反归一化——X-VLA 是手写缩放系数，这里是用数据集统计量。(3) delta 模式：模型预测的是关节增量，要 += init_action 把当前关节角加回去变成绝对目标。(4) gripper 离散化成两个值：< 0.005 → 0.0（闭）、否则 0.044（开）——和 X-VLA 的阈值门控异曲同工，但这里是离散成两个具体开度，不是钳到 -0.01。

base：相对加回 chunk 起点，base_is_rel=True

底盘的处理是这一节的点睛之笔，因为它把 state 置零和 chunk_start_base 串了起来。在 get_action 弹出每个动作时：

baselines/InternVLA-A1/inference.py
curr_base = np.asarray(obs["obs"]["state.base"], dtype=np.float32)
if self.chunk_start_base is None:
    self.chunk_start_base = curr_base.copy()
predicted_rel_base = action[16:]                          # 模型出的相对底盘位移
target_base_abs    = self.chunk_start_base + predicted_rel_base   # 加回 chunk 起点 = 绝对目标
base_motion        = target_base_abs - curr_base          # 再减当前 = 这一步该走的增量
output["base_motion"] = base_motion.tolist()
output['base_is_rel'] = True
output["control_type"] = "joint_position"

逻辑链：模型预测的是相对 chunk 起点的位移 → 加 chunk_start_base 得到绝对目标 → 再减当前底盘得到"这一步要走多少"的增量，配 base_is_rel=True。这和 π0 的差分思路殊途同归（都把累积相对换成逐步相对），但 InternVLA-A1 多绕了一圈"绝对目标"，因为它的相对基准是 chunk 起点而非上一步。

_pack_action_fields：又一次 joint 重排

和 π0、X-VLA 一样，关节也要重排成契约的 [左臂6, 左夹2, 右臂6, 右夹2] 布局。这里封装成一个静态方法：np.concatenate([action[:6], action[12:14], action[6:12], action[14:16]])。三个 baseline 写了三遍同一件事——足见 4.3 那个 19 维向量的内部顺序有多容易出错。

args 与启动

CLI 用 tyro 解析 InferenceArgs dataclass，关键参数：ckpt_path、stats_key（默认 lift2，决定反归一化用哪套统计量）、resize_size（224）、image_history_interval（15，回看多远）、action_mode（delta）、infer_horizon（50，模型一次预测多长）、action_horizon_size（30，队列实际装多长）、以及连接用的 worker_ids / url / token / run_id。主循环和另两者同构：reset → get_action → step，出错重建 client，并在收到 obs["reset"] 时调 policy.reset() 清空队列和图像历史。外层 eval_pjsim.sh 再包一层多 worker 启动。

三者集齐，可以对比了

InternVLA-A1 带来了三个独有的复杂度：world-model 式的 visual foresight、有状态的队列驱动 replan、以及"相对 chunk 起点 → 绝对 → 逐步增量"的两段式底盘换算。但剥开这些，它和 π0、X-VLA 吐回去的仍是同一个 EvalClient 契约里的 action dict。下一节，我们把三者并排放进一张表，看清"同一个契约，三种迥异策略"到底差在哪几个字段。
