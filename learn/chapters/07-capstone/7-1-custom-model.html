接入自己的模型：最小 ModelClient

前面六个 Part 我们拆透了 EBench 的方方面面——仿真栈、契约、VLA 理论、三个 baseline 的全链路。现在轮到你了。本节只回答一个问题：要让 EBench 评测你自己的策略，你最少要写什么代码？答案出奇地短。

只有一个契约要满足

回忆 4.2 里我们反复强调的那条主循环：EvalClient 负责和 server 通信，你只需要往它手里塞一个能把 observation 变成 action 的东西。EBench 把这个东西叫做 ModelClient，它的契约可以缩成一行：

model_client.get_action(obs) -> action ——一次推理，一次调用。

没有基类要继承，没有注册表要登记。只要你的对象有这个方法、吃 4.2 那套 observation、吐 4.3 那套 action dict，它就接入完成了。三个官方 baseline（π0、X-VLA、InternVLA-A1）本质上都只是这个契约的不同填法。

从 RL 看 这就是一个 policy 的 act()

你在 RL 里写惯的 policy.act(state) -> action，和这里的 get_action(obs) 是同一个东西。区别只在 obs/action 的结构更复杂（多相机图 + 多段 state；双臂关节 + 底盘增量），以及策略内部不是 argmax 一个 Q，而是 flow matching 采样一段 chunk。接口形状没变。

最小骨架

下面是一个能跑通的 ModelClient 骨架。它把要做的四件事标了出来：加载权重（__init__）、清状态（reset）、解包 obs、产出 action dict。把每个 TODO 换成你自己模型的逻辑即可。

my_model_client.py
import numpy as np
import torch


class MyModelClient:
    """满足 EBench 契约的最小模型封装：get_action(obs) -> action。"""

    def __init__(self, ckpt_path, device="cuda"):
        # 1) 加载权重。只在进程启动时做一次，别放进 get_action。
        self.device = device
        self.policy = load_my_policy(ckpt_path).to(device).eval()
        # 训练时存下来的归一化统计量（mean/std）；推理要用同一套。
        self.stats = load_norm_stats(ckpt_path)
        self._t = 0  # 内部时间戳 / recurrent state 的占位

    def reset(self):
        # 2) 每个新 episode 调一次：清掉一切跨步携带的状态。
        #    若模型有 action 队列 / recurrent hidden / 历史帧缓存，全在这里清。
        self._t = 0
        if hasattr(self.policy, "reset"):
            self.policy.reset()

    @torch.no_grad()
    def get_action(self, obs):
        # 3) 解包 observation（按 worker_id 取，见 4.2）。
        o = obs["0"]["obs"]            # 单 worker：worker_id == "0"
        imgs = {
            "head":  o["video.top_view"],     # (480,640,3) uint8 HWC
            "left":  o["video.left_view"],
            "right": o["video.right_view"],
        }
        joints  = np.asarray(o["state.joints"],  dtype=np.float32)   # (12,)
        gripper = np.asarray(o["state.gripper"], dtype=np.float32)   # (4,)
        base    = np.asarray(o["state.base"],    dtype=np.float32)   # (3,) x,y,theta
        prompt  = o["instruction"]                                   # str

        # ---- TODO: 预处理 + 你的模型前向 ----
        images = self._preprocess(imgs)                  # 归一化 / resize / CHW
        state  = self._norm(np.concatenate([joints, gripper, base]))
        pred   = self.policy.infer(images, state, prompt)  # 出一段 (H, 19) chunk
        a      = self._unnorm(pred[0])                    # 取 chunk 第 0 步，反归一化

        # 4) 产出 action dict（契约形状见 4.3）。
        self._t += 1
        return {
            "0": {
                "action": a[:16].astype(np.float32),   # 6 左关节+2 左夹+6 右关节+2 右夹
                "base_motion": a[16:19].astype(np.float32),  # Δx, Δy, Δyaw
                "control_type": "joint_position",
                "is_rel": False,        # 臂为绝对关节角；相对则 True
                "base_is_rel": True,    # 底盘多为相对增量
            }
        }

就这些。把 load_my_policy / _preprocess / infer 换成你的实现，配上 4.2 的那段 reset → get_action → step 主循环，你就能在本地 server 上跑评测了。回看三个 baseline，它们多出来的代码全是填充这个骨架的细节：π0 的 prep_input/prep_output 做通道重排（6.1），X-VLA 的 proprio 缩放与 gripper 门控（6.2），InternVLA-A1 的 action 队列与 temporal image pair（6.3）——骨架一模一样。

可选：get_action_chunk，省掉往返

如果你的策略一次出一整段 chunk（flow matching 模型几乎都是），实现可选的 get_action_chunk(obs) 能让 server 在内部整段执行，只在 re-inference 点把观测回传给你，省掉每一步的网络往返。代价是 server 要带 --no_save_process 起。回顾 4.4 的 chunk vs step 对比。

my_model_client.py（追加）
    @torch.no_grad()
    def get_action_chunk(self, obs):
        """返回长度 chunk_size 的 list[action dict]；server 内部逐个执行。"""
        o = obs["0"]["obs"]
        images = self._preprocess({
            "head":  o["video.top_view"],
            "left":  o["video.left_view"],
            "right": o["video.right_view"],
        })
        state = self._norm(np.concatenate([
            o["state.joints"], o["state.gripper"], o["state.base"]
        ]).astype(np.float32))

        chunk = self._unnorm(self.policy.infer(images, state, o["instruction"]))  # (H,19)
        chunk = chunk[: self.chunk_size]           # 只回 chunk_size 步（受 receding horizon 约束）
        return [self._to_action_dict(a) for a in chunk]

注意：执行的步数（chunk_size）通常短于模型预测的 horizon——这正是 5.1 的 receding horizon：预测 50、只执行 16/25，再 replan。chunk_size 由 gmp 的 --chunk_size 选项控制（4.5）。

三个最常踩的坑

接入失败十有八九不是契约写错，而是这三类静默错误——不报错，但分数趴在地上。

图像是 uint8 HWC，归一化得你自己来

obs 里的 video.{cam}_view 是 (480,640,3) 的 uint8、HWC 排布、0–255。绝大多数 VLM backbone 要 float、CHW、且按训练时的 mean/std 归一化（或 /255 到 [0,1]）。忘了这步，模型看到的是一片全白噪声，照样推理出动作，照样得 0 分却不报错。用和训练时完全一致的预处理。相机命名对齐也在这里：EBench 的 top/left/right/overlook 要映射到你模型期望的 head/hand_left/hand_right 等视角。

reset 必须真的清干净跨步状态

新 episode 的首步 obs 里 reset == True（见 4.2）。如果你的策略带 recurrent hidden state、action 队列、历史帧缓存（InternVLA-A1 就有 action 队列 + temporal pair），reset 里不清，上一个 episode 的残留会污染这一个的开头几步——表现为"每个 episode 一开始动作乱飘然后才正常"。在主循环里务必 if obs[...]["reset"]: model.reset()。

chunk_size 要调，长短各有代价

执行多少步再 replan 直接影响成绩，且和任务类型相关。长 chunk 动作平滑、网络往返少，但对扰动/接触反应迟钝（dexterous 任务吃亏）；短 chunk 反应快、纠错及时，但 covariate shift 暴露更多、也更费网络（5.1）。X-VLA 默认 --used_chunk_size 30 是个合理起点；precise 任务可往下调，long-horizon 可往上。先在 val_unseen 上扫几个值，别在 test 上调参。
