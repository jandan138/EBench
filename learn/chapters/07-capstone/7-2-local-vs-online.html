本地 vs online，读懂诊断报告

你的 ModelClient 写好了。现在有两条路：在自己机器上起 server 跑本地评测，或登上 internrobotics.shlab.org.cn/eval 打online challenge。两者契约相同、流程几乎一致，差别都在边界上。本节把这两条路并排放，再教你读懂跑完后那张诊断报告。

本地评测：自己当裁判

本地评测你同时拥有 server 和 client。先起 server（它把 Isaac Sim 仿真当黑盒包起来，见 4.1），再跑 client worker 去打。

本地评测：两个终端
# 终端 A：起评测 server（GenManip / Isaac Sim 黑盒）
python ray_eval_server.py --host 0.0.0.0 --port 8087 --no_save_process

# 终端 B：提交一个 run，再跑 client worker 去打
gmp submit ebench/table_top_manip/val_unseen --run_id my_first_run
gmp eval -a r5a -g lift2 --worker_ids 0

# 查进度 / 拿结果
gmp status
# 结果落在 saved/eval_results/<task>/<run_id>/

几个要点：--no_save_process 是 chunk 模式（4.4）的前提；提交粒度可以是全部 ebench、单 track（ebench/table_top_manip/val_unseen）、甚至单任务（ebench/table_top_manip/test/collect_coffee_beans）。跨机时给 client 加 --host <ip> --port <port> 指向 server。一次 full pass 验证大约 30 分钟 / 8× RTX 4090——本地的好处是 val_train / val_unseen 全开放，可以随便调参、反复 debug。

本地评测里的"裁判"是谁

本地 server 用的是和官方同源的 GenManip 任务配置与打分逻辑（3.3），所以你本地看到的 val_unseen 分数对 online 的 test 有相当的预测力。但 test split 本身是留出的，本地拿不到——这是 3.2 训练/测试隔离的延续。

Online challenge：server 在云端

online 时 server 不在你手里——它跑在平台云端，你只负责跑 client。流程多一步登录拿 token，然后把 client 指向平台返回的 endpoint。

online challenge：注册一次 run
# 1) 登录 internrobotics.shlab.org.cn/eval 拿 token，存到 $T
# 2) 注册一次提交（当前线上仅开放 ebench_generalist）
gmp online submit \
  --base_url https://internrobotics.shlab.org.cn/eval \
  --token $T \
  --benchmark_set ebench_generalist \
  --model_name my_vla --model_type VLA
# 返回 task_id（即此次 run 的 run_id）+ endpoint
online challenge：用平台 endpoint 跑 client
from genmanip_client.eval_client import EvalClient

client = EvalClient(
    base_url=endpoint,        # 平台返回的 endpoint，不是本地 127.0.0.1
    token=T,
    run_id=task_id,           # 复用返回的 task_id
    worker_ids=["0"],         # 最多 16 个并发 worker
)
obs = client.reset()
done = False
while not done:
    if obs["0"]["obs"]["reset"]:
        model.reset()
    action = model.get_action(obs)   # 你在 7-1 写的那个
    obs, done = client.step(action)
client.close()
# 全部跑完后：gmp online stop

并发上限看你的 GPU（如 8× 4090）最多 16 个 worker
断连规则本地自管10 分钟无活动断连，同 task_id 可续跑

提交礼仪：别浪费配额

online 是共享资源，几条规矩能让你跑得顺、也不挡别人：

  • 同 run_id / task_id 续跑：worker 崩了、网断了、或你分批跑，重新连用同一个 task_id，平台会把结果接着累计，而不是新开一次评测。别每次 retry 都 online submit 一个新 run。
  • 并发别超 16：online 最多 16 个 worker。多进程跑就给每个进程不同的 worker_id（["0"]、["1"]…，见 4.2），别让一个 run 申请超过上限。
  • 留意 10 分钟断连：连续 10 分钟没有活动（没 step）会被断开。模型加载慢、或单步推理太久卡住，都可能触发。先在本地把推理延迟压下来再上 online；client 出错时按契约重建 client 继续（三个 baseline 都带 reinit 重试逻辑，X-VLA 还暴露了 --client_reinit_retries/backoff）。
  • 跑完 gmp online stop：正常结束后主动 stop，释放云端 worker。

读懂诊断报告

跑完你会拿到一组数。怎么从中看出"我的模型强在哪、弱在哪"，而不是只盯一个总分？分三层看，并且分清哪些是可验证事实、哪些是平台/愿景（这条贯穿全书，见 0.1 和 3.5）。

各 track 分数（地面真相）

最硬的数是 track score：某 track/split 下所有 episode 的 task score（每个 ∈ [0,1]，含子目标部分分与时序条件，见 3.3）取平均。leaderboard 聚合 track score。这是 docs 与代码里实打实存在的口径——三个 track（mobile_manip / table_top_manip / generalist）各一个数，先看这三个数谁高谁低，就知道你的策略偏移动操作、偏桌面精细、还是偏综合泛化。

val → test transfer（地面真相 + 一点平台）

把你本地的 val_unseen 分数和 online 的 test 分数并排：两者接近，说明你的策略真泛化了；val 高而 test 掉得多，几乎一定是过拟合到了 val 分布或被你拿来调过参。这就是 3.2 训练/测试隔离要守的东西。注意 split 的细分与每 track 的 episode 数在 docs 上仍标 WIP，具体数字以平台为准。

capability radar / heatmap（平台 & 愿景）

那张沿多轴展开的 capability profile 雷达图（0.1 里你玩过的那种）、以及任务级 heatmap、transfer curve——它们出现在 README 与线上平台页面上，是把 track score 进一步按"能力轴"分解的愿景呈现（5-axis atomic diagnostic / 4-axis generalization）。

别把愿景当成 docs 里已落地的 API

README/paper 的 5-axis diagnostic、794 task instances、Test-Mini 是愿景语言；docs challenge 页当前并没有逐项列出这些 radar/heatmap 字段。读报告时：track score 与 val→test 对比当作硬事实去优化你的模型；雷达图/能力轴当作平台呈现层去理解方向，别照着某个轴名去硬抠数字。哪些是地面真相、哪些是愿景，3.5 讲得最细。

这一节你需要带走的

本地与 online 契约一致、流程同形，差别在：server 归谁、要不要 token、能碰哪个 split、并发/断连配额。读报告分三层——track score 和 val→test transfer 是硬事实，照它优化；capability radar 是平台/愿景呈现，照它理解方向。下一节是全书的收束：我们牵着一个 episode，从 reset 一路走到打分，把七个 Part 串成一条完整的线。
