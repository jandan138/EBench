为什么是 client–server：依赖隔离

从这一节起我们打开评测引擎的引擎盖。你会发现 EBench 的整个评测系统，被一条看似工程化、实则非常关键的设计决策切成了两半：server 跑仿真，client 跑模型，中间只隔着一条网络。先讲清楚为什么必须这样切，再看这一刀切出来的完整评测循环长什么样。

痛点：两套依赖谁也不让谁

EBench 的仿真地基是 NVIDIA Isaac Sim（我们在 2.1 拆过它 = USD + PhysX + RTX）。它不是一个轻量库，而是一整套 Omniverse 应用：固定到 Isaac Sim 4.1.0、CUDA 12.1、torch==2.4.0，还拖着一长串渲染、物理、USD 解析的原生依赖。它对 Python 版本、CUDA、PyTorch 的口味都很挑。

而你的 VLA 模型呢？它有自己的一套世界。一个 π0/π0.5 跑在 openpi 的环境里；X-VLA 要 transformers 的某个版本配 trust_remote_code；InternVLA-A1 要 Qwen3-VL 那一套。这些模型对 torch、CUDA、flash-attention 的版本同样挑剔，而且和 Isaac Sim 的口味几乎不可能对齐。

如果把模型和仿真硬塞进同一个 Python 进程、同一个 conda env，你立刻会撞上经典的 dependency hell：torch 版本一个要 2.4、一个要 2.6；CUDA 一个要 12.1、一个要 12.4；某个原生库被两边以不兼容的方式各自链接。即便你侥幸调通了一种模型，换一个 baseline 又得从头再来。评测平台要同时接纳 π0、π0.5、X-VLA、InternVLA-A1 以及你自己的模型——这种「每来一个模型就重装一次仿真栈」的世界是不可维护的。

这不是「微服务时髦」，是物理约束

很多系统拆成 client–server 是为了水平扩展。EBench 拆开的首要原因更朴素：Isaac Sim 这只重型黑盒和现代 VLA 的依赖树根本无法共存于一个进程。隔离不是优化项，是能不能跑起来的前提。

解法：server 是仿真黑盒，client 极简寄生

EBench 把这条裂缝变成了架构边界。一边是 server：它把整个 Isaac Sim 仿真——加载场景、推进物理、渲染 4 路相机、判分——全部封在自己那套重型环境里，对外只暴露一个统一的黑盒 API。它来自 GenManip（CVPR'25），那套曾压测过 1,500 并发 Isaac Sim 实例的仿真平台。

另一边是 client：一个极其轻量的 Python 包 genmanip-client。它不含任何仿真依赖，只负责「按一个固定协议收发 observation 和 action」。因为足够薄，它可以直接 pip install -e . 进你模型自己的 env 而不会污染任何东西。

把 client 装进你的模型环境
git clone https://github.com/InternRobotics/genmanip-client.git
cd genmanip-client && pip install -e .

于是分工变得干净：server 那边，你完全不用关心模型是什么，它只管把仿真状态吐出来、把动作执行下去；client 这边，你完全不用关心 Isaac Sim 怎么装，你只在自己熟悉的训练环境里写「拿 obs → 推理 → 回 action」。两边各自用各自最舒服的依赖，通过一条 HTTP/WebSocket 连接对话。这条连接还天然支持跨机——server 跑在带 8× RTX 4090 的仿真机上，client 跑在你的推理机上，加 --host/--port 即可（线上 challenge 更是直接连远端 endpoint，见 4.5）。

从 RL 看 这就是 env 与 agent 的物理分居

你在 RL 里早就习惯了 env 与 agent 的接口分离：obs = env.reset()、obs, reward, done = env.step(action)。EBench 没有发明新概念，它只是把这条本来在进程内的接口，推到了进程之间。server = env（Isaac Sim 仿真 + 判分），client = agent 的外壳（把你的模型包成符合协议的 caller）。Gym 那套 reset/step/done 的心智模型可以原样搬过来——区别只是 step 现在要跨网络，而且 reward 在评测里以 episode 末的 task score 形式出现（见 3.3）。

从 GAMES103 看 仿真器作为可被远程驱动的时间步进器

你在 GAMES103 写过的物理循环——「解析约束 → 推进一个时间步 → 写回状态」——正是 server 内部每次 step 在做的事，只不过被 Isaac Sim 的 PhysX 在 GPU 上并行到了整个场景。client 在外面看不到这些积分细节，它拿到的是积分之后的状态快照（关节角、底盘位姿、相机图）。把仿真器当成一个「外部可驱动的时间步进黑盒」，正是 client–server 想给你的抽象。

这一刀切出来的评测循环

边界划好之后，一次评测就是 client 和 server 之间反复打的一套乒乓。把它写成一句话的循环：server reset 给出首帧 obs → client 拿 obs 推理出 action → client 把 action 发回 server → server 在仿真里执行、推进、再吐下一帧 obs → 直到 server 报告 done。每个 episode 都是这个循环跑一遍，跑满整个 track/split 的所有 episode 就得到分数。

下面这个交互部件把这条时序铺开了。先建立直觉，后面几节会把循环里的每一格——obs 的字段（4.2）、action 的结构（4.3）、chunk 还是 step（4.4）——逐一拆开。

看 reset→obs→infer→step→done 的往返时序；切换 chunk / step 模式，注意 chunk 模式如何把多步执行收进 server 一侧、省掉中间往返（细节见 4.4）。

注意这张图里有一条贯穿全书的暗线：所有「聪明」都在 client 那一侧。server 只是个忠实、无状态记忆的执行器（它不知道你的模型在想什么）；模型怎么把视觉和指令变成动作、要不要做 action chunking、要不要 receding horizon replan——全是 client 的事。这也解释了为什么本书 Part 5 的 VLA 理论和 Part 6 的 baseline 全链路，几乎都在讲 client 这半边。

这一节你需要带走的

记住一句话：EBench 把仿真和模型用一条网络隔成两个进程，是为了让两套互不兼容的依赖各自活下去。server 是 Isaac Sim 黑盒，client 是能寄生进任何模型环境的极简包；两者用 reset→obs→infer→step→done 的循环对话。这条边界一旦理解，后面所有内容都只是在填这个循环的细节。
