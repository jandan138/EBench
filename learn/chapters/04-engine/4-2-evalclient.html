EvalClient 契约：reset/step/close 与 obs

上一节我们说「所有聪明都在 client 一侧」。这一节就盯住 client 的入口对象——EvalClient。它出奇地小：只有三个动作 reset / step / close，以及一个它递给你的 observation 字典。把这两样东西吃透，你就掌握了接入任何模型的全部协议。

三个动作撑起整条循环

EvalClient 把 4.1 那条 reset→obs→infer→step→done 循环包成了三个方法。先看完整骨架——这就是接入任何模型时你照抄的那段：

EvalClient 评测主循环
from genmanip_client.eval_client import EvalClient

client = EvalClient(base_url="http://127.0.0.1:8087", worker_ids=["0"], run_id="my_eval")
obs = client.reset()
eval_finished = False
while not eval_finished:
    if obs[worker_ids]["obs"]["reset"]:
        model.reset()
    action = model.get_action(obs)
    obs, eval_finished = client.step(action)
client.close()

三个动作各司其职：

  • reset()——向 server 要第一帧 observation，开启评测。它返回的 obs 就是循环的种子。
  • step(action)——把你算出的 action 发给 server，server 在仿真里执行、推进、判分，然后把下一帧 obs 和一个 eval_finished 布尔一起还给你。这是循环的心脏，每轮跳一次。
  • close()——所有 episode 跑完（eval_finished 为真）后收尾，释放连接。

注意循环条件用的是 eval_finished——它标志的是整段评测（这个 worker 分到的所有 episode）是否结束，而不是单个 episode 结束。server 会在内部把一个 episode 跑完、自动切到下一个 episode，并通过下面要讲的 reset 标志告诉 client「新 episode 开始了」。client 不需要自己数 episode 边界。

从 RL 看 和 Gym 几乎同构，但 done 的语义抬高了一层

对照 Gym：reset()↔env.reset()，step(action)↔env.step(action)。区别在 done：Gym 的 done 标志单个 episode 终止，而 EvalClient 的 eval_finished 标志整段评测终止；episode 内部的边界改由 obs 里的 reset 字段传达。这样设计是因为 server 要替你把一长串 episode 串起来连续跑，client 只关心「我还要不要继续 infer」。

reset 标志：清空你的 recurrent 状态

循环开头那行 if obs[worker_ids]["obs"]["reset"]: model.reset() 是极易被忽略却必须写对的一行。observation 里有个布尔字段 reset：它在每个 episode 的首步为 True，其余步为 False。它是 server 向 client 发出的「新 episode 开始，请把上一段的记忆全部清零」信号。

为什么重要？很多 VLA 推理是有状态的：

  • 做 action chunking 的模型（见 4.4 与 5.1）会缓存一整段未到期的 action chunk；
  • InternVLA-A1 维护一个 action queue（deque），空了才重新规划；还会用 temporal image pair（[past, current] 两帧）做推理，里面藏着上一帧的历史；
  • 任何带 KV cache 或历史帧缓存的策略，都把「上一个 episode 的轨迹」记在了内部。

如果新 episode 开始时不清空这些，上一段任务的残留动作和历史会污染新任务——机器人可能一开局就执行属于上一局的动作。所以看到 reset==True，就调你模型自己的 model.reset()，把 queue、cache、history 一律清空。

最常见的接入 bug

忘记响应 reset 标志，是接入有状态模型时最隐蔽的错误：单 episode 看着正常，连续评测时分数莫名其妙地塌。务必在 episode 首步清空一切跨步状态。

observation schema：逐字段拆开

obs 是个嵌套字典，按 worker 寻址：顶层 key 是 worker_id（字符串，如 "0"），下一层的 "obs" 才是这一帧的真正内容。所以你总是这样取：obs["0"]["obs"]["state.joints"]。那层 "obs" 字典的字段如下：

obs["<worker_id>"]["obs"] 的字段
{
    "instruction":      str,                  # 本 episode 的自然语言指令
    "state.joints":     (12,),                # 双臂关节角：左 6 + 右 6
    "state.gripper":    (4,),                  # 夹爪状态（左 2 + 右 2）
    "state.base":       (3,),                  # 底盘位姿 x, y, theta
    "state.ee_pose":    [[pos, quat], [pos, quat]],   # 双臂末端位姿 (=14)
    "video.top_view":   (480, 640, 3) uint8,  # 还有 left / right / overlook 共 4 路
    "timestep":         int,                  # 本 episode 内的步计数
    "reset":            bool,                  # 仅 episode 首步为 True
}

逐字段读一遍：

  • instruction——这一局要做什么的自然语言（如 “put the apple in the fruit bowl”）。它是 VLA 里的「L」，会被 tokenize 进 VLM。同一任务在数据集里常有多套释义（meta/tasks.jsonl），训练时见过多种说法，泛化轴里的 Instruction 扰动正是冲它来的（见 3.4）。
  • state.joints (12)——双臂的本体感觉，左臂 6 关节 + 右臂 6 关节。这是 VLA 里的 proprioception q_t，会和图像、语言一起作为条件喂给策略。
  • state.gripper (4)——两只夹爪的开合状态。
  • state.base (3)——移动底盘在世界系里的 x, y, theta。这是 lift2 「mobile」那一维的来源；纯桌面 baseline 会把它当背景。
  • state.ee_pose——双臂末端执行器位姿，每臂一组 [pos(3), quat(4)]，两臂合起来 14 维。当你用 ee_pose 控制模式时（见 4.3），这就是你要对话的坐标空间。
  • video.{cam}_view——4 路 480×640×3 的 uint8 RGB：top / left / right / overlook（录制 15 fps）。这是 VLA 里的「V」。注意各 baseline 只挑自己要的视角喂模型（如 head / hand_left / hand_right 的映射），并不是 4 路全用。
  • timestep (int)——episode 内步数，常用来做日志或触发周期性逻辑。
  • reset (bool)——上一节讲过的 episode 首步信号。
这正好是 VLA 的输入三元组

把 obs 重新归类，你会看到它恰好是 VLA 策略 $\pi(A_t \mid o_t)$ 的条件 $o_t$：视觉（4 路相机）+ 语言（instruction）+ 本体状态（joints / gripper / base / ee_pose）。Part 5 讲 flow matching 时反复出现的观测前缀 $o_t=[I^1..I^n, \ell_t, q_t]$，落到 EBench 这里就是这个字典。obs 是「世界给模型看的」，action（下一节）是「模型还给世界的」。

worker_id 寻址：为并行留的接口

为什么 obs 要先按 worker_id 套一层？因为 EBench 天生为并行评测设计。一个 EvalClient 在构造时就拿一个 worker_ids=["0"] 列表；不同的推理进程认领不同的 worker_id，各自连同一个（或远端）server 的不同评测分片。obs 用 worker_id 当顶层 key，就是为了让一个 client 能寻址自己负责的那个（或那几个）worker 的观测。

单机调试时你通常只用 "0" 一个 worker，循环里直接 obs["0"]["obs"] 即可。但这层寻址在你要开多进程、或上线 challenge 跑 16 并发时就派上用场了——并行的机制和上限我们留到 4.4 和 4.5 详谈。

这一节你需要带走的

EvalClient 的契约小到可以背下来：reset 拿首帧、step 收发一来一回并报告整段是否结束、close 收尾；observation 是一个 obs[worker_id]["obs"] 的嵌套字典，装着视觉 + 语言 + 本体状态，外加一个 episode 首步置 True 的 reset 标志——看到它就清空你模型的所有跨步记忆。这套协议对所有 baseline 完全一致，你自己的模型也照这个壳接入（Part 7 会实战一遍）。
