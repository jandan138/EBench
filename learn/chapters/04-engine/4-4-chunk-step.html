Chunk vs step、receding horizon 与并行

上一节末尾留了个问题：每个 step 只发一帧 action，还是一口气发一整段？这背后是两种评测模式（step 与 chunk）、一个来自 VLA 理论的核心直觉（receding horizon），以及 EBench 怎么把同一套循环并行到多个 worker。这一节把这三件事讲清楚。

两种模式：step 与 chunk

现代 VLA 很少一次只预测一帧动作；它们做 action chunking——一次推理吐出未来 H 帧（π0 的 H=50）。这跟评测协议产生了张力：模型脑子里是一整段，但 server 的 step 是一帧一帧推进的。EBench 给了两种对接方式。

step 模式是最直白的：client 每个 server step 调一次模型的 get_action(obs)，拿回一帧 action，发给 server，往返一次。即便模型内部预测了一整段，client 也可以自己缓存这段、每步 popleft 一帧出来——这时模型表面上仍是「一次推理一帧调用」的 get_action 接口（InternVLA-A1 的 action queue 就是 client 侧自缓存的典型）。

chunk 模式更激进：client 实现 get_action_chunk(obs)，一次性把整段长度 chunk_size 的 action list 交给 server；server 在自己内部把这一整段连续执行下去，只在需要重新推理（re-inference）的那个点才把观测回传给 client。中间那些步的 obs↔action 往返被整个省掉了。

两种模型接口
# step 模式：一次推理，返回一帧
def get_action(self, obs):
    return action            # 形如契约里的一帧

# chunk 模式：一次推理，返回长度 chunk_size 的一整段
def get_action_chunk(self, obs):
    return [action_0, action_1, ..., action_{chunk_size-1}]
chunk 模式需要 server 端 --no_save_process

chunk 模式让 server 替你连续执行整段动作。要打开这条「省往返」的快路，server 必须以 --no_save_process 启动（见 4.5 的 ray_eval_server.py 命令）。没有它，server 会按逐步保存过程的方式运行，chunk 的批量执行优化就不成立。

取舍很清楚：chunk 模式省网络往返、吞吐更高，尤其当 server 远在另一台机、网络延迟不可忽略时收益最大；代价是这一整段执行期间 server 看到的是「开环」的——模型在段内不重新看世界。step 模式每步都能（最多）重新决策，反应更灵，但每帧一次往返，远程评测时延迟叠加得厉害。下面这个时序部件把两种模式的往返画在一起。

receding horizon：为什么是「开环执行 + replan」

chunk 模式开环执行整段，会不会因为「执行期间不看世界」而失控？这正是 receding horizon（滚动时域）这个直觉要回答的。它的核心想法：预测一长段，但只执行其中前面一截，然后丢掉剩下的、用新观测重新预测——决策点像一个不断向前滑的窗口。

为什么不每步都重新预测（即纯 step）？因为 action chunking 的全部意义，是用「更少、更稀疏的决策点」来对抗 compounding error / covariate shift：单步误差会把机器人推到训练没见过的状态，误差滚雪球；预测整段、稀疏 replan 能缩短有效决策 horizon、让动作更平滑。但执行得太长又会反应迟钝。于是折中——执行一截就 replan。π0 的实测口径：20Hz 机器人执行 16/50 帧后每 0.8s 重推；50Hz 执行 25 帧后每 0.5s 重推（π0 还发现 ACT 的 temporal ensembling 反而更差，所以选了「开环执行 + replan」而非逐帧融合）。

从 RL 看 和 MPC 的滚动时域是同一招

receding horizon 这个名字你在最优控制 / MPC 里见过：求解一段 horizon 的最优序列，只执行第一步（或前几步），下一时刻拿新状态重解。VLA 的 chunk + replan 是它的学习版——「求解」换成了一次策略前向（flow matching 出 chunk，见 5.5），但「预测一长段、只走一截、再重规划」的骨架完全一致。长 chunk = 更平滑但更迟钝；短 chunk = 反应快但 covariate shift 更多——这条 tradeoff 是 5.1 的主线。

这一节只建立直觉；receding horizon 的完整展开——它从 ACT 来、怎么对抗 compounding error、H 和 exec horizon 怎么选——在 5.1，那里也有专门的 action-chunking 交互部件让你拨弄 H 与 exec 看效果。

多 worker 并行

一次 full pass（一个 track/split 的所有 episode）要跑很多 episode；串行会很慢。EBench 的并行模型很直接：不同的推理进程认领不同的 worker_id，各自跑一条独立的 reset→obs→infer→step 循环，分担整体的 episode。这也解释了 4.2 里 obs 为什么要先按 worker_id 套一层——那层寻址就是为并行预留的。

具体怎么开：构造 EvalClient 时给不同进程不同的 worker_ids，或用 gmp eval --worker_ids 指定（见 4.5）。真实 baseline 就是这么做的：X-VLA 的 run_xvla_eval.sh 每个 WORKER_ID 起一个进程、GPU_IDS round-robin 分配显卡；openpi 起 N 个 model server 配并行 client。本地你有几张卡就能开几个 worker；这也是 README 里「30 分钟 / 8× RTX 4090」那个数字背后的并行度来源。

online 的并发上限是 16

本地并行的 worker 数受你的显卡数限制；但上线到 online challenge 时，平台对单次提交最多允许 16 个并发 worker。这个上限和「10 分钟无活动断连」「同 task_id 续跑」一起，构成线上评测的并发约束——完整流程见下一节。

这一节你需要带走的

三件事：(1) step 模式每步一往返、灵活；chunk 模式一次发整段、server 内执行、省往返，但需 server --no_save_process——对应模型侧的 get_action 与 get_action_chunk。(2) receding horizon = 预测一长段、只执行一截、用新观测 replan，是 chunk 开环执行不失控的关键，细节在 5.1。(3) 并行靠不同进程认领不同 worker_id，本地随显卡数，online 最多 16 并发。
