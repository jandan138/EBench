Action 向量解剖：16/19-D 与 base_motion

obs 是世界给模型看的，action 是模型还给世界的。但「还回去」这一步藏着最多坑：双臂关节怎么排序、夹爪插在哪、移动底盘为什么单独拎出来、相对还是绝对。这一节把 server 契约里的 action 字典逐位解剖，再看模型那侧吐出来的一坨 19-D raw 向量是怎么被重排进这个契约的。

server 契约里的 action 字典

step 收下的 action 不是一个裸数组，而是一个按 worker_id 寻址的字典，每个 worker 配一组结构化字段。完整长这样：

step(action) 期望的 action 字典
{
    "0": {
        "action":       (16,),   # 6 左臂关节 + 2 左夹爪 + 6 右臂关节 + 2 右夹爪
        "base_motion":  (3,),    # 底盘增量 dx, dy, dyaw
        "control_type": "joint_position",
        "is_rel":       False,   # 臂为相对量则 True
        "base_is_rel":  True,    # 底盘几乎总是相对（增量）
    }
}

逐字段读：

  • action (16)——双臂的指令，顺序是 6 左臂关节 → 2 左夹爪 → 6 右臂关节 → 2 右夹爪。注意夹爪是交错插在两臂之间的，不是「12 关节在前、4 夹爪在后」。这个顺序是契约硬性规定，排错了机器人就乱动。
  • base_motion (3)——移动底盘的运动指令 dx, dy, dyaw。它被单独拎出来，不混进那 16 维里。
  • control_type——控制空间，"joint_position"（关节位置控制）是默认，也可切 "ee_pose"（末端位姿控制，见下）。
  • is_rel——臂的 action 是绝对目标还是相对增量。False 表示绝对（直接给目标关节角）；模型若输出 delta 则置 True。
  • base_is_rel——底盘是相对还是绝对。底盘几乎总是True（增量），因为「往前挪 dx」比「移动到世界系某绝对坐标」对策略更自然。
为什么 base 要从 16 维里拆出来

双臂关节和移动底盘是语义截然不同的两套自由度：关节通常给绝对目标角、单位是弧度、动力学是机械臂的；底盘是 SE(2) 平面运动、几乎总以增量表达、动力学是轮式底盘的。把它们硬拼成一个向量，会让 is_rel 这种「相对/绝对」语义没法分别设定。拆成 action + base_motion，并配独立的 is_rel / base_is_rel，正是为了让两套自由度各自选最自然的表达——这也是 lift2「mobile manipulation」本体在接口层的直接体现。

模型的 19-D raw action 如何重排进契约

这里要解开一个最容易混的点：你常听到「模型输出 19 维 action」，而契约里却是「16 + 3」。两个数字说的是同一件事：16（双臂）+ 3（底盘）= 19。模型内部把整套自由度当成一个 19-D raw action 向量一次性预测出来，client 再把它拆开、重排，塞进上面那个结构化字典。

各 baseline 的重排细节略有不同（Part 6 会逐个细看），但模式一致——拿模型吐出的 19 维 chunk，前段对应双臂、后 3 维对应底盘，按契约要求的顺序搬运：

概念示意：19-D raw → 契约字典（顺序为关键）
# raw 是模型直接吐的一帧 (19,)
arm   = pack_arm(raw)          # 重排成 [j0..j5(左), gripL, j6..j11(右), gripR] -> (16,)
base  = raw[16:19]             # 取末 3 维做底盘 -> (3,)
# 多数 baseline：base_motion = base - last_base（化成增量），base_is_rel=True
action = {
    "0": {
        "action": arm, "base_motion": base_motion,
        "control_type": "joint_position",
        "is_rel": False, "base_is_rel": True,
    }
}

几个真实 baseline 的具体做法（FACTS 核对过的细节，供对照）：

  • openpi（π0/π0.5）：prep_output 重排 joint 顺序，base_motion = action[16:19] - last（即化成相对增量），base_is_rel=True；generalist 输出取 action[:, :19]，fixedbase 配置只取 16。
  • X-VLA：把模型输出 chunk 反缩放还原后，按 [j0-5, gripL, j6-11, gripR] 的顺序 _pack，且独一份地用 base_is_rel=False（绝对底盘）。
  • InternVLA-A1：_pack_action_fields 重排成 [j0-5, gripL, j6-11, gripR]；底盘 target = chunk_start_base + predicted_rel_base，base_is_rel=True。

这三家排关节顺序、处理底盘、缩放约定都不一样，但它们最终都收敛到同一个 16+3 契约字典——这正是 client–server 边界的价值：server 只认契约，不认模型。下面这个部件让你把 19-D 向量一位一位点亮，并切换看不同 baseline 的映射差异。

interactiveaction-dissector：19-D action 逐位解剖与 baseline 映射
逐位高亮 19 维 raw action，看它如何切成 16(双臂) + 3(底盘)；切换 baseline，对比各家关节重排顺序与底盘相对/绝对约定。
从 GAMES103 看 关节位置控制 = 给积分器设目标，而非直接给力

control_type="joint_position" 意味着你交的是目标关节角，仿真内部的控制器（PD 之类）再把目标变成力矩、由 PhysX 积分推进。这正好接上你在 GAMES103 里的「约束/目标驱动 → 数值积分 → 更新状态」那条链：action 设的是约束目标，server 负责把它积分成真实运动。底盘的 base_motion 增量则像每步给 SE(2) 位姿加一个小位移，由仿真消化成轮式运动。

替代控制模式：ee_pose

关节位置不是唯一的对话语言。把 control_type 切成 "ee_pose"，你就改用末端执行器位姿说话：每臂提供一组 [pos, quat, gripper]——目标位置、目标姿态四元数，外加夹爪开合。这和 obs 里的 state.ee_pose（[[pos,quat]×2]=14）是同一个坐标空间，闭环很自然。

什么时候用哪个？关节位置控制更贴近策略直接输出的量、在仿真里稳定可预测，是多数 baseline 的默认；ee_pose 让你在任务空间（笛卡尔）里描述意图——「把手移到这个位姿」——对需要精细对位的 dexterous & precise 任务（插销入孔、套齿轮）有时更直观，代价是 server 端要解 IK 把任务空间目标变回关节空间（cuRobo 那套 GPU IK，见 2.2）。两种模式走的是同一个 step 协议，只是字段语义随 control_type 切换。

这一节你需要带走的

记住三件事：(1) action 契约是 {"action"(16) + "base_motion"(3)}，双臂那 16 维的顺序是 6 左关节 + 2 左夹 + 6 右关节 + 2 右夹（夹爪交错插入），底盘单拎出来；(2) 模型吐的 19-D raw = 16 双臂 + 3 底盘，client 负责重排进契约，各 baseline 的关节顺序与底盘相对/绝对约定各不相同，但都收敛到同一契约；(3) is_rel/base_is_rel 控制相对还是绝对，control_type 可切 joint_position 或 ee_pose。
