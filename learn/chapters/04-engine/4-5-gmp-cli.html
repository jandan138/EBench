gmp CLI 与 online challenge

前四节讲的协议，最终都收束到一个命令行工具——gmp（genmanip-client 提供的 CLI）。这一节把它落地：先走本地的完整流程（起 server → 提交 → 跑 client → 看结果），过一遍子命令清单，再走 online challenge 从拿 token 到出诊断报告的链路。Part 4 在这里收口，并把你交给后面三个 Part。

本地流程：从起 server 到看结果

本地评测是「4.1 的 client–server」最直接的体现：一边起仿真 server，一边用 gmp 提交任务并跑 client worker。完整四步：

本地评测：server + 提交 + client + 查看
# 1. 起仿真 server（Isaac Sim 黑盒；chunk 模式需要 --no_save_process）
python ray_eval_server.py --host 0.0.0.0 --port 8087 --no_save_process

# 2. 提交一个评测任务：ebench/<track>/<split>
#    track ∈ {mobile_manip, table_top_manip, generalist}
#    split ∈ {val_train, val_unseen, test}
gmp submit ebench/generalist/test --run_id my_first_run

# 3. 跑 client worker（认领 worker 0；推理在你的模型 env 里）
gmp eval -a r5a -g lift2 --worker_ids 0

# 4. 查进度
gmp status

几个要点：

  • submit 的目标可粗可细：ebench 跑全部；ebench/<track> 单 track；ebench/<track>/<split> 单 split；甚至单任务，如 ebench/table_top_manip/test/collect_coffee_beans。
  • eval 里 -a r5a（agent）、-g lift2（本体）指定要评的对象，--worker_ids 0 认领 worker（多 worker 并行见 4.4）。
  • 跨机：server 在另一台机时，给 gmp eval 加 --host <ip> --port <port>（默认 127.0.0.1:8087）即可，协议不变。
  • 结果路径：跑完后结果落在 saved/eval_results/<task>/<run_id>/，gmp status 给进度，run_id 是你区分多次实验的标签。
一次 full pass 的体量

跑满一个 track/split 的全部 episode 约需 30 分钟 / 8× RTX 4090（README 口径）。这个数字背后正是 4.4 的多 worker 并行——有几张卡就开几个 worker 分担 episode。

gmp 子命令一览

gmp 是个小工具箱。常用子命令：

子命令作用
submit提交评测任务（ebench / 单 track / 单 split / 单任务）
eval启动 client worker 跑推理循环（-a / -g / --worker_ids）
status查看评测进度
plot绘制结果图表
clean清理产物（--dry-run 预演 / --all 全清）
visualizeRerun HTTPS viewer 可视化（需 Py3.11+ 与 [visualize] extra）

常用选项横跨多个子命令：--run_id（实验标签）、--host/--port（server 地址，默认 127.0.0.1:8087）、--worker_ids（认领哪些 worker）、--frame_save_interval（每隔几帧存一次）、--chunk_size（chunk 模式整段长度，见 4.4）。

online challenge：连远端、出诊断报告

本地能跑通后，把它推上 online challenge（internrobotics.shlab.org.cn/eval）。线上平台 24/7 运行，在留出的 Test-Mini 上评你的提交，并自动生成诊断报告（capability radar、validation→test transfer curve、generalization radar、task heatmap——这些是 README/平台口径的愿景报告，docs challenge 页未逐一列出，见 3.5）。整条链路：

online challenge：submit → 连 endpoint → 跑 → stop
# 1. 登录 internrobotics.shlab.org.cn/eval 拿到 token，导出
export T=<your-token>

# 2. 在线提交（当前仅允许 benchmark_set = ebench_generalist）
gmp online submit \
  --base_url https://internrobotics.shlab.org.cn/eval \
  --token $T \
  --benchmark_set ebench_generalist \
  --model_name my-model --model_type VLA
#    -> 返回一个 task_id（充当 run_id）和一个 endpoint

# 3. 用返回的 endpoint + task_id 跑 EvalClient（最多 16 并发 worker）
#    EvalClient(base_url=endpoint, token=$T, run_id=task_id, worker_ids=["0"])

# 4. 结束
gmp online stop

线上几条必须记住的纪律：

  • benchmark_set 限制：当前仅允许 ebench_generalist。
  • submit 的产物：返回 task_id（之后充当 run_id）+ endpoint；你用它们构造 EvalClient(base_url=endpoint, token=..., run_id=task_id, worker_ids=["0"]) 去连远端 server——和 4.2 的本地 EvalClient 同一个类，只是 base_url 指向云端、并带 token。
  • 并发上限 16：单次提交最多 16 个并发 worker（见 4.4）。
  • 10 分钟断连：10 分钟无活动会被断开；但同一个 task_id 可以续跑，断了重连即可接着评，不必从头。
  • 收尾：评完用 gmp online stop 结束这次提交。
本地 vs online 的差别只在「连哪台 server」

别被两套命令吓到：协议是同一套。本地是 EvalClient(base_url="http://127.0.0.1:8087", ...) 连你自己起的 ray_eval_server.py；online 是 EvalClient(base_url=endpoint, token=..., ...) 连平台分配的远端 server。reset/step/close 循环、obs schema、action 契约——全都不变。online 只是多了 token、并发上限、断连续跑这几条平台约束。

Part 4 收口，去往后面三个 Part

到这里，评测引擎的全貌齐了：为什么 client–server（4.1）、EvalClient 与 obs（4.2）、action 契约（4.3）、chunk/step 与并行（4.4）、gmp 与 online（4.5）。你现在能完整想象一个 episode 从 reset 到判分的全程，也能把任意模型接进这套协议。

但有一件事我们一直只点到为止：循环里那个 model.get_action(obs) 内部到底发生了什么？为什么要 action chunking、flow matching 怎么把噪声变成动作、action expert 是什么——这是 Part 5（VLA 理论）。理论备齐后，Part 6 会把 π0/π0.5、X-VLA、InternVLA-A1 三条全链路逐个拆开，看它们如何各自把 19-D raw action 重排进同一契约；最后 Part 7 让你亲手接入自己的模型、读懂诊断报告、追踪一个 episode 的全程。
