数据引擎 / benchmark / recipe：三层生态位

VLA 领域的名字满天飞：RoboGenesis、LabUtopia、AutoBio、GenManip、EBench……它们经常在一篇论文里同时出现，但不是同一类东西。这一节给你一个三格分类器：任何一个新名字出现，先问它是造数据的、出考题的，还是训模型的方法——归对格子，关系立刻清晰。

三个格子，不要混

格子是什么核心产出类比
数据引擎
data engine		自动生成训练数据的系统：造场景 → 执行任务 → 过滤成功 rollout → 导出训练数据集教材印刷厂
benchmark考试场 + 评分器：reset 场景、让模型尝试、算成功率与失败原因分数与诊断考场 + 阅卷组
model recipe训练模型的方法学：backbone、action head、loss、阶段、数据混合模型 checkpoint教学大纲

三者当然会握手——引擎产的数据喂 recipe，recipe 训出的模型进 benchmark——但职责边界决定了你能对每个系统提什么要求、下什么结论。混淆它们的典型症状：拿数据引擎去和 benchmark 比「任务多少」，或者把「模型在某 benchmark 赢了」错记到 recipe 头上（5b-8 细说这类错账）。

RoboGenesis：一台实验室数据引擎

RoboGenesis 是 LabVLA 论文的核心贡献之一——一台基于 Isaac Sim 的实验室 workflow + 数据引擎。它的流水线值得完整看一遍，因为它是「数据引擎」这个格子的标准解剖：

RoboGenesis pipeline（LabVLA 论文口径）
① environment building
   text-to-image 生成参考图 → TRELLIS 2.0 重建 3D 资产 → LabAssetLibrary
   + 1,000+ 纹理（LabTextureLibrary）→ 组装场景 → 10 项 validation check
   + 0-100 质量分过滤 → 已产出 10,000 个验证过的实验室场景

② agentic workflow generation
   自然语言指令 → 拆成原子技能有序序列
   例：Pick(beaker1) → Pour(beaker1, beaker2) → Place(hot_plate) → Press
   → 在 16 个 robot platforms 上跨本体实例化
   → 六轴 domain randomization（场景/相机/光照/杂物/物体/空间）

③ structured export
   skill controller 在仿真里真实执行 → 按执行成功过滤 rollout
   → 附加 subtask / object state / camera / spatial 等标注流
   → 导出 LabEmbodied-Data（喂给 5b-3 的 KI posttraining）

关于数据引擎，最常见的问题是：「它需不需要先给它一批人工录好的专家轨迹？」RoboGenesis 的答案揭示了数据引擎的本质：

不需要「预录制专家轨迹」，但需要「可执行专家逻辑」

RoboGenesis 不吃现成轨迹——轨迹是它产的，靠 skill controller 在仿真里真执行、真过滤。但天下没有免费午餐，它需要一整套可执行的专家逻辑：可交互资产（哪里能抓、按钮是不是 articulation）、robot profile、原子技能库、workflow 模板、success checker、导出 schema。换句话说：专家知识没有消失，只是从「轨迹」升维成了「程序」——程序可以被 domain randomization 放大成千上万条轨迹，这正是引擎的杠杆所在。

反过来说，只有一堆 beaker.usd 资产和一张任务清单是开不动引擎的——系统不知道烧杯哪里能抓、按钮按到什么程度算按下、任务成功怎么判。资产和指令之间缺的正是上面那套「可执行专家逻辑」。

LabUtopia 与 AutoBio：两个实验室考场

换到 benchmark 格子。LabUtopia（NeurIPS'25 D&B）是一套实验室仿真评测组件：LabSim（Isaac Sim 高保真仿真，支持多物理与化学反应语义）+ LabScene（程序化场景生成，200+ 资产）+ LabBench（5 个复杂度层级、30 个任务，从 atomic 操作到 long-horizon 移动操作）。它的职责是考试：给任务、reset 场景、统计成功率、区分 ID/OOD——LabVLA 就在它上面被评（5b-7 的考卷）。

AutoBio 是另一个实验室考场，偏生物实验：真实仪器数字化管线、螺纹/棘轮等实验室机构的专用物理插件、透明液体的 PBR 渲染，三个难度层级的任务，然后拿现有 VLA baseline 来微调与评测。它和 LabVLA 的定位差异是一面好镜子：

同在实验室，不同生态位

AutoBio / LabUtopia：「我造一个更像真实实验室的考场，看现有 VLA 在这里能考多少分。」——benchmark 层。

LabVLA（+ RoboGenesis）：「我造一条实验室 VLA 的训练流水线，用合成数据和多源数据养一个实验室特化模型。」——引擎层 + recipe 层。

把 EBench 放进这张地图

现在回头看这本书的主角。EBench 的星座（0.2）用三格分类器重新读一遍，会发现它三格都有布局：

格子EBench 星座里的对应物实验室生态的对应物
数据引擎GenManip rule-based 管线 + teleop 采集 → EBench-Dataset（6,600 episodes）RoboGenesis → LabEmbodied-Data
benchmarkEBench 本体：GenManip server + EvalClient 契约 + 26 任务打分 + leaderboardLabUtopia / AutoBio

这个对应关系给你一条现成的类比通道：RoboGenesis 之于 LabEmbodied-Data，恰如 GenManip 数据管线之于 EBench-Dataset——都是「可执行专家逻辑 + 成功过滤」的引擎产物（EBench 的 rule-based 子集正是脚本化专家逻辑跑出来的，2.3/2.4）；LabUtopia 之于 LabVLA，恰如 EBench 之于它的 baselines。学会一套解剖，两个生态都能读。

从 GAMES103 看 引擎与考场共享同一副物理底座

注意 RoboGenesis、LabUtopia、GenManip/EBench 全都建在 Isaac Sim（USD + PhysX + RTX，2.1）上。你学过的那套刚体 + 约束 + 时间步进，同时是「印教材的印刷机」和「考场的桌椅」——数据引擎用它执行 skill 产轨迹，benchmark 用它 reset 场景做评测。区别只在包装在物理循环外面的那层职责：一个接 exporter，一个接 evaluator。

这一节你需要带走的 · 下一节

(1) 三格分类器：数据引擎（造训练数据）/ benchmark（考试 + 评分）/ model recipe（训练方法学），新名字先归格；(2) RoboGenesis 的关键性质——不需要预录轨迹，但需要可执行专家逻辑（资产 + skill 库 + workflow + success checker），专家知识从轨迹升维成程序；(3) LabUtopia / AutoBio 是实验室考场，与 LabVLA（引擎+recipe）互补；EBench 星座三格皆有——GenManip 管线是引擎，EBench 本体是考场。
