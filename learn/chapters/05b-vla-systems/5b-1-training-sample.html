一条训练样本长什么样：episode → 切片

Part 5 把 VLA 的「发动机」拆完了：你知道了 action chunking 为什么存在、flow matching 怎么把噪声积分成动作。但有一个问题我们一直没有正面回答：这台发动机吃的「油」到底长什么样？一条训练样本包含哪些字段？一条成功的 rollout 贡献几条样本？这一 Part 我们不再讲新的数学，而是把 VLA 从数据到评测的整条训练流水线解剖一遍——从最小的单位开始：一条样本。

先拆一个常见误区

很多初学者对「用成功数据训练 VLA」的想象是这样的：

误区

「VLA 后训练 = 给模型看最终成功的画面 + 一条轨迹，模型就学会了成功。」

这个说法把监督信号想错了。模型从来不学「成功画面」——最后一帧微波炉门关上的样子对控制毫无直接用处。真正的监督信号是：在成功 episode 的每一个时刻 $t$，专家接下来是怎么动的。形式化写出来，一条训练样本是：

$$\underbrace{(o_t,\; q_t,\; \ell)}_{\text{输入：图像、状态、指令}} \;\longrightarrow\; \underbrace{A_t = [a_t, a_{t+1}, \dots, a_{t+H-1}]}_{\text{目标：未来 } H \text{ 步专家动作}}$$
  • $o_t$：当前时刻的图像观测（通常多相机 RGB）。
  • $q_t$：机器人本体状态——关节角、夹爪、末端位姿。
  • $\ell$：语言指令，整条 episode 共用一句。
  • $A_t$：从 $t$ 时刻起往后 $H$ 步的 action chunk——正是 5.1 那个 chunk。π0/π0.5 取 $H=50$，RDT 预测 64 步，LabVLA 取 $K=50$。

换句话说，5.1 讲 chunk 时你看到的是推理端（一次吐 50 步）；现在你看到它的训练端——监督目标本身就是按 chunk 组织的。两端严丝合缝。

从 RL 看 和 replay buffer 里的 (s, a) 对比一下

在 RL 里你存的是 $(s, a, r, s')$ 转移，一步一条；训练信号来自 reward。这里存的是专家演示，一条样本是 $(o_t, q_t, \ell) \to a_{t:t+H}$——没有 $r$、没有 $s'$、目标是一整段动作。而且注意分布：replay buffer 混着各种策略的行为，而这里全部来自成功的专家轨迹——这正是 5.1 讲过的 covariate shift 的根源（训练分布 = 专家分布，部署分布 = 学生自己走出来的分布）。

一条 episode 切出的不是一条样本

接着是第二个关键认知：episode 和样本不是一比一的关系。以 EBench 的 long-horizon 任务 Microwave 为例（开微波炉 → 放蛋挞 → 关门），一条成功 episode 从 $t=0$ 录到 $t=T$，每一帧都存了观测和动作。训练时，我们把它逐时刻切片：

episode → training samples（示意）
sample @ t=0:   input: image_0 + state_0 + instruction   target: action_0  … action_49
sample @ t=1:   input: image_1 + state_1 + instruction   target: action_1  … action_50
sample @ t=2:   input: image_2 + state_2 + instruction   target: action_2  … action_51
...
sample @ t=T-1: input: image_T-1 + ...                   target: action_T-1 …（不足 H 步则 pad/mask）

所以一条长度 $T$ 的 rollout 大约贡献 $T$ 条 $(obs,\ \text{chunk})$ 样本。EBench-Dataset 的 6,600 条 episodes 合计约 986 万帧——按这个切法，可用的监督样本数量级是百万级，而不是六千级。这就是为什么录数据时「每一帧都要存好」：每一帧都是一个潜在的训练样本起点。亲手切一下：

interactiveepisode 切片器：拖动 t，看每条样本怎么切
拖「切片起点 t」或直接点格子，蓝色是这条样本的输入观测，青色是它的 action chunk 目标；调 $H$ 看 chunk 覆盖范围。注意切到尾部时 chunk 会「不够长」——工程上用 pad + mask 处理。

用 EBench 自己的数据对号入座

这个抽象格式你其实已经见过实物了。回忆 2.4：EBench-Dataset 是 LeRobot v2.1 格式，每帧的字段正好就是上面公式里的三类东西：

公式里的量EBench-Dataset 里的实物
$o_t$ 图像观测video.top/left/right/overlook_view（4 相机 480×640，AV1-MP4，15 fps 录制）
$q_t$ 本体状态state.joints(12)、state.gripper(4)、state.base(3)、state.ee_pose(14)
$\ell$ 指令meta/tasks.jsonl（每任务多套指令释义）
$a_t$ 动作action.joints/action.gripper/action.base 及各自 _delta 增量通道

注意 _delta 通道的存在：同一个物理动作既存了绝对值也存了增量表示。哪个当训练目标是 recipe 的选择——比如 LabVLA 预训练/后训练用 absolute targets、微调阶段切到 delta targets（5b-3 会展开）。数据集把两种都存好，就是把选择权留给训练方。

设计建议：一个「可训练」的 benchmark 应该导出什么

反过来看，这也定义了对评测/数据系统的要求。如果一个 benchmark 希望自己的 rollout 未来能喂给 VLA 训练，只存视频和成功率是不够的，至少要每帧存齐：instruction、多相机帧、本体状态、实际执行的动作、物体状态，外加 episode 级的 success/failure 与失败原因、（最好有）子任务阶段标注。这是设计建议而非 EBench 现有代码事实——但 EBench-Dataset 的字段设计已经基本对齐这个标准。

这一节你需要带走的 · 下一节

(1) VLA 的监督信号不是「成功画面」，而是每个时刻的观测 → 往后 $H$ 步的专家动作，一条样本 = $(o_t, q_t, \ell) \to a_{t:t+H}$；(2) 一条 episode 逐时刻切片，贡献 ~$T$ 条样本——6,600 episodes 是百万级样本；(3) EBench-Dataset 的字段正是这个格式的实物，absolute/delta 双通道把表示选择权留给 recipe。

但「VLA」这个词本身，其实比这条样本格式模糊得多——π0.5 是 VLA，RDT 也自称 VLA-style，它们内部却很不一样。下一节我们把「VLA」这个词拆成三层定义，你会发现很多争论只是在不同层面各说各话。
