Case study：π0.5 vs LabVLA

压轴案例来了。前六节的所有工具——三层定义、训练账本、数据混合、KI、生态位——现在一次性用上，把 π0.5 和 LabVLA 并排放上解剖台。目的不是裁判谁「绝对更强」（你马上会看到这个问题本身就问错了），而是通过一对「同 recipe、不同世界」的模型，把 VLA 系统的迁移理解焊牢。

一句话定位

两个模型，两个世界

π0.5：通用开放世界 / 家庭 VLA——走进训练中从未见过的家，清厨房、铺床、挂毛巾；强调对新场景、新物体、长流程的泛化（5.8）。

LabVLA：科学实验室特化 VLA——烧杯、加热台、仪器门、倒液、按钮；强调固定 protocol workflow 的可靠执行（arXiv 2606.13578）。

训练结构对比：recipe 同源，器官不同

用 5b-2 的三层定义扫一遍：接口层两者都是标准 VLA；recipe 层高度同源（FAST 预训练 → flow 后训练 → KI）；差异集中在架构选型与数据来源：

训练数据对比：一个靠采，一个靠造

最深刻的差异在数据层。π0.5 的开放世界泛化建立在真实数据的广度上：多本体 + ~400h 真实家庭移动操作 + web 共训，泛化随场景数 scaling（~100 个场景饱和，5.8 讲过）。LabVLA 的实验室能力建立在合成数据的针对性上：通用先验来自公开数据集（DROID、AgiBot、OXE-AugE ≈572k trajectories），实验室领域知识全部来自 RoboGenesis 合成的 LabEmbodied-Data。

别直接比数据量——单位就没对齐

π0.5 的「~400 小时」、OXE-AugE 的「572k trajectories」、EBench 的「6,600 episodes / 9.86M frames」、LabVLA 的「100k steps」——小时、轨迹数、episode 数、帧数、训练步数是五种不同单位，之间没有普适换算（一条轨迹几秒到几分钟不等、一步梯度吃多少帧取决于 batch 与采样）。论文对比表里把不同口径的数字排在一列，是初学者最容易踩的读数陷阱。

LabUtopia 考场上的成绩单——以及怎么读它

LabVLA 论文在 LabUtopia（5b-6 认识的实验室考场）上评了 6 类任务：Pick Up / Press Button / Open Door / Pour Liquid / Heat Beaker / Transport Beaker，每类任务 × ID（贴近训练分布）与 OOD（扰动物体位置、外观、场景配置）两种设置 × 120 episodes——每个模型共 6 × 2 × 120 = 1,440 个 episode。关键数字（论文 Table 2）：

几个值得注意的模式：Pour Liquid 全场没人过 50%（倒液需要精确倾角且没有液面反馈——考场暴露了整个领域的短板）；多个 baseline 呈「单科满分、单科挂科」的尖刺形（SmolVLA：Heat Beaker 98.3% / Pour Liquid 1.67%），LabVLA 则最均衡——对要串联多步操作的 protocol 来说，均衡比单科峰值重要得多。这正是 0.1 capability profile 哲学的又一例证：平均分相近的模型，画像可以完全不同。

这张成绩单能证明什么、不能证明什么

现在到本节最重要的部分。先看一个论文里写明、却极易被忽略的实验条件：

训练暴露（training exposure）声明

LabUtopia 对比中，所有 baseline 都以 public checkpoint 接入评测 harness（只做 action/state schema 适配），没有在 LabEmbodied-Data 上微调；而 LabVLA 的后训练吃了大量 RoboGenesis 合成的实验室数据。也就是说：一边是「实验室特训生」，一边是「通用公开模型直接进考场」。

于是结论必须小心措辞：

  • 可以说：实验室特化训练后的 LabVLA，在 LabUtopia 上显著优于以 public checkpoint 接入的 π0.5——说明 lab-specific 数据与训练对实验室任务非常有效。
  • 不能说：LabVLA 的架构本身强于 π0.5 的架构。这不是 same-data 的 architecture ablation——数据暴露完全不同，架构贡献无法从中分离（5b-2 的三层定义：赢在 recipe+数据层，不能记账到架构层）。

有意思的是，LabVLA 论文自己补了一个佐证「赢在数据」的实验（Table 3）：把 X-VLA（EBench 读者的老朋友，6.2）拿来在 LabEmbodied-Data 上微调，5 个非饱和任务平均 ID +15.0pp、OOD +19.3pp（Heat Beaker 从 25.8% 涨到 68.3%）。监督不绑定 LabVLA 架构——换个模型喂同样的实验室数据，也涨。这几乎就是「数据 vs 架构」归因实验的教科书做法。

论文还做了真机迁移（Franka 平台 4 个 2–4 步任务，各 50 rollouts/setting）：LabVLA 与 DreamZero 互有胜负、差距多在方差内，两者都稳定超过 π0.5——仿真预训练能上真机，但排名逻辑与上面同理：训练暴露不同，读数要带上下文。

从 RL 看 这是「换 reward 再比较」的老陷阱

RL 里你不会拿「在迷宫 A 训练的 agent」和「没见过迷宫的 agent」在迷宫 A 里比分，然后宣布前者的网络结构更好——分布内经验才是主导变量。VLA 的对比同理：训练分布与评测分布的重叠度（domain exposure）是第一解释变量，架构差异只有在暴露对齐后才可归因。这个直觉将在下一节被系统化成一张 checklist。

这一节你需要带走的 · 下一节

(1) π0.5 与 LabVLA 是同一 recipe 谱系在两个世界的实例——差异集中在 backbone 选型、数据来源（真实广度 vs 合成针对性）与高层语义形态；(2) LabUtopia 成绩单（LabVLA 71.1/70.0 vs π0.5 52.4/52.1）证明的是实验室数据+训练的价值，不是架构优劣——所有 baseline 都是 public checkpoint 接入；(3) X-VLA + LabEmbodied-Data 微调 +15/+19.3pp 是「赢在数据」的干净佐证；(4) 读任何对比表，先查训练暴露、再看数字单位。

「先查训练暴露」——这句话值得升级成一套完整的方法论。下一节（也是全 Part 的收束）：怎么公平地比较 VLA，以及一个 benchmark 报告应该披露什么。
