训练账本：预训练、后训练、微调

这是全 Part 最重要的一节。5.8 已经告诉你 π0.5「预训练用 FAST 离散、后训练切 flow matching」——但那是为什么层面的故事。工程上真正要紧的问题是四个：每个阶段输入什么数据、监督什么目标、算什么 loss、更新哪些参数。把这四栏对每个阶段记清楚，就是一本训练账本（training ledger）。账本在手，任何 VLA 论文的 training 章节你都能秒读。

为什么需要账本：「使用了某某技术」是句废话

论文里常见这种句子：「我们采用 FAST tokenization 与 flow matching，并使用 knowledge insulation」。名词全对，信息量近零——因为它没告诉你这些东西发生在哪个阶段、作用在哪个模块上。同一个「flow matching」，出现在预训练还是后训练、梯度回不回 backbone，是完全不同的系统。账本格式强迫把这些说清：

账本的四栏

输入（什么数据进来）→ 监督目标（拿什么当标准答案）→ loss（怎么衡量差距、CE 还是 flow MSE）→ 更新谁（梯度落在哪些参数上）。第四栏最常被论文一笔带过，却是 5b-5 要讲的 knowledge insulation 的全部内容。

π0.5 的账本

先给熟人立账。π0.5 的训练可以粗略拆成两阶段 + 推理（依 π0.5 论文与 KI 论文口径）：

阶段输入监督目标loss更新谁
Pretraining
co-training		 多本体机器人数据、~400h 真实家庭移动操作、高层子任务标注、物体检测、web 图文 text token / 子任务文本 / FAST 离散 action token cross-entropy（next-token） VLM backbone（PaliGemma 3B）
Posttraining
flow matching		 机器人轨迹（obs → 连续 chunk，$H=50$） flow 速度场（5.5 那个 $A-\varepsilon$） flow matching MSE；KI 方案下另有离散 CE 继续喂 backbone action expert（~300M，随机初始化）；backbone 只被离散 CE 温和更新，flow 梯度被 stop-gradient 挡住
Inference 当前观测 + 指令 先离散解码高层 subtask，再 flow 积分出低层 chunk （无——没有梯度） （不更新任何参数）

用一句大白话总结这本账：第一阶段教大脑认识「机器人动作」这门外语（用它熟悉的离散 token 方式教）；第二阶段接上小脑学连续控制（但不许小脑的乱梯度打扰大脑）；推理时先想后做。

LabVLA 的账本：100k / 80k / 80k

LabVLA（arXiv 2606.13578）把同一套谱系落得更「有账可查」——论文附录直接给了三列超参表。先处理一个口径问题：

口径提醒：two-stage 还是三阶段？

LabVLA 论文正文说 recipe 是 two-stage（FAST 预训练 → flow matching 后训练）；附录超参表却有三列：VLM pretraining 100k steps / KI posttraining 80k / finetuning 80k。两者不矛盾——正文讲的是 recipe 骨架，附录多出的第三列是下游任务微调。本书按三阶段记账，但标注来源。这种「正文与附录口径并存」在论文里很常见，读的时候要习惯对齐它们。

阶段steps输入数据监督目标更新谁
VLM pretraining100k Robointer-VQA、AgiBot World Beta、OXE-AugE（LeRobot 子集 ≈572k trajectories）、DROID VQA 答案、language subtask、FAST 离散 action token（absolute targets）——全部 CE Qwen3-VL-4B backbone（此阶段 DiT 根本没实例化）
KI posttraining80k OXE-AugE + LabEmbodied-Data（RoboGenesis 合成的实验室 rollout） 连续 action chunk（$K=50$，absolute targets）的 flow 速度场 DiT action expert + projection；flow 梯度 stop-gradient 不回 backbone，backbone 仍被 annotation CE 更新
Finetuning80k 下游任务 / 具体机器人 schema 数据 delta action targets（注意目标表示切换了） 整个 VLA policy 向具体任务对齐

三个阶段的角色，还是那个比喻：pretraining 教大脑、posttraining 接小脑、finetuning 把整套神经调到具体任务和具体 action schema 上。注意两个账本级细节：(a) absolute → delta 的目标切换发生在微调阶段——回忆 5b-1，EBench-Dataset 之所以双通道都存，就是为这种切换留余地；(b) LabVLA 论文附录还坦白了一个教训——微调效果最大的单一因素是 warmstart checkpoint 的数据构成（posttraining 数据更多样，下游 TransportBeaker 从 60% 涨到 86%），而不是 DiT 结构或学习率。数据在账本里的权重，往往大过架构。

为什么要分阶段：两种 loss 的「脾气」不同

账本立完，回答一个更根本的问题：为什么不把所有数据、所有 loss 一锅炖？因为两类监督的稳定性完全不同：

  • 离散 CE（图文问答、FAST token）：和语言模型预训练同构，梯度温和、训练稳定，天然兼容 VLM 已有的机制——适合动 backbone。
  • 连续 flow matching：更贴近控制本质，但动作头是随机初始化的新模块，早期梯度大而乱（5.8 的「乱流」比喻），直接回传会把 backbone 的语义空间拉歪。

于是现代 VLA 的阶段设计可以浓缩成四句口诀：先离散、后连续；先语义、后控制；先大脑、后小脑；连续 loss 小心回传。前三句是课程安排（curriculum），第四句就是 knowledge insulation——5b-5 专门拆它。

从 GAMES103 看 分阶段 = 先松弛再耦合

这套「先把子系统各自调稳、再耦合起来精调」的思路，在仿真里你见过：解一个刚柔耦合系统时，直接全耦合迭代常常发散，实践中先让各子系统分别收敛（松弛），再引入耦合项做联合迭代，稳定得多。VLM pretraining 就是把「大脑」先松弛到位；posttraining 引入耦合（expert 读 hidden states）但用 stop-gradient 限制耦合方向；finetuning 才做全系统的精调。

一个来自论文附录的工程注脚

LabVLA 把 action 向量 pad 到固定 32 维（单臂机器人常只有 8 维是活跃的）。早期实现把 flow loss 在全部 32 维上平均——padded 维贡献 0 误差却计入分母，等于把 expert 的梯度悄悄缩小 4 倍；修正成只在活跃维平均后，等效学习率变了 4 倍，还得重新调参。这个故事有两个教益：账本的「loss」一栏要精确到在哪些维度上平均；而「按 mask 只训有效字段」正是下一节多源混训的核心原则。
