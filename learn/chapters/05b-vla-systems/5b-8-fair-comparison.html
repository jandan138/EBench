公平比较 VLA：训练暴露与 claim boundary

上一节我们对一张成绩单反复追问「能证明什么、不能证明什么」。这一节把那套追问系统化：一张评测前必问的 checklist、一套模型状态的分类学、一个报告模板。它是 Part 8 评测科学在 VLA 身上的落地——也是你未来读 leaderboard、写评测报告时可以直接抄的作业。

评测前必问的 10 个问题

任何「模型 A 在 benchmark B 上得了 X 分」的陈述，价值都取决于这 10 个问题的答案。前五问关于训练暴露（training exposure），后五问关于接口与口径：

VLA 评测 checklist
 1. 这是 public checkpoint，还是用本 benchmark（同源）数据 fine-tune 过？
 2. 它见过同类任务吗？（protocol/任务模板是否与评测重叠）
 3. 它见过同类资产吗？（物体/场景/纹理分布）
 4. 它见过同一个仿真器吗？（渲染风格本身就是分布）
 5. 它用过同一批 expert trajectories 吗？
 6. action space 一致吗？（joint / delta_eef / 绝对 vs 相对——回忆 4.3）
 7. observation schema 一致吗？（相机数量、视角、分辨率、本体状态字段）
 8. 外面接了 planner / retry wrapper / safety critic 吗？
 9. ID / OOD 是怎么定义的？扰动了什么？
10. 成绩是单一 success rate，还是带失败阶段与过程证据？

问题 1–5 决定「分数里有多少是见过换来的」；6–8 决定「你在测模型还是在测整个系统」；9–10 决定「这个数字支持多宽的结论」。5b-7 的 LabUtopia 案例里，问题 1 的答案（baseline 全部 public checkpoint、LabVLA 特化训练）直接决定了结论只能落在数据层而非架构层。

模型状态五档：同一个名字，五种东西

Leaderboard 上并排的两行「π0.5：52.4」「LabVLA：71.1」，看起来是同类对象——其实未必。按训练暴露把「参赛者」分成五档，报告时必须声明档位：

状态含义报告时怎么写
zero-shot public checkpoint公开权重直接接入，只做 schema 适配「未使用本 benchmark（同域）数据后训练」
same-domain fine-tuned用同领域（如实验室仿真）数据训过，但非评测同源「使用 X 类数据微调，与评测任务不同源」
same-benchmark fine-tuned用与评测高度同源的数据（同任务/同资产/同仿真器）训过必须显式披露——不披露就是误导
system-enhanced VLA裸模型外接 planner / recovery wrapper / safety critic报「系统成绩」，不得当作裸 VLA 能力
oracle / scripted expert脚本化专家（如 RoboGenesis 的 skill controller）只作数据生成器或性能上限参考，不进排名
从 RL 看 这就是「和 oracle 比什么」的老规矩

RL 论文里你早就习惯了这些区分：报告 agent 成绩时要说清有没有 reward shaping、有没有 curriculum、demo 是不是喂过、对照的是 random policy 还是 oracle。VLA 只是把同一套「实验条件披露」的规矩搬到了模仿学习语境——五档状态表就是 VLA 版的 experimental setup 声明。

想真正归因架构？对照要这么设计

如果你确实想回答「A 架构是不是比 B 架构好」（而不只是「A 系统在这个考场分高」），对照实验要把训练暴露对齐掉。以 π0.5 vs LabVLA 为例，一套干净的设计是两组对照：

  • 对照一（都不喂）：π0.5 public checkpoint vs LabVLA base checkpoint——比较两者的先验。
  • 对照二（都喂同样的）：π0.5 + LabEmbodied-Data 微调 vs LabVLA + LabEmbodied-Data 微调——同数据、同任务、同 held-out 划分、同 action/obs schema、同梯度步数、同评测 harness。此时的差距才可以（谨慎地）记到架构头上。

LabVLA 论文的 X-VLA 迁移实验（5b-7：+15.0/+19.3pp）正是朝这个方向迈的半步——它证明了数据对外部架构也有效，从反面支撑「主要赢在数据」的解读。完整的双向对照（π0.5 也喂 LabEmbodied-Data）论文没有做，这类「缺失的对照」正是你读论文时要习惯性寻找的空位。

给 benchmark 报告的模板

把上面所有内容压缩成一份可执行的东西：报告一个 VLA 成绩时应该附带的结构化披露。这是设计建议（不是 EBench 现有代码的字段），但它的每一项都直接对应前面的 checklist：

model_report_card.yaml（设计建议）
model_card:
  model_name: LabVLA
  checkpoint: public | private | finetuned
  backbone: Qwen3-VL-4B-Instruct
  action_head: DiT flow-matching expert
  action_horizon: 50

training_exposure:            # ← checklist 问题 1-5
  used_this_benchmark_data: false
  used_same_domain_sim_data: true
  used_same_assets: unknown
  used_same_tasks: unknown
  used_same_expert_trajectories: false

interface:                    # ← 问题 6-7
  action_schema: delta_eef | joint | custom
  observation_schema: { cameras: [front, wrist], proprioception: true }

system_components:            # ← 问题 8
  planner: none
  safety_critic: none
  recovery_wrapper: none

evaluation:                   # ← 问题 9-10
  setting: ID | OOD
  episodes_per_task: 120
metrics:
  success_rate: ...
  step_completion: ...        # 失败在哪一步
  failure_stage: ...
  collision_rate: ...
和 Part 8 的接线

这个模板不是新发明，而是 Part 8 三个原则的 VLA 实例化：training_exposure 就是 8.6 的 train/test isolation 与 leakage 审计；system_components 与 metrics 的多字段化对应 8.1 的 construct validity（你到底在测什么）；整张卡片就是 8.4 说的 claim boundary——把「这个数字支持什么结论」写成结构化字段。EBench 自己的 val_train / val_unseen / test 隔离（3.2）正是 training_exposure 思想在 benchmark 侧的镜像。

这一节你需要带走的 · 也是整个 Part 5.5 的收束

四件事：(1) 评测前先过 10 问——训练暴露五问 + 接口口径五问；(2) 「参赛者」有五档状态（zero-shot / same-domain FT / same-benchmark FT / system-enhanced / oracle），报告必须声明档位；(3) 要归因架构，必须做暴露对齐的双向对照（same-data fine-tune）；(4) 把这一切固化成 report card——它就是 claim boundary 的结构化形态。
