「VLA」的三层定义：接口 / 架构 / recipe

读论文时你迟早会撞上这类问题：「RDT 是不是 VLA？」「为什么 OpenVLA 和 π0.5 都叫 VLA，内部却完全不同？」这些争论大多不是事实分歧，而是在三个不同层面各说各话。这一节把「VLA」这个词拆成三层——接口、架构、训练 recipe——以后你听到任何「X 算不算 VLA」的讨论，先问一句：你说的是哪一层？

三层定义，一图看清

1.2 给过一个工作定义：「拿预训练 VLM 骨干，加上输出 robot action 的能力」。那是架构层的说法。现在我们把镜头拉远，一共有三档焦距：

三层定义（由宽到窄）

① 接口定义（最宽）：只看输入输出——Vision + Language + state → Action。是个黑盒判据。

② 架构定义（较窄）：盒子里必须是「pretrained VLM backbone + action expert / head」——先有视觉语言大脑，再接动作小脑。

③ recipe 定义（最细）：连训练流程都要像——多源数据混训、FAST 离散预训练、连续 flow matching 后训练、knowledge insulation、任务微调这一整套谱系。

第一层：接口——几乎大家都是 VLA

按接口定义，只要一个模型吃图像 + 指令 + 本体状态、吐连续动作（或 action chunk），它就是 VLA。这一层的判据在 EBench 语境里再具体不过：能不能装进 4.2 的 EvalClient 契约——`obs` 进、`action` 出。π0、π0.5、X-VLA、InternVLA-A1、LabVLA、RDT、OpenVLA、RT-2……全部通过。接口定义有用（它定义了评测的公平接入面），但它不区分内部构造——就像「四个轮子会跑」既包括燃油车也包括电车。

第二层：架构——大脑 + 小脑，还是条件生成器？

把盒子打开，差异出现了。典型的「窄定义 VLA」是两段式：一个在 web 图文上预训练过的 VLM 负责「理解世界」，一个动作模块负责「生成控制」。π0.5（PaliGemma 3B + ~300M action expert，5.7）和 LabVLA（Qwen3-VL-4B + DiT action expert）都是教科书样本。

但 RDT-1B（Robotics Diffusion Transformer，清华 TSAIL，1.2B 参数）走的是另一条路：它的主体就是一个 Diffusion Transformer 策略——在 46 个数据集、100 万+ episodes 上预训练，用 Physically Interpretable Unified Action Space 统一各种本体的动作表示，一次预测 64 步动作。图像和语言在这里是扩散生成的条件，而不是「先由 VLM 大脑读懂、再交给小脑」的两段结构。一句话对比：

两条架构路线

π0.5 / LabVLA：先用视觉语言大脑理解世界，再由动作专家生成动作——VLM-centric。

RDT：一个条件扩散动作轨迹生成器，图像和语言是生成动作的条件——policy-centric。所以说 RDT 是「VLA-style robot policy」比说它是窄义 VLA 更准确。

还有第三种老路线：RT-2 / OpenVLA 把动作离散成 token、当语言吐——架构上是纯 VLM，动作只是词表里多出来的一些「词」。5.8 讲过 π0 为什么放弃这条路（自回归太慢、分箱损精度，不适合高频灵巧控制）。

第三层：recipe——π0.5 和 LabVLA 惊人地像

再把焦距推到最细：怎么训练。有趣的事情发生了——目标场景天差地别的 π0.5（开放世界家庭）和 LabVLA（科学实验室），训练配方几乎同源：

共享的 recipe 谱系
VLM backbone
+ 多源机器人 / 图文数据混训
+ FAST action token 离散预训练（让 backbone 先「懂动作」）
+ 连续 action expert（flow matching 后训练）
+ knowledge insulation（stop-gradient 护住 backbone）
+ 任务微调

这说明「recipe」已经开始标准化、可迁移——就像 CV 里的「ImageNet 预训练 + 下游微调」范式。把三层放进一张表：

模型① 接口是 VLA？② 架构路线③ 训练 recipe

从 RL 看 「policy」这个词也有一模一样的三义性

你其实早就习惯这种分层了。RL 里说「policy」，可以指接口（任何 $\pi: s \to a$ 的映射）、可以指参数化（高斯 actor？确定性 DDPG actor？还是 Q 函数隐式诱导的贪心策略？），也可以指训练法（policy gradient 训的、TD 训的、还是 BC 训的）。「SAC 的 policy 和 DQN 的 policy 是不是一回事」——和「RDT 是不是 VLA」是同构的问题：接口层是，参数化和训练法不是。

为什么这个分层重要

因为三层对应三种不同的结论边界。接口层相同 → 可以进同一个 benchmark 公平接入（EBench 正是这么做的）；架构层不同 → 性能差异可能来自结构先验（比如 VLM 带来的 web 知识 vs 扩散策略的动作多模态表达）；recipe 层不同 → 差异可能只是数据和训练暴露的差异，和架构无关。5b-7 对比 π0.5 和 LabVLA、5b-8 讨论公平比较时，我们会反复用到这个分层：「谁赢了」必须说清是在哪一层赢的。

这一节你需要带走的 · 下一节

(1) 「VLA」有三层定义：接口（V+L+state→action，EvalClient 契约就是它的化身）、架构（VLM 大脑 + 动作小脑）、recipe（FAST→flow→KI 谱系）；(2) RDT 在接口层是 VLA-style，架构层是条件扩散策略而非「VLM+expert」；(3) π0.5 与 LabVLA 在 recipe 层高度同源——这层的标准化正是本 Part 的主线索。

下一节是全 Part 的核心：把 recipe 这层彻底摊开，给 π0.5 和 LabVLA 各立一本训练账本——每个阶段输入什么、监督什么、用什么 loss、更新谁。
